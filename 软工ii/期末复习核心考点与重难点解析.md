
针对期末考试的复习方向，刘钦老师明确了重点考察范围及易错点：
1 . UML 图与建模规范 UML 图的重要性：UML 图贯穿软件工程全生命周期，从需求阶段的用例图、系统顺序图，到设计阶段的包图（4+1视图）、顺序图、状态图，再到详细设计的流程图，均需掌握。 包图（Package Diagram）绘制规范：包图用于表现体系结构的分层（如 MVC），需画出展示层、逻辑层、数据层及各层间的接口（如 PO、VO）。虽然考试时间有限，评分会看整体结构，但建议尽量画出详细的包内依赖以体现层内复杂性。 顺序图（Sequence Diagram）消息类型：明确区分同步消息（实线+实心三角箭头）与异步消息（虚线+鱼骨箭头）。 状态图（State Diagram）核心逻辑：绘制状态图前必须明确“主体”是什么（如单个红绿灯或整个路口），并理解状态图如何转化为代码（如 Switch 语句或 State 模式）。 概念类图（Conceptual Class Diagram）：概念类图仅包含类与属性，不含方法。 2. 设计原则与代码质量 设计原则考察重点：考试将重点考察 SOLID 等设计原则，可能以简答题形式出现，要求分析代码是否符合或违背原则。 内聚与耦合（Cohesion & Coupling）：虽然今年未深入讲解，但需警惕两种常见耦合错误。控制耦合（Control Coupling）指通过 Flag 参数控制方法内部逻辑，应改为多态实现；印记耦合（Stamp Coupling）指传递了超出方法所需的数据，应只传递必要参数。 关联关系区分：普通关联、聚合（Aggregation）和组合（Composition）均属于关联。聚合具有整体与部分的关系但无生命周期绑定，组合则具有生命周期关系（整体消亡则部分消亡）。 3. 架构分层与对象定义 PO 与 VO 的区别：PO（Persistent Object）是数据层与逻辑层交互的对象，用于 ORM 映射；VO（Value Object）是展示层与逻辑层交互的对象，仅包含展示所需数据，避免返回多余信息。 构建接口定义：构建（Component）之间的接口不仅包括供接口（Provided Interface），还包括需接口（Required Interface），共同构成完整的体结构接口。 4. 非重点与考试形式 非重点内容：包的原则（Package Principle）、团队管理概念、具体技术细节（如 Java 注解、Spring Boot 配置）及需求规格说明书的书写格式均不会考察。 设计模式考察形式：今年设计模式题目将采用“诱导式”出题，通过小问题引导考生运用设计原则进行设计，而非单纯背诵模式名称。
