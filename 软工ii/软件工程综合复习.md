[[软件工程]]

## 1-2章

**软件工程是将系统化、规范化、可量化的方法应用于软件开发、运行和维护，并研究这些方法等学科**

- 软件工程是一种**工程化方法**，不是单纯写代码。它强调需求、设计、构造、测试、维护、配置管理、项目管理、质量保证等活动
- 软件工程强调**系统化、规范化、可量化**。系统化是指开发活动有组织、有过程；规范化是指文档、评审、测试、配置管理等有规则；可量化是指通过度量、缺陷统计、进度跟踪、覆盖率等方式控制项目
- 软件工程等目标是在特定成本、时间、资源和质量约束下，构造能够解决现实问题的软件系统。工程成功不仅是“实现功能”，还要考虑真实问题、科学知识、软件方案和成本收益有效性

1950s ～ 2000s之间的特点

- 1950s 软件依附硬件，高级语言出现
- 1960s 软件危机暴露，软件工程概念提出
- 1970s 结构化方法和瀑布模型发展
- 1980s 重视过程管理、质量管理、复用和面向对象萌芽
- 1990s 面向对象、UML、RUP、CMM 和互联网软件发展
- 2000s Web、敏捷、开源、持续集成、服务化和云计算推动软件工程进一步演化

## 第4章 项目启动

**如何管理团队**？
团队是由少量具有互补技能的人组成，他们承诺共同目标、共同绩效目标和共同工作方法，并相互承担责任。软件团队应有共同目标、共享责任、互补技能和清晰结构。

### 团队结构

主程序员团队：以一个核心程序员或技术负责人为中心，核心成员负责关键设计和技术决策，其他成员围绕其工作
优点：决策快；技术路线统一；协调成本低；适合小规模、高技能、目标明确的项目；
缺点：过度依赖核心成员；核心成员容易成为瓶颈；其他成员参与感和主动性可能下降；核心成员离开时风险很大

民主团队：强调成员平等，大家共同讨论和决策
优点：创造性强；成员参与感高；有利于探索性任务；知识共享充分；
缺点：决策慢；责任边界可能模糊；容易争论过多；

开放团队：介于主程序员团队和民主团队之间，既有正式分工和责任人，又允许成员充分沟通和协作
优点：兼顾效率和沟通；角色清晰但不僵化；适合中大型工程项目；有利于团队协作和任务并行
缺点：需要较好的沟通机制；需要团队成员有一定的自我管理能力；如果规则不清晰，也可能变得混乱

### 质量保障
质量保障包括产品质量和过程质量。产品质量关注软件本身是否正确、可靠、可维护；过程质量关注团队是否按照合理流程开发、评审、测试、发布

核心思想：通过规范的过程和多种验证手段，尽早发现问题，降低缺陷进入后续阶段的成本

**质量保障的措施**：
需求阶段：
检查需求是否正确、完整、一致、可测试
避免“快速”“方便”“尽可能”等模糊词
为关键需求设置验收标准
进行需求评审，确保用户、开发者、测试者理解一致

设置阶段：
评审体系结构是否满足质量属性，如性能、安全、可靠性、可维护性
检查模块职责是否清晰，接口是否稳定
确认设计能覆盖需求
对高风险技术做原型验证

编码阶段：
采用编码规范
进行代码审查
使用静态分析工具检查潜在缺陷
执行单元测试
对AI生成代码进行人工审查和运行验证

测试阶段
进行单元测试、集成测试、系统测试和验收测试
设计正常路径、异常路径、边界条件测试
记录缺陷、修复缺陷并回归测试
统计测试通过率、缺陷数量、覆盖率等

发布阶段
进行发布检查
确认版本、配置、数据库脚本、环境变量一致
准备回滚方案
进行配置审计

**审查和测试的区别**
审查：代码/文档说了什么？
通过人工或工具阅读工件，发现逻辑、规范、可维护性、安全性等问题

测试：代码运行时做了什么？
通过执行程序，观察实际行为是否符合预期

### 配置管理
**什么是配置管理**？
配置管理用于识别、控制、记录和审计软件产品及其变化。
配置项包括管理文档、评审记录、软件文档、源代码、可执行文件、环境配置等；
基线是经过正式评审并作为后续工作的稳定版本，只有通过正式变更控制才能修改。

**管理配置有哪些活动**？

配置项识别：
确定哪些内容需要纳入配置管理
例如源代码、需求文档、接口文档、测试用例、配置文件、部署脚本等

版本管理：
记录配置项等不同版本
例如使用Git管理代码版本，通过commit、branch、tag记录历史

变更控制：
当需求、设计、代码、配置发生变化时，要提出变更请求，分析影响，审批后修改。
重点是防止随意修改造成范围蔓延和系统不一致

配置审计：
检查实际配置是否与记录一致
比如发布前确认服务器运行的是正确版本，配置文件和数据库脚本是否匹配

状态报告：
记录和报告配置项当前状态
例如哪些需求已实现，哪些缺陷已修复，当前版本包含哪些功能，哪些变更正在审批

发布管理：
组织软件发布，确认发布包内容完整、一致、可追踪
包括打包、版本号、发布说明、部署脚本、回滚方案等

## 第五章 需求基础

### 需求
需求是用户为解决问题或达成目标所需要的条件或能力，也是系统为了满足合同、标准、规范或用户期望而必须具备的条件或能力，同时还是对这些条件和能力的文档化表述

**需求的三个层次**
业务需求、用户需求、系统级需求

业务需求：为什么要做？
⬇️
用户需求：用户要用系统完成什么任务？
⬇️
系统级需求：系统具体应该表现出什么行为？

不同层级需求对应不同开发技术：
业务需求更接近目标分析，用户需求适合用场景/用例分析，系统级需求则通过OO分析或结构化分析进一步细化

**业务需求** ：为什么要开发这个系统？组织或客户希望通过系统获得什么业务价值？
站在组织、客户、业务目标角度描述，通过比较高层
看到“提高、降低、减少、扩大、增强、支持业务目标”，大概率是业务需求

**用户需求**：用户要借助系统完成什么任务？
站在用户任务角度描述，通常可以写成用例或用户故事
看到”某类用户可以完成某项任务“，通常是用户需求

**系统级需求**：为支持用户任务，系统具体要做什么？输入、处理、输出、异常、约束是什么？
站在软件系统行为角度描述，比用户需求更细、更可测试
看到“系统应当在某条件下执行某操作，并产生某结果”，通常是系统级需求

### 需求的类型

**功能需求**
描述系统应该提供什么服务、执行什么行为
> 系统应允许...；系统应能够...；当...时，系统应....

功能需求最适合用用例图、系统顺序图、测试用例展开

**数据需求**
数据需求描述系统需要保存、处理或展示哪些数据
数据需求经常对应分析类图、ERD或数据字典

**质量需求/非功能需求**
质量需求描述系统“做的怎么样”，不是做什么。
常见质量属性包括性能、可靠性、安全性、可用性、可维护性、可移植性等

非功能需求也必须可验证

**约束需求**
约束需求限制系统开发、部署、运行或技术选择
约束需求不是系统功能，但会强烈影响设计和实现

**外部接口需求**
外部接口需求描述系统与外部人、设备、系统之间如何交互

**业务规则**
业务规则是问题域中的规则，软件必须遵守
业务规则常常会转化成功能需求、状态图条件或测试用例

## 第六章 需求建模与绘图

### 用例图
核心元素：
参与者 Actor：系统外部的人、组织、设备或外部系统
用例 Use Case：参与者希望通过系统完成的目标
系统边界：表示哪些功能属于本系统
关系：参与者与用例的关联、参与者泛化、用例包含include、扩展extend

### 分析类图/概念类图
分析类图，也叫领域类图、概念类图，描述问题域中的重要概念、属性和关系

概念类图只有属性、没有方法
在需求分析中，对象应代表现实世界事物；视窗、按钮、算法、连接、数据库等软件设计对象不应作为需求分析对象出现。对象模型以对象和类为基础，描述系统中的对象及其关系。

系统顺序图SSD把系统当作黑盒，只描述：
外部参与者向系统发送了哪些系统事件，系统返回了哪些结果
系统顺序图常用于把用户需求细化为系统级需求。

状态图描述一个对象在生命周期中经历的状态，以及事务触发的状态迁移

## 第七章 需求规格说明与测试用例

### 为什么需要需求规格说明

**沟通作用**
让客户、用户、开发人员、测试人员对系统要做什么达成一致

**契约作用**
需求规格说明可以作为验收和合同基础，避免后期争论“这是不是原本要做的”

**设计依据**
体系结构设计、详细设计和接口设计都要依据需求

**测试依据**
测试人员要根据需求设计测试用例，验证系统是否满足要求

**变更控制依据**
需求文档形成基线后，变更需要分析影响、审批和跟踪

**维护依据**
后期维护人员需要通过需求文档理解系统为什么这样设计

### 好的需求规格说明有哪些特性：
正确性、无歧义、完备性、一致性、按重要性和稳定性分级、可验证、可修改、可跟踪

正确：需求真实反映用户需要。  
无歧义：每句话只能有一种解释。  
完备：正常流程、异常流程、边界条件都写到。  
一致：不同需求之间不能冲突，高层与低层也不能矛盾；课件中特别指出系统级需求不能和业务需求、用户需求冲突，同层需求也不能互相冲突。  
可验证：可以通过测试、审查或分析判断是否满足。  
可跟踪：能追溯来源，也能追踪到设计、代码、测试。

**常见需求错误与修正**
**模糊词**：`系统应快速返回查询结果`
“快速”不可验证
修改：系统应在普通网络环境下，在2秒内返回前50条匹配查询结果

**歧义**：`系统应现实最近的借书记录`
“最近”有歧义
修改：系统应按借书时间倒序显示该借书人最近10条借书记录

**不完整**：`用户输入PIN后系统进行验证`
没有说明成功、失败、失败次数、异常情况
修改：用户输入PIN后，系统应向安全系统发送验证请求，若验证成功，系统进入交易选择界面；若验证失败，系统提示错误并允许重新输入；连续失败3次后，系统应终止本次交易并锁定银行卡交易功能

**混入设计实现** ： `系统应使用Redis缓存用户登录信息，以便快速验证用户身份`
如果 Redis 是强制技术约束，可以写在约束需求；如果只是实现方案，不应混入功能需求。
修改：
功能需求：系统应在用户登录后维护其会话状态，并在用户访问受保护功能时验证登录状态。
约束需求：系统会话管理应使用学校指定的 Redis 服务。

**多个需求混在一句话中** ： `系统应支持用户登录、修改密码、查看订单、导出报表，并且界面友好`
多个功能 + 一个模糊质量需求混杂。

**不可实现或现实** ： `系统应保证永远不出错`
修改：系统应记录运行异常日志，并在发生不可恢复错误时向用户显示错误提示，同时保留错误编号供管理员排查。

### 如何根据需求设计功能测试用例

系统测试的功能测试以需求规格说明或用例文档为基础。
主要使用随机测试和基于规格的测试技术设计功能测试用例，非功能需求则需要使用对应的专门测试技术

等价类划分：把输入分为有效类和无效类。  
边界值分析：测边界附近的数据。  
决策表：适合多条件组合。  
状态迁移测试：适合有状态对象。  
用例场景测试：覆盖基本流程和扩展流程。

## 第八章 软件设计基础

**软件设计** 是在需求规格说明的基础上，为即将构造的软件系统制定结构、组成部分、接口、数据、行为和约束的过程。它把“系统应该做什么”的需求，转换为“系统应该如何组织和实现”的软件构建方案

### 软件设计的核心思想
分解、抽象、信息隐藏、关注点分离

**分解**
把大系统拆成较小的子系统、模块、类、函数，使每部分可以独立理解、开发、测试和维护

**抽象**
抽象是隐藏细节，只暴露关键特征和接口

**信息隐藏**
信息隐藏强调模块应隐藏容易变化的设计决策，只暴露稳定接口

**关注点分离**
不同问题不要混在一起
例如界面显示、业务规则、数据存储、权限验证、日志记录应尽量分离

### 软件工程设计的三个层次
体系结构设计、详细设计、人机交互设计

**体系结构设计**
整体功能组织与质量属性

关注：系统整体如何分解？各大模块如何组织？模块之间如何通信？如何满足性能、安全、可靠性、可维护性等质量属性？
系统的大骨架是什么？

**详细设计**
模块内部结构及质量
关注：每个模块内部如何设计？类怎么划分？职责如何分配？方法如何写作？数据结构和算法如何选择？
每个模块里面具体怎么组织

**人机交互设计**
易用性并产出界面原型
关注：用户如何高效、清晰、低错误地完成任务
用户这样使用系统

## 第九、十章 软件体系结构
**体系结构的概念**
体系结构是对软件系统高层结构的描述，它包括系统的主要部件、部件之间的连接件、部件组织配置方式，以及这些结构决策背后的设计理由。
体系结构决定了系统的功能组织、模块边界、部署方式和关键质量属性，如性能、安全性、可维护性、可扩展性、可靠性等

### 体系结构的风格的优缺点
#### 分层体系结构
按系统抽象层次分为若干层
表现层：负责界面、请求接收、输入输出
业务逻辑层：负责业务规则、流程控制、事务处理
数据层：负责数据访问、持久化、数据库操作

**优点**
结构清晰，容易理解
每层职责明确，便于分工
上层不直接依赖底层实现，便于置换
便于测试
适合大多数企业信息系统和Web应用

**缺点**
层间调用有额外开销
如果层次划分过死，简单功能也要穿越多层，开发繁琐
容易出现“贫血业务层”，即业务逻辑散落在Controller 或 DAO 中
如果允许跨层调用，会破坏结构

#### MVC风格
MVC 将系统分为 Model、View、Controller

Model：业务数据和业务逻辑
View：界面显示
Controller：处理用户输入，调用Model，选择View

**优点**
界面和业务逻辑分离
多个视图可以复用同一模型
便于前后端分工
适合交互式应用

**缺点**
小项目中结构可能显得复杂
如果Controller 过重，会变成 God Controller
如果Model 边界不清，会导致业务逻辑分散

MVC 和分层架构经常结合使用，比如前端页面是 View，后端 Controller 是表现层入口，Service 是业务层，DAO 是数据层

#### 主程序 - 子程序风格
主程序控制整体流程，子程序完成局部功能
```
main()
 ├── readInput()
 ├── process()
 └── output()
```
**优点**
控制流程清晰
容易实现
适合规模较小、流程固定的程序

**缺点**
主程序容易过重
模块之间常通过共享数据耦合
扩展性较差
不适合复杂交互和长期演化系统

#### 管道-过滤器风格
系统由一系列过滤器组成，每个过滤器处理输入并输出给下一个过滤器

**优点**
每个过滤器职责单一
可复用、可替换
适合批处理和数据流处理

**缺点**
不适合强交互系统
如果中间数据格式不统一，集成成本高
全局状态处理困难

### 仓库风格
多个部件共享一个中心数据仓库
```
模块A ↘
模块B → 中央仓库
模块C ↗
```

**优点**
数据集中管理
多个工具或模块可共享数据
便于一致性维护

**缺点**
中心仓库容易成为性能瓶颈
仓库模式一变，所有模块都受影响
模块之间可能通过共享数据产生隐式耦合

#### 客户端-服务器风格 
客户端发起请求，服务器提供服务。
Web 心态就是典型的 C/S 或 B/S

**优点**
职责清晰
服务器集中管理数据和业务规则
客户端可以轻量化
适合网络应用

**缺点**
依赖网络
服务器可能成为瓶颈
接口设计和安全机制要求高

#### 事件驱动风格
组件不直接调用彼此，而是通过事件发布和订阅协作
```
订单创建事件 → 库存模块订阅
             → 积分模块订阅
             → 通知模块订阅
```

**优点**
模块解耦
扩展方便，新增订阅者不影响发布者
适合异步处理和复杂业务联动

**缺点**
控制流程不直观
调试困难
事件顺序、重复消费、失败补偿复杂

#### 微服务风格
系统被拆分为多个独立服务，每个服务独立部署、独立数据库或数据边界

**优点**
服务可独立开发、部署、扩展
适合大型团队
局部故障隔离更好
技术栈可异构

**缺点**
分布式复杂性高
网络调用、服务发现、配置管理、日志追踪、数据一致性都更复杂
小团队和小项目容易过度设计

### 体系结构设计过程

分析 Analysis -> 综合 Synthesis -> 评价 Evaluation

分析 收集架构重要需求和利益相关者的关注点
综合 构造候选架构并权衡
评价 验证方案是否满足架构重要需求

#### Analysis 分析架构需求
识别系统类型、利益相关者、架构重要需求ASR、关键质量属性、约束条件

#### Synthesis 综合候选架构
选择架构风格
划分模块和包
定义接口
确定部署结构
确定依赖方向
选择关键技术

#### Evaluation 评价架构
检查架构是否满足质量需求
检查依赖方向是否合理
检查接口是否清晰
检查风险点
设计集成测试验证架构连接

### 包的原则

包设计的核心目标是：
高内聚、低耦合、稳定依赖、可发布、可复用、可维护
#### 包内聚原则
**共同复用原则 CRP**
一起被复用的泪应该放在同一个包里；不会一起复用的类不要强行放在一起

**共同闭包原则 CCP**
会因为同一种变化原因而在一起变化的类，应该放在同一个包里

**复用发布等价原则 REP**
一个包应该是可发布、可版本化的复用单位
#### 包耦合原则
**无环依赖原则 ADP**
包之间不能形成循环依赖

**稳定依赖原则 SDP**
不稳定的包应该依赖稳定的包，而不是不稳定包依赖不稳定包

**稳定抽象原则 SAP**
越稳定的包越应该抽象，稳定包中应放接口、抽象类、核心模型，而不是频繁变化的具体实现

### 体系结构构件之间接口的定义
核心原则：
- 展示层不能直接访问数据库
- 展示层通过 Controller 或 Service Interface 调用业务逻辑
- 业务层通过 DAO/Repository Interface 调用数据层
- 接口应放在稳定包中，实现类放在 impl 包中
- 接口参数和返回值要使用 DTO/VO/PO 区分
