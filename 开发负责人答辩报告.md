
## 1. 开发负责部分概述  
  
我负责的是 CampusHub 校园互助服务平台的开发实现部分，主要目标是把前期需求分析和架构设计落到可以运行、可以演示、可以维护的系统功能中。  
  
开发工作主要围绕一个核心业务闭环展开：  
  
```text  
用户登录  
  -> 浏览需求  -> 发布任务  -> 查看详情  -> 接单  -> 确认完成  -> 双向评价  -> 消息通知和积分反馈  
```  
  
这个流程对应校园互助平台最核心的使用场景：一个学生发布校园互助需求，另一个学生接单，双方完成任务后进行评价，系统记录积分和消息反馈。  
  
除了主流程外，开发部分还实现了话题广场、评论回复、点赞收藏、搜索、消息中心、校园认证、反馈、排行榜和管理后台等辅助功能，保证平台不是一个单纯的任务列表，而是具备互动、治理和反馈能力的校园互助系统。  
  
从技术实现上看，项目采用前后端分离结构：  
  
- 前端使用 Vue 3、Vite、TypeScript、Vue Router、Axios 和 UnoCSS。  
- 后端使用 Spring Boot、Java 17、MyBatis、MySQL、Redis、JWT 和 WebSocket。  
- 前端负责页面交互和状态展示。  
- 后端负责接口、业务规则、数据读写、权限校验和消息通知。  
  
开发负责人部分的重点不是简单介绍写了哪些页面，而是说明系统如何通过前端页面、接口封装、后端业务层和数据库状态共同支撑完整业务流程。

## 2. 整体流程和主要业务  
  
### 2.1 平台主要业务  
  
CampusHub 的主要业务可以分为六类：  
  
| 业务模块 | 说明 |  
|---|---|  
| 用户认证 | 用户注册、登录、Token 鉴权、用户资料维护 |  
| 任务互助 | 发布任务、浏览任务、接单、取消接单、完成确认 |  
| 社区互动 | 话题帖、评论、回复、点赞、收藏 |  
| 评价积分 | 任务完成后评价，记录积分和信用反馈 |  
| 消息通知 | 接单、完成、评论、认证等事件触发消息或通知 |  
| 管理治理 | 管理员审核内容、管理用户、处理反馈和校园认证 |  
  
其中最核心的是任务互助流程。其他模块围绕这个主流程提供支撑：  
  
- 评论用于任务沟通。  
- 消息用于状态提醒。  
- 评价和积分用于建立信任。  
- 校园认证用于提高用户可信度。  
- 管理后台用于平台治理。  
  
### 2.2 主业务流程  
  
主业务流程如下：  
  
```text  
学生 A 登录系统  
  -> 学生 A 发布一个互助任务  -> 任务进入首页列表  -> 学生 B 浏览任务列表  -> 学生 B 进入任务详情并接单  -> 系统更新任务状态并通知学生 A  -> 双方完成线下互助  -> 学生 A 或学生 B 确认完成  -> 双方进入评价页  -> 系统保存评价、积分记录并发送消息  
```  
  
这个流程中需要重点处理几个业务规则：  
  
1. 用户必须登录后才能发布、接单、评论和评价。  
2. 用户不能接自己发布的任务。  
3. 已经被接单的任务不能重复接单。  
4. 只有任务相关用户可以确认完成。  
5. 任务完成后才能进行评价。  
6. 同一用户不能对同一任务重复评价。  
7. 接单、完成、评价等动作需要触发消息或积分反馈。  
  
开发时，前端负责让用户清楚地看到当前状态，后端负责真正校验状态和权限，避免只依赖前端按钮控制造成安全问题。

## 3. 前端主要结构  
  
前端代码位于 `frontend/src` 目录，整体按页面、组件、接口、工具函数和路由组织。

```text  
frontend/src/  
  main.ts  App.vue  router/index.ts  services/api.ts  pages/  pages/admin/  pages/settings/  components/  composables/  utils/  style.css  
```  
  
各部分职责如下：  
  
| 文件或目录             | 作用                         |     |

| `App.vue`         | 根组件，承载路由出口和全局组件            |     |
| `router/index.ts` | 路由配置、登录守卫、管理员守卫            |     |
| `services/api.ts` | Axios 实例和所有业务接口封装          |     |
| `pages/`          | 学生端主要页面                    |     |
| `pages/admin/`    | 管理后台页面                     |     |
| `pages/settings/` | 设置相关子页面                    |     |
| `components/`     | 顶部导航、底部导航、弹窗、Toast、上传等通用组件 |     |
| `composables/`    | Toast、确认弹窗、偏好设置等组合式逻辑      |     |

| `style.css`       | 全局样式                       |     |

### 3.3 前端路由和权限控制  
  
前端路由在 `frontend/src/router/index.ts` 中定义。  
  
路由中通过 `meta.requiresAuth` 标记需要登录的页面，通过 `meta.requiresAdmin` 标记需要管理员身份的页面。  
  
典型逻辑包括：  
  
```text  
未登录用户访问 /home  
  -> 跳转到 /auth?tab=login  
普通用户访问 /admin  
  -> 跳转到 /home  
管理员登录后访问 /home  
  -> 跳转到 /admin/community  
```  
  
这样可以在前端层面先过滤明显不合法的访问，提高用户体验。真正的权限安全仍然由后端接口校验。  
  
### 3.4 前端接口封装  
  
前端所有接口集中封装在 `frontend/src/services/api.ts`。  
  
这个文件主要做了几件事：  
  
1. 创建 Axios 实例。  
2. 自动在请求头中携带 JWT。  
3. 处理 401 错误并尝试刷新 Token。  
4. 为文件上传创建独立的 `uploadApi`。  
5. 按业务模块封装 API。  
  
主要 API 分组包括：  
  
| API 分组 | 功能 |  
|---|---|  
| `authApi` | 登录、注册、刷新 Token |  
| `taskApi` | 任务查询、发布、接单、完成、评论、评价、收藏 |  
| `userApi` | 当前用户、用户设置、积分记录、排行榜 |  
| `messageApi` | 消息列表、发送消息、标记已读、未读数 |  
| `notificationApi` | 通知列表、通知已读 |  
| `verificationApi` | 校园认证提交和审核 |  
| `feedbackApi` | 提交反馈和查看我的反馈 |  
| `adminApi` | 管理后台用户、任务、反馈、公告和认证接口 |  
| `searchApi` | 搜索 |  
| `fileApi` | 文件上传 |  
  
这种集中封装方式可以减少页面中的重复代码。页面只需要调用对应 API 方法，不需要每个页面单独处理请求路径、Token 和错误逻辑。

## 4. 后端主要结构  
  
后端代码位于 `backend/src/main/java/com/campushub`，采用 Controller、Service、Mapper、Entity、DTO、Config、Util 分层结构。  
  
### 4.1 后端目录结构  
  
```text  
backend/src/main/java/com/campushub/  
  CampusHubApplication.java  controller/  service/  mapper/  entity/  dto/  config/  util/  
```  
  
各层职责如下：  
  
| 层次 | 职责 |  
|---|---|  
| Controller | 接收 HTTP 请求，解析参数，获取当前登录用户 |  
| Service | 处理业务规则、权限校验、状态流转、消息通知和积分逻辑 |  
| Mapper | 通过 MyBatis 访问数据库 |  
| Entity | 对应数据库表结构 |  
| DTO | 定义接口请求体、查询参数和响应视图 |  
| Config | 配置安全、Redis、WebSocket、MyBatis 等基础能力 |  
| Util | JWT、密码、任务模式等工具类 |  
  
后端开发时，我们尽量让 Controller 保持轻量，把复杂业务逻辑放在 Service 层。这样做的好处是结构清晰，也方便后续测试和维护。  
  
### 4.2 Controller 层  
  
主要 Controller 包括：  
  
| 文件 | 功能 |  
|---|---|  
| `AuthController` | 登录、注册、刷新 Token |  
| `UserController` | 当前用户、用户设置、积分、排行榜 |  
| `TaskController` | 任务列表、详情、发布、接单、完成、取消、删除、点赞、收藏 |  
| `TaskCommentController` | 评论、回复、评论点赞、删除评论 |  
| `TaskReviewController` | 任务评价 |  
| `MessageController` | 消息列表、发送消息、标记已读、未读数 |  
| `NotificationController` | 通知列表、通知已读 |  
| `VerificationController` | 用户校园认证 |  
| `FeedbackController` | 用户反馈 |  
| `AnnouncementController` | 公告列表 |  
| `AdminController` | 后台管理、内容审核、认证审核、反馈处理 |  
| `FileController` | 文件上传和访问 |  
| `SearchController` | 搜索 |  
| `GlobalExceptionHandler` | 全局异常处理 |  
  
Controller 层主要负责接口入口，不直接写复杂业务规则。例如任务接单接口只接收请求并传入当前用户 id，真正判断能不能接单是在 `TaskService.acceptTask` 中完成的。  
  
### 4.3 Service 层  
  
主要 Service 包括：  
  
| 文件 | 功能 |  
|---|---|  
| `AuthService` | 登录注册、密码校验、Token 生成 |  
| `UserService` | 用户资料、设置、积分和信用分 |  
| `TaskService` | 任务和话题核心业务 |  
| `TaskRecommendationService` | 智能推荐和匹配理由 |  
| `TaskCommentService` | 评论、回复和评论点赞 |  
| `TaskReviewService` | 任务评价和防重复评价 |  
| `MessageService` | 私信、系统消息和批量已读 |  
| `NotificationService` | 通知生成和已读状态 |  
| `VerificationService` | 校园认证提交、审核和撤销 |  
| `AdminService` | 管理员权限、用户管理、任务审核和后台统计 |  
| `FeedbackService` | 反馈提交和处理 |  
| `AnnouncementService` | 公告查询和创建 |  
  
Service 层是后端开发的重点。例如 `TaskService` 中处理任务发布、查询、接单、取消、完成、点赞、收藏等核心逻辑；`TaskReviewService` 负责评价权限和重复评价校验；`MessageService` 负责在业务事件发生时创建消息。

Mapper 负责数据库访问，复杂 SQL 位于 `backend/src/main/resources/mapper`。

| `Task` | `tasks` | 任务和话题主体 |  
| `TaskParticipant` | `task_participants` | 接单参与记录 |  
| `TaskComment` | `task_comments` | 评论和回复 |

| `TaskFavorite` | `task_favorites` | 收藏 |  
| `TaskReview` | `task_reviews` | 任务评价 |  
| `Message` | `messages` | 私信和系统消息 |  
| `Notification` | `notifications` | 通知 |  
| `UserVerification` | `user_verifications` | 校园认证 |  
| `Feedback` | `feedback` | 用户反馈 |  
| `Announcement` | `announcements` | 公告 |  
  
数据库结构定义在 `backend/src/main/resources/schema.sql`。任务和话题共用 `tasks` 表，通过 `task_mode` 区分普通任务和话题帖。

后端配置类包括：  
  
| 文件 | 功能 |

```text

```

## 5. 主要业务流程实现内容和流程  
  
### 5.1 登录注册流程  
  
前端入口：  
  
- `Auth.vue`  
- `authApi.login`  
- `authApi.register`  
  
后端入口：  
  
- `AuthController`  
- `AuthService`  
- `UserMapper`  
- `PasswordUtil`  
- `JwtUtil`  
  
登录流程：  
  
```text  
用户输入学号和密码  
  -> 前端调用 POST /auth/login  -> 后端查询用户  -> 校验密码  -> 生成 accessToken 和 refreshToken  -> 返回用户信息和 Token  -> 前端保存 Token 并跳转首页  
```  
  
注册流程：  
  
```text  
用户填写学号、姓名、邮箱、密码  
  -> 前端调用 POST /auth/register  -> 后端校验学号和邮箱是否重复  -> 加密密码  -> 创建用户  -> 初始化用户设置  -> 返回登录状态  
```  
  
登录注册是所有核心业务的前提，后续发布、接单、评论和评价都依赖当前登录用户身份。  
  
### 5.2 发布任务流程  
  
前端入口：  
  
- `Publish.vue`  
- `taskApi.createTask`  
- `fileApi.upload`  
  
后端入口：  
  
- `TaskController.createTask`  
- `TaskService.createTask`  
- `TaskMapper`  
- `TaskCreateRequest`  
  
流程：  
  
```text  
用户进入发布页  
  -> 填写标题、描述、分类、地点、时间等信息  -> 可选上传图片  -> 前端调用 POST /tasks  -> 后端校验当前用户  -> 判断 taskMode 是任务还是话题  -> 写入 tasks 表  -> 返回创建结果  
```  
  
发布时，普通互助任务和话题帖共用同一套创建逻辑，通过 `taskMode` 区分。这样减少了重复建模，也方便首页和话题广场统一处理内容。  
  
### 5.3 浏览任务和智能推荐流程  
  
前端入口：  
  
- `HomeGateway.vue`  
- `Discovery.vue`  
- `TopicSquare.vue`  
- `taskApi.getTasks`  
  
后端入口：  
  
- `TaskController.getTasks`  
- `TaskService.getTasks`  
- `TaskRecommendationService`  
- `TaskMapper`  
  
流程：  
  
```text  
用户进入首页  
  -> 前端根据 Tab 和筛选条件请求任务列表  -> GET /tasks 携带 mode、category、location、page、size、taskMode 等参数  -> 后端进行分页和条件查询  -> 推荐模式下计算匹配分和推荐理由  -> 返回任务列表  -> 前端展示卡片、匹配度、评论数、点赞数和收藏状态  
```  
  
任务列表支持：  
  
- 最新发布。  
- 智能推荐。  
- 分类筛选。  
- 地点筛选。  
- 时间筛选。  
- 任务和话题模式切换。  
- 服务端分页。  
  
推荐逻辑不是简单排序，而是结合分类、位置、时间、用户历史偏好和新鲜度进行综合打分。  
  
### 5.4 任务详情、接单和完成流程  
  
前端入口：  
  
- `RequestDetail.vue`  
- `taskApi.getTaskById`  
- `taskApi.acceptTask`  
- `taskApi.completeTask`  
- `taskApi.cancelTask`  
  
后端入口：  
  
- `TaskController`  
- `TaskService`  
- `TaskParticipantMapper`  
- `MessageService`  
- `UserService`  
  
查看详情流程：  
  
```text  
用户点击任务卡片  
  -> 前端进入 /detail/:id  -> 调用 GET /tasks/{id}  -> 后端查询任务详情、发布者、参与者、点赞收藏状态  -> 前端根据任务状态展示接单、完成、评价等操作按钮  
```  
  
接单流程：  
  
```text  
用户点击接单  
  -> 前端调用 POST /tasks/{id}/accept  -> 后端校验任务存在  -> 校验不能接自己的任务  -> 校验任务未被接单  -> 写入 task_participants  -> 更新任务状态  -> 发送任务消息  -> 返回更新后的任务  
```  
  
完成流程：  
  
```text  
任务相关用户点击确认完成  
  -> 前端调用 POST /tasks/{id}/complete  -> 后端校验当前用户是否为发布者或接单者  -> 记录完成时间  -> 双方完成后更新任务完成状态  -> 增加积分或生成积分记录  -> 发送消息通知  
```  
  
这一部分是开发实现的核心，因为它不仅是接口调用，还涉及任务状态、参与关系、权限校验、消息通知和积分记录的联动。  
  
### 5.5 评价流程  
  
前端入口：  
  
- `TaskReviewPage.vue`  
- `taskApi.getTaskReviews`  
- `taskApi.createTaskReview`  
  
后端入口：  
  
- `TaskReviewController`  
- `TaskReviewService`  
- `TaskReviewMapper`  
- `UserService`  
  
流程：  
  
```text  
任务完成后进入评价页  
  -> 前端获取已有评价  -> 用户提交评分和评价内容  -> 后端校验任务是否已完成  -> 校验评价人是否为任务相关用户  -> 校验是否已经评价过  -> 写入 task_reviews  -> 更新积分或信用反馈  
```  
  
评价模块的意义在于建立平台信任机制。校园互助涉及陌生同学之间协作，完成后的互评可以让后续用户参考历史信用。  
  
### 5.6 消息通知流程  
  
前端入口：  
  
- `Messages.vue`  
- `Notifications.vue`  
- `messageApi`  
- `notificationApi`  
  
后端入口：  
  
- `MessageController`  
- `MessageService`  
- `NotificationController`  
- `NotificationService`  
  
消息流程：  
  
```text  
业务事件发生  
  -> Service 创建消息或通知  -> 写入 messages 或 notifications  -> 前端消息页轮询未读数  -> 未读数变化时拉取消息列表  -> 用户查看后批量标记已读  
```  
  
消息模块主要用于提升流程反馈，例如：  
  
- 接单后通知发布者。  
- 完成后通知相关用户。  
- 评论或回复后通知被回复用户。  
- 认证审核后通知申请人。  
  
消息页采用未读数轮询和批量已读，避免频繁拉取完整列表和逐条发送已读请求。

## 7. 前后端联调与代码规范  
  
### 7.1 前后端联调方式  
  
前后端联调主要通过 `services/api.ts` 统一完成。  
  
联调流程通常是：  
  
```text  
后端定义接口  
  -> 前端在 api.ts 中封装方法  -> 页面调用 API 方法  -> 根据接口返回更新页面状态  -> 处理成功、失败和加载状态  
```  
  
例如接单功能的联调：  
  
```text  
RequestDetail.vue  
  -> taskApi.acceptTask(taskId)  -> POST /tasks/{id}/accept  -> TaskController.acceptTask  -> TaskService.acceptTask  -> 返回更新后的 Task  -> 前端刷新详情状态  
```  
  
### 7.2 Token 和错误处理  
  
前端 Axios 请求拦截器会自动携带 JWT：  
  
```text  
Authorization: Bearer <token>  
```  
  
如果接口返回 401，前端会尝试调用刷新 Token 接口。如果刷新失败，则清空登录信息并跳转登录页。  
  
后端通过 `JwtAuthenticationFilter` 校验 Token，并将用户 id 放入 Spring Security 的 `Authentication` 中。Controller 可以通过 `authentication.getName()` 获取当前用户。  
  
错误处理方面，后端通过 `GlobalExceptionHandler` 统一返回错误信息，前端通过 Toast 或页面提示反馈给用户。  
  
### 7.3 前端代码规范  
  
前端代码规范主要体现在：  
  
- 页面负责展示和交互，不直接写底层请求逻辑。  
- API 统一放在 `services/api.ts`。  
- 通用交互使用组件或 composable，例如 Toast、确认弹窗。  
- 登录状态判断放在 `utils/auth.ts`。  
- 路由权限集中写在 `router/index.ts`。  
- 页面根据任务状态展示不同操作，减少用户误操作。  
  
### 7.4 后端代码规范  
  
后端代码规范主要体现在：  
  
- Controller 保持轻量。  
- Service 负责业务规则。  
- Mapper 负责数据库访问。  
- DTO 用于请求体和查询参数。  
- Entity 对应数据库表。  
- 统一异常处理。  
- 密码、JWT、任务模式等通用逻辑抽成工具类。  
  
例如任务接单逻辑不会只在前端按钮上限制，而是在 `TaskService.acceptTask` 中进行后端校验。这样即使用户绕过前端直接调用接口，后端也能保证业务规则正确。  
  
### 7.5 可维护性考虑  
  
为了方便后续维护，开发中做了以下设计：  
  
1. 任务和话题复用同一张表，通过 `taskMode` 区分。  
2. 接口统一封装，页面不直接散落请求路径。  
3. 状态流转集中在 Service 中。  
4. 消息、通知、积分等副作用由业务层统一触发。  
5. 管理端和学生端路由分离。  
6. 异常和权限校验在后端统一处理。

## 8. AI 工具的使用  
  
### 8.1 AI 在开发中的定位  
  
在开发阶段，我们对 AI 的定位不是“代替开发者写完整系统”，而是作为辅助工具参与开发流程。  
  
我们使用 AI 的基本流程是：  
  
```text  
提出具体问题  
  -> 提供代码上下文或业务约束  -> 让 AI 给出建议或初稿  -> 人工审查和修改  -> 本地运行和测试验证  -> 将有效内容合并到项目  
```  
  
也就是说，AI 可以提高效率，但最终代码是否采用，需要由团队根据业务需求、代码结构和测试结果判断。  
  
### 8.2 AI 辅助的内容  
  
开发阶段 AI 主要帮助了以下几类工作：  
  
| 使用场景 | AI 的作用 | 人工处理 |  
|---|---|---|  
| 代码初稿 | 辅助生成 Controller、Service、DTO 或前端页面片段 | 按项目结构修改字段、接口和业务规则 |  
| 接口设计 | 帮助整理接口路径、请求体和返回字段 | 结合实际前端页面和数据库结构调整 |  
| Bug 调试 | 根据报错信息提供排查方向 | 本地复现、修改代码、重新运行测试 |  
| 异常流程 | 提醒可能遗漏的边界情况 | 转化为后端校验和测试用例 |  
| 文档整理 | 辅助总结模块职责和调用链 | 人工核对代码，删除不准确内容 |  
  
### 8.3 AI 使用中的问题  
  
AI 对开发有帮助，但也存在局限：  
  
1. 容易生成看起来合理但不符合现有代码结构的代码。  
2. 可能使用项目中不存在的字段或接口。  
3. 对业务状态流转理解不完整。  
4. 容易遗漏异常流程。  
5. 不能替代运行结果和自动化测试。  
6. 有时会给出过度设计方案，增加开发复杂度。  
  
因此我们采用的原则是：  
  
> AI 可以给建议和初稿，但不能直接作为最终代码依据；所有 AI 输出必须经过人工审查、联调和测试验证。  
  
### 8.4 AI 在开发负责人部分的价值  
  
从开发负责人的角度看，AI 最大的价值主要有三点：  
  
第一，提升初稿效率。    
例如在设计接口、整理页面结构或编写文档时，AI 可以快速给出初始版本，减少从零开始的时间。  
  
第二，补充异常场景。    
例如任务接单、重复评价、权限校验等场景中，AI 可以提示可能遗漏的边界情况，帮助我们完善业务规则。  
  
第三，辅助调试和解释。    
当接口联调或测试失败时，AI 可以帮助解释错误含义、给出排查顺序，但最终仍需要通过本地运行和测试结果确认。  
  
### 8.5 答辩中的总结表达  
  
答辩时可以这样总结 AI 使用：  
  
> 在开发阶段，我们把 AI 当作编码助手、调试助手和文档整理助手，而不是代码负责人。AI 能帮助我们提高初稿效率、补充异常场景和分析报错，但它不能替代人工判断和工程验证。所有 AI 生成或建议的内容，都需要经过人工审查、前后端联调、自动化测试和构建检查后，才能作为项目成果的一部分。

## 9. 开发负责部分总结  
  
开发负责部分的核心成果可以概括为三点：  
  
第一，打通了完整主流程。    
系统已经支持用户登录、发布任务、浏览任务、接单、完成确认、评价、消息通知和积分反馈，形成校园互助的完整业务闭环。  
  
第二，形成了清晰的前后端结构。    
前端按页面、组件、API、工具函数组织，后端按 Controller、Service、Mapper、Entity、DTO 分层组织，结构清晰，方便后续维护。  
  
第三，保证了联调和规则一致性。    
前端通过 `api.ts` 统一调用接口，后端通过 Service 层集中处理业务规则，避免只依赖前端控制状态。任务状态、参与关系、评价、消息和积分之间能够联动。  
  
最终，开发工作的目标不是只做出若干页面，而是把需求转化为可以演示、可以测试、可以继续扩展的系统功能。
