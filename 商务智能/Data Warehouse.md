## 从数据库到数据仓库

### 为什么从数据库讲起
关系数据模型推动了数据库和OLTP的发展，数据库被大量用于商业管理、政府办公、科研、工程开发等场景；同时，数据库中的数据量也从MB、GB发展到TB、PB级
这说明 **企业已经积累了大量数据，但这些数据最初主要是为日常业务处理服务的，而不是为决策分析服务的**

### 操作型处理与分析型处理

**操作型处理**，也就是事物处理，既 OLTP
面向日常业务，通常只是对一个或一组记录进行查询或修改
操作简单、执行频率高、并发多、对响应时间和数据安全完整性要求高

**分析型处理**，也叫信息型处理，主要服务企业管理者的决策分析
需要对大量事物数据进行统计、归纳和分析，需要访问大量历史数据，执行频率不一定高，对实时响应时间要求没有OLTP那么严苛
典型分析型应用是 DSS ， 即决策支持系统

### 传统数据库环境为什么不适合做DSS
虽然数据库技术在事务处理方面非常成功，但直接在事务处理环境中建立分析型应用并不合适

**性能特性不同**
事务处理每次访问的数据量小，但频率高、并发多；
分析处理每次访问的数据量大，可能连续运行很久，占用大量资源
如果把复杂分析直接放在业务数据库上，就可能出现
```
管理者跑一个年度销售分析
        ↓
扫描大量订单和明细表
        ↓
业务系统变慢
        ↓
收银、下单、库存更新受影响
```

**数据集成问题**
分析处理需要全面、正确的数据，可能来自企业内部不同部门，也可能来自外部、竞争对手、互联网、文件系统、异构数据库等
事务处理系统通常只关注本部门当前业务，导致数据分散、不一致

例如：
```
销售系统叫“客户编号”
客服系统叫“会员ID”
财务系统叫“客户代码”
```
它们可能指同一个人，也可能规则不同。直接分析会非常混乱

**数据动态集成问题**
静态集成上一次性集成数据，以后不再变化
动态集成则需要周期性刷新
传统事务处理环境不具备良好的动态集成能力，如果数据源变化不能及时反映到分析系统，决策者看到的就是过时数据

**历史数据问题**
事务处理通常只关心当前数据，数据库里常常只保存短期数据，比如3-6个月
；而分析处理更看重5-10年的历史数据。
没有历史数据，决策分析就缺少趋势判断基础
```
事务系统关心：今天库存够不够
分析系统关心：过去五年春节前库存波动规律是什么？
```

**数据综合问题**
事务系统积累大量细节数据，但分析处理往往需要总结性数据。细节数据过多会影响效率，也会让分析人员难以聚焦有用信息
“数据丰富、信息贫困”

**数据访问问题**
事务处理需要多种访问操作，特别是实时更新
分析处理则以读操作为主，不需要实时更新，但需要定时刷新

### 数据仓库出现的根本原因
不能直接在事务处理环境中构建分析型应用，必须把分析型处理及其所需的综合性分析数据，从传统事务型处理和细节性操作数据中分离出来，重新组织成单独的分析处理环境。数据仓库正是为了建立这种新环境而出现的数据存储和组织技术

业务数据库负责“运行企业”
数据仓库负责“理解企业“

把数据仓库与操作型数据库分析，可以提高两个系统的性能，提高操作型数据库的事务吞吐量；
同时两者数据结构、内容和用法不同。数据仓库不是替代传统数据库，而是适应分析型处理需要

## 数据仓库的定义与四大特征
### 数据仓库的定义
数据仓库是一个面向主题的、集成的、非易失的、时变的数据集合，用于支持经营管理过程中的决策制定

数据仓库把企业中的不同信息岛上的商业数据集成到一起，存储在一个单一的集成关系型数据库中，方便用户访问，也方便决策人员分析历史数据、研究发展趋势

### 特征一：面向主题
传统操作型数据是 **面向应用** 的，比如采购子系统、销售子系统、库存子系统、人事子系统，各自围绕业务流程和单据组织数据，其特点是表达数据流程、对应业务单据、逻辑与数据不完全分离、和部门组织相关

而数据库是**面向主题**的。主题是企业信息系统中较高层次的分析对象，是为了分析和决策而抽象出来的数据组织中心
可以从原来的采购、销售、库存、人事等系统中抽取出三个主题：商品、供应商、顾客

例如商品主题包含：
```
商品固有信息：商品号、商品名、类别、颜色等
商品采购信息：商品号、供应商号、供应价、供应日期、供应量等
商品销售信息：商品号、顾客号、售价、销售日期、销售量等
商品库存信息：商品号、库房号、库存量、日期等
```
一个主题会从多个操作型系统中抽取与它有关的数据，并抛弃与分析无关的数据，形成完整一致的主题描述

**主题之间可以有重叠**
不同主题之间可能有重叠，但这种重叠是逻辑上的，不一定是物理存储上的；
是部分细节的重叠，不是统计信息的重叠；这种重叠可以反映主题之间的直接或间接联系

```
商品主题中有顾客购买商品的信息
顾客主题中也有顾客购物信息
供应商主题中也会涉及商品
```
可以通过重叠帮助分析 "商品 - 顾客 - 供应商" 之间的关系

**主题的物理存储**

两类物理存储方式：
多维数据库 MDDB ： 用多维数组存储数据
关系数据库：用一组关系表组织数据，同一主题的一组关系有公共关键字

目前常用的是关系数据库方式，但其中存放的不再只是原始业务细节，还包括经过一定程度综合的数据
以“商品”主题为例，它的公共码键是“商品号”。商品固有信息可以有细节表，也可以有综合表；采购信息、销售信息、库存信息也都可以既保留细节数据，又按月、季度、年、周等时间粒度形成综合数据。

### 特征二：集成
数据仓库中的数据为分析服务，分析需要从多种数据源获取数据，包括数据库、文件系统、互联网数据等
因此，数据仓库必须通过数据集成把这些数据整理成一致的数据环境

两类集成方法：统一 和 综合

**统一** 主要解决不一致问题：
```
同名异义：字段名一样，含义不同
异名同义：字段名不同，含义相同
单位不统一：元、万元、美元混用
字长不一致：客户编号有的 8 位，有的 12 位
编码不一致：性别用 M/F、0/1、男/女
```

**综合** 对原始数据进行统计、抽样、计算
```
日销售额
月采购总量
季度库存平均值
年度客户消费金额
```

集成是数据仓库区别于简单数据复制的核心
如果只是把多个系统的数据堆在一起，并没有统一口径，那还不能算真正的数据仓库

### 特征三：非易失、也叫稳定
数据仓库的数据与操作性数据环境隔离
数据仓库中的数据是经过抽取形成的分析型数据，主要用于企业决策分析，主要执行查询操作，一般不执行更新操作
稳定的数据环境有利于分析和决策

“非易失”不是说数据仓库永远不变，在新的分析决策需要出现时，可能需要新的数据抽取、插入和更新；
过时数据也可能被删除，只是相对于OLTP系统，数据仓库的存储管理简单很多

```
数据仓库不是像业务库那样频繁被事务更新。
它主要通过批量抽取、周期刷新、历史追加等方式变化。
```

### 特征四：时变
数据仓库不是只记录企业当前某个时点的信息，而是记录企业从过去某一时点到现在各阶段的信息，用于对企业发展历程和未来趋势进行定量分析和预测
因此，数据仓库中的数据通常带有时间属性，并按一定时间段统一更新
时变性包括
```
不断增加新的数据
不断删除旧的数据
更新与时间有关的综合数据
```

```
销售事实表中必须有销售日期
库存快照中必须有快照日期
客户信息中可能有生效日期和失效日期
```
时变性是后续“快照“”历史完整性“”渐变维度“的基础

## 数据仓库的基本结构和关键技术
### 数据仓库的关键技术
ETL：抽取、转换、装载(数据怎么进来)
存储和管理：数据怎么放、怎么管
数据访问和表现：用户怎么查、怎么分析、怎么展示
元数据：系统怎么知道这些数据从哪里来、代表什么、怎么转换

### 数据源
数据仓库的数据来自多个数据源，这些数据源可能格式不同、平台不同、物理位置不同；
可能包括大型关系数据库、对象数据库、桌面数据库、非格式化数据文件等

数据源是可以递归的，即数据仓库的数据源也可能是另一个数据仓库、数据集市，或者基于本数据仓库的分析型应用

### 数据抽取与数据刷新
**数据抽取**
把数据源中的数据通过网络抽取出来，经过加工、转换、综合后形成数据仓库中的数据

**数据刷新**
数据进入仓库一段时间后，对过时数据进行修正，保存不变数据，把原系统中的变化反映到仓库中。

刷新过程类似抽取，但刷新数据量通常小于抽取；
由于只需要处理修改过的数据，刷新实现反而更复杂

数据抽取非常重要，因为它是数据进入仓库的入口。它必须屏蔽底层数据的结构复杂性和物理位置复杂性，同时支持自动刷新，并维护元数据和数据

## 转换和集成到复杂性
数据从操作型环境进入数据仓库，不是简单复制，而要完成许多处理

**技术环境变化**：
不仅是DBMS变化，还可能设计操作系统、硬件甚至底层数据结构变化；
还要尽量避免在在线业务高峰窗口抽取数据；
操作型输入关键字可能需要重建或转换，有时还要加入时间；
非关键字数据也要重新格式化，比如日期格式转换

**数据清理和合并**：
数据清理包括取值范围检查、交叉记录检验、格式检验；
多个数据源进入仓库时需要合并；
记录合并前可能要做关键字解析；
输入记录顺序不同可能要重新排序；
一个传递过程可能产生不同综合层次的多个输出；
如果某些输出值没有对应数据源，还要提供默认值

**抽取汇率和总结问题**：
如果无法区分抽取和不需要抽取的数据，就可能读区整个记录，影响在线系统性能；
多个操作型记录可能需要合成为一个仓库记录；
数据元素重命名要被跟踪；
非标准格式必须转换，并明确转换逻辑；

ETL还必须理解操作性应用程序逻辑中的数据关系，处理编码转换，如 EBCDIC 与 ASCII；
数据仓库设计要符合企业数据模型；
从当前型数据进入历史型数据时可能要加入时间元素；
还要考虑新创建记录如何输出到数据仓库

> ETL 的难点不在于“搬数据”，而在于把不同来源、不同口径、不同结构、不同质量的数据变成统一、可信、可分析的数据

### ETL工具
ETL工具分为两类：
- 产生源代码的软件
- 产生参数化运行时模块的软件

产生源代码的软件更强大，可以以原有数据格式访问数据；运行时模块则需要先把原有数据格式统一，之后才能访问。

两类工具都可以自动完成转换、重新格式化、从多个传统操作型数据源中集成数据的过程。

## 数据目标：原子层、数据集市、ODS、缓冲区

其中，原子层、集成数据和数据集市时数据仓库的主要数据存储

**ODS**
ODS，即操作数据存储，是在企业范围内、针对特定主题区域、用于支持战术决策的综合数据更新集合
其特点为：
```
面向特定分析应用
完整性
当前有效
可变
详尽
```
它介于操作型数据库和数据仓库之间，更偏当前、可更新、战术决策

**缓冲区**
缓冲区是数据流的中间站
白板式，没有特定结构，系统中可以有多处缓冲区
```
原始数据先临时落地
        ↓
清洗、转换、校验、去重
        ↓
再进入正式数据仓库
```

### 数据刷新方法
数据仓库系统必须感知 OLTP 数据库中的变化，并及时有效地反映到数据仓库中
常见刷新方式包括：
```
时间戳
DELTA 文件
建立映象文件
日志文件
```
一个数据仓库系统可以同时采用多种刷新方式，以适应不同数据源

**时间戳**
如果数据库记录有时间属性，就可以根据更新标志和修改时间判断哪些数据需要刷新
缺点：很多数据库记录没有时间属性

**DELTA 文件**
有些应用会生成DELTA文件记录修改操作，可以用它刷新数据仓库
优点：不用全库扫描，效率高
缺点：这类应用不普遍，修改既有应用成本大

**映象文件**
在上次刷新后对数据库做快照，本次刷新前再做快照，通过比较两个快照找出变化
优点：对数据库和应用没有特别要求
缺点：占用大量资源，可能影响原数据库性能

**日志文件**
现代 OLTP 数据库通常有日志文件，可以根据日志实现刷新
优点：日志是数据库固有机制，不影响OLTP性能，效率比DELTA和映象文件更高
缺点：不能用与没有日志机制的遗留系统

### 数据周期
定义：从操作型环境中的数据发生变化，到这个变化反映到数据仓库中所用的时间，通常数据周期不低于24小时

原因：
- 操作型环境和数据仓库结合得越紧密，技术越复杂、成本越高
- 间隔24小时可以让操作型处理和数据仓库处理相互隔离，避免把仓库变成实时操作系统，也让数据在进入仓库前达到稳定状态
