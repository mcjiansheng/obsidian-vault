[[编译原理]]
**编译器为了让源语言中的抽象概念在真实机器上运行，而在目标程序中安排的一整套存储、调用、访问和回收机制**

### 为什么编译器需要运行时环境
编译器不能只把表达式翻译成指令，还必须保证程序运行时的各种语义都成立
例如
```C
int x = 1;

int f(int a) {
    int y = a + x;
    return y;
}
```
这里有很多问题不是靠一条加法指令能解决的

`x` 放在哪里？  
`a` 是怎么传进来的？  
`y` 的生命周期到哪里结束？  
`return y` 的值怎么传回调用者？  
如果函数递归调用，多个 `y` 如何互不干扰？  
如果对象用 `new` 创建，它什么时候释放？  
如果语言支持嵌套函数，内层函数怎么访问外层函数变量？

这些问题统称为 **运行时环境问题**。
编译器生成目标代码时，必须先约定好这些机制：
- 哪些数据放静态区，哪些放栈，哪些放堆；
- 函数调用时栈帧如何布局；
- 非局部变量如何查找；
- 堆对象如何分配和回收。

### 存储相关基本概念

#### 逻辑地址与物理地址
程序运行时看到的地址通常不是直接的物理内存地址，而是**逻辑地址或虚拟地址**
逻辑地址空间由编译器、操作系统和目标机器共同组织；操作系统负责把逻辑地址映射到物理地址，物理地址才是真实内存的编号

所以编译器关心的是：
```
变量名 -> 运行时地址计算方式
```

操作系统关心的是：
```
虚拟地址 -> 逻辑地址
```

例如，编译器可以把局部变量i表示为：
`[BP + 8]`
意思是，相对于当前栈帧基址便宜8字节的位置，但`[BP + 8]`对应哪块真实物理内存，是操作系统和硬件内存管理单元负责的

#### 字节、字与连续存储
假设运行时存储由连续字节块组成，字节是最小编址单位；
1字节包含8位， 4字节构成一个机器字；
多字节对象总是存储在连续字节中，并把第一个字节作为该对象地址

#### 对齐与补白
数据对象的存储布局受目标机器寻址约束影响，很多机器要求某些类型必须放在特定边界上，比如4字节整数最好放在4的倍数地址处，8字节double最好放在8的倍数地址处

为了满足对齐要求，编译器可能在结构体字段之间插入不用的空白，这些空间成为 **补白**

补白本身没有语义价值，但能让CPU更高效访问数据

### 运行时存储的典型布局

![[Pasted image 20260617155523.png]]
代码区、静态区、堆区、栈区分别服务于不同生命周期的数据

#### 代码区
代码区存放目标程序的机器指令或可执行目标代码。代码区的大小在编译时已经确定，英雌它可以被放在一个静态确定的区域中


#### 静态区
静态区存放生命周期贯穿整个程序运行期间的数据
例如
```C
int g = 10; // 全局变量
static int count = 0; // 静态局部变量
const char* s = "abc"; // 常量字符串相关数据
```

静态区中的数据对象大小通常可以在编译时刻知道，包括全局常量和编译器生成的数据，例如用于支持垃圾回收的信息

#### 栈区
栈区用来存放 **活动数据**，也就是函数调用时产生的栈帧。
局部变量、临时值、返回地址、保存的寄存器、参数等通常和栈帧有关

栈的核心特点：**后进先出 - 后调用的函数先返回**

#### 堆区
堆区用于存放生命周期不容易由函数调用关系决定的数据，例如
```c
int* p = malloc(sizeof(int));
```
或
```java
Object obj = new Object();
```
堆对象可能比创建它的函数活得久，所以不能简单放在栈里。
堆存储用于生命周期超过创建过程调用的数据对象，C/C++ 常用程序员手工回收，Java、C#、Python 等语言通常使用垃圾回收机制

### 静态分配与动态分配

#### 静态分配
**静态分配**指编译器在编译时就能确定数据对象的存储位置，不需要依赖程序运行时的具体路径

例如全局变量
```c
int global = 100;
```
它不随函数调用产生或消失，所以可以放在静态区
静态分配的优点是简单、高效；
缺点是无法处理运行时才知道大小或生命周期的数据；

#### 动态分配
动态分配发生在运行时

**栈式存储**
栈式存储和函数调用、返回同步
```
调用函数 → 分配活动记录
函数返回 → 回收活动记录
```
局部变量通常放在栈上

**堆存储**
堆存储不和函数调用返回严格同步
```c
int* make(){
	int* p = malloc(sizeof(int));
	return p;
}
```
这里P指向对象在make后仍然存在，所以它必须放在堆上，而不是放在make的栈帧里

### 栈式分配
#### 活动、过程活动与活动树
一次函数调用称为一次**活动**
```c
int f(int x) {
	return x+1;
}
int main(){
	f(1);
	f(2);
}
```
这里`f(1)`和`f(2)`是同一个函数`f`的两次不同活动

因为过程调用在时间上通常是嵌套的
```
main 调用 f
f 调用 g
g 返回
f 返回
main 返回
```
所以当前未结束的调用总是形成一条链
程序运行期间所有过程活动可以用一颗 **活动树** 表示：每个节点对应一次过程活动，根节点是 `main`活动，某活动的子节点是它调用的各个过程活动，并按调用顺序从左到右排列

活动树和程序行为之间有几个重要对应关系
- **过程调用序列**对应活动树的前序遍历
- **过程返回序列**对应活动树的后序遍历
- 如果当前控制流位于某个活动节点`N`,那么当前尚未结束的活动就是`N`及其所有祖先节点

这解释了为什么可以用栈管理函数调用：当前活跃活动正好是活动树中从根到当前节点的一条路径

#### 活动记录 也叫栈帧
每次函数调用都需要一块运行时存储区域，用来保存这次调用所需的信息，这块区域被称为 **活动记录**，也常叫 **栈帧**
过程调用和返回由控制栈管理，每个活跃活动对应栈中一个活动记录，活动记录按活动开始时间从栈低到栈顶排列，栈中全部活动记录对应活动树从根到当前活动节点的路径
![[Pasted image 20260617161411.png]]

**临时值**
表达式求值过程中产生的中间结果，如果不能全部放在寄存器中，就要放到活动记录中

**局部数据**
函数内部声明的局部变量

**机器状态**
调用函数前，调用者的某些机器状态必须保存，否则被调用者执行完后无法恢复现场
机器状态通常包括：
```
返回地址
旧的栈帧基址
部分寄存器内容
```

**控制链**
控制链指向 **调用者的活动记录**，它体现了“谁调用了我”

控制链主要用于函数返回时恢复调用者现场

**访问链**
访问链用于支持 **嵌套过程中的非局部变量访问**
它体现的不是“谁调用了我”，而是“我的静态外层作用域是谁”

当被调用过程需要其他地方的数据时，需要使用访问链定位；如果过程 `p` 在声明时嵌套在过程 `q` 的声明中，那么 `p` 的活动记录中的访问链指向最上层的 `q` 的活动记录。

**返回值**
如果函数有返回值，活动记录中可能需要一个位置保存返回值，
当然，实际编译器中小型返回值也通常通过寄存器返回

**实在参数**
调用函数时传入的实际参数
实在参数通常尽可能放在寄存器中，而不是活动记录中
但如果参数较多或调用约定要求，参数也可能入栈

### 调用代码序列与返回代码序列
函数调用并不是一条简单的跳转指令，它必须完成参数传递、保存现场、跳转、建立新栈帧、返回值传递、恢复现场等工作

两类代码序列：
**调用代码序列**：为被调用者活动记录分配空间，并填写必要字段
**返回代码序列**：恢复机器状态，使调用者能继续运行

#### 调用/返回代码序列要满足的要求
数据方面
```
正确传递参数
正确返回传递值
```
控制方面
```
正确跳到被调用过程入口
正确回到调用位置的下一条指令
```
同时，调用代码序列和活动记录布局密切相关

#### 活动记录布局规则

**调用者和被调用者之间传递的值放在被调用者活动记录开始的位置**
这一调用者在还没有完整创建被调用者活动记录时，就可以先把参数和返回值空间准备好

**固定长度字段放在中间**
例如
```
控制链
访问链
机器状态
返回地址
保存寄存器
```
这些大小固定，方便调试器或异常处理机制解码栈内容

**编译时大小位置的项放在活动记录尾部**
例如变长数组
```c
void f(int n){
	int a[n];
}
```
数组 `a` 的大小运行时才知道，所以不能像普通局部变量那样固定偏移访问

栈顶指针`top_sp`通常指向固定长度字段的末端，使固定长度数据可以通过相对于`top_sp`的固定偏移访问；而变长域位于栈顶之上，其偏移量需要运行时计算

#### 一个典型调用序列

调用函数 `fun(i, j)` 时，大致过程如下：
```
1. 调用者计算实参 i、j 的值
2. 调用者把参数放到约定位置
3. 调用者保存返回地址和旧的栈指针/基址
4. 跳转到 fun 的代码入口
5. 被调用者保存必要寄存器
6. 被调用者初始化局部变量
7. 执行函数体
```
返回时
```
1. 被调用者把返回值放到约定位置
2. 恢复寄存器和栈指针
3. 跳转到返回地址
4. 调用者取得返回值并继续执行
```

课件给出的返回代码序列正是：被调用者将返回值放到与参数相邻的位置，恢复 `top_sp` 和寄存器，然后跳转到返回地址

### 栈中的变长数据
有些数据对象大小在编译时不知道，但生命周期又局限在当前函数调用内，这是可以放在栈上

两个指针区分：
```
top     : 指向实际栈顶，即下一个活动记录开始位置
top_sp  : 用于寻找顶层活动记录的定长字段
```

变长数据虽然在栈中，但不一定被视为活动记录固定部分；
固定字段用相对 `top_sp` 的偏移访问，变长字段则通过运行时计算得到地址

### 一个简单的例子
```c
int fun(int a, int b);

int m = 10;

int main() {
    int i = 4;
    int j = 5;
    m = fun(i, j);
    return 0;
}

int fun(int a, int b) {
    int c = 0;
    c = a + b;
    return c;
}
```
```
代码区：main 和 fun 的机器指令
静态数据区：全局变量 m
动态数据区：main 和 fun 调用时产生的栈帧
```
程序开始前，`m`在静态数据区，`main`和`fun`的代码在代码区，动态数据区还没有对应的函数栈帧，
程序运行的本质就是代码区指令不断执行，并驱动静态数据区和动态数据区发生变化

#### eip、ebp、esp
eip：指向代码区下一条要执行的指令
ebp：指向当前栈帧的栈底
esp：指向当前栈顶
函数调用和返回的过程，本质上就是不断调整`eip`、`ebp`、`esp`
```
eip 决定执行哪条指令
ebp 用于稳定访问当前函数的参数和局部变量
esp 随着压栈、退栈不断变化
```

#### main的执行过程
进入`main`时，保存旧`ebp`,然后令`ebp`指向`main`的栈底
接着执行
```c
int i=4;
int j=5;
```
于是4和5作为局部变量值被压入main的栈帧

#### 调用fun(i,j)
调用`fun(i,j)`时，需要先传参
在C语言中，参数入栈的顺序和书写顺序相反，所以`b`先入栈，`a`后入栈。
接着返回保存地址,`eip`跳转到`fun`的代码区

然后建立`fun`的栈帧
```
保存 main 的 ebp
设置 fun 的 ebp
为局部变量 c 分配空间
执行 c = a + b
把 c 作为返回值
```
访问`a`,`b`,`c`,时，底层其实是通过`ebp`加减偏移量完成的
```'
a 在 [ebp + 某偏移]
b 在 [ebp + 某偏移]
c 在 [ebp - 某偏移]
```

#### fun 返回 main
`fun`执行完后，需要恢复`main`调用`fun`前的现场，包括
```
恢复 main 的栈底和栈顶
找到返回地址
让 eip 跳回 main 中 fun 调用后的下一条指令
把返回值赋给全局变量 m
```
最后`main`返回，清除`main`的栈帧，程序执行结束

**代码属于代码区，全局/静态数据属于静态区，局部变量、参数、返回地址、旧 BP 等属于栈帧；SP 指向栈顶，BP 用于定位当前栈帧。**

### 非局部数据访问

#### 没有
