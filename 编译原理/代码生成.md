[[编译原理]]
**代码生成到底在做什么？**
源程序经过词法分析、语法分析、语义分析、中间代码生成和机器无关优化之后，到了代码生成阶段。代码生成的任务就是：
> 把中间表示IR翻译成目标机器上的代码

中间代码通常是比较抽象的，比如三地址码：
```
t1 = y + z
x = t1
```
但真实机器不能直接执行这种形式，目标机器通常需要显式说明：
```
LR  R1,y
ADD R1,R1,z
ST x,R1
```
即：先从内存加载到寄存器，再从寄存器中计算，最后必要时写回内存

代码生成不是简单的“逐句翻译”，它同时涉及三个互相影响的问题：
1. **指令选择**：用那些机器指令实现某条IR语句
2. **寄存器分配和指派**：哪些值放进寄存器？具体放进哪个寄存器？
3. **指令排序**：这些指令按照什么顺序执行更好？

## 代码生成器设计中的主要问题
### 正确性是第一位

**生成的目标代码必须和源程序语义等价**

### 易于实现、测试和维护
代码生成器设计时，除了追求最优代码，还要考虑工程可维护性

### 输入IR的选择
不同IR会影响代码生成难度
高层 IR 保留语义多但离机器远；低层 IR 接近机器但优化空间较小；中层 IR 如三地址码、SSA 形态的 TAC 是常见折中。

## 指令选择：把IR变成机器指令

### 最朴素的翻译
对于三地址语句
```
x = y + z
```
可以直接翻译成：
```
LD   R0, y
ADD  R0, R0, z
ST   x, R0
```
含义为
```
R0 = y
R0 = R0 + z
x = R0
```
这是一种简单、规则、容易实现的翻译方式
但它往往不是最优的

### 为什么不是最优
假设有
```
a = b + c
d = a + e
```
朴素生成
```
LD   R0, b
ADD  R0, R0, c
ST   a, R0
LD   R0, a
ADD  R0, R0, e
ST   d, R0
```
但是可以优化为
```
LD   R0, b
ADD  R0, R0, c
ADD  R0, R0, e
ST   d, R0
```
如果a后面不需要保存在内存中，就可以不写回

这说明代码生成器需要知道：
- 当前寄存器里有哪些值
- 某个变量后面是否还会使用
- 某个变量在内存中的值是否已经是最新的
- 某个寄存器是否可以覆盖
这引出了后续的 **寄存器描述符** 、 **地址描述符** 、 **活跃信息** 、 **待用信息**

### 指令选择受哪些因素影响？
1. **IR 层次**
    - IR 越高层，语义信息多，但离机器远；
    - IR 越低层，容易映射到机器，但优化空间少。
2. **目标机器指令集**
    - RISC 通常指令简单，load/store 分明；
    - CISC 可能有复杂寻址模式和复合指令；
    - 有的机器支持内存操作数，有的要求操作数必须在寄存器中。
3. **目标代码质量要求**
    - 编译快；
    - 代码短；
    - 运行快；
    - 能耗低；
    - 寄存器使用少；
    - 缓存行为好。

## 目标语言 ： 三地址机器模型
**加载指令** `LD dst,addr` 把地址 `addr` 中的内容加载到寄存器 `dst`。
**保存指令** `ST x,r` 把寄存器r中的内容保存到变量x的地址中
**计算指令** `OP dst,src1,src2` 
**无条件跳转** `BR L` 表示跳到标号L
**条件跳转** `Bcond r,L` 表示测试寄存器r，条件满足则跳到L

## 寻址模式：目标代码如何访问数据
寻址模式决定“操作数在哪里，以及如何找到它”

### 1.变量名x
`x` 表示变量x被分配到的内存位置

### 2.a(r)
表示地址
`a + contents(r)`
常用于数组访问

例如数组 `a[i]`，如果每个元素宽度为 8 字节，可以先算：
```
LD   R2, i
MUL  R2, R2, #8
LD   R1, a(R2)
```
其中 `a(R2)` 表示地址 `a + R2`。

### 3.constant(r)
表示地址：
`contents(r) + constant`
常用于活动记录中的局部变量访问

例如
```
LD R1, 8(SP)
```
表示从 SP + 8 处加载

### 4.\*r
表示寄存器r中的内容是地址，要访问这个地址的内容
类似指针解引用

### 5.\*constant(r)
更复杂的间接寻址

### 6. 常量 \#constant
表示立即数

## 目标代码示例

### 1. 普通赋值：`x = y - z`

朴素代码：

```
LD   R1, y
LD   R2, z
SUB  R1, R1, R2
ST   x, R1
```

含义：

```
R1 = y
R2 = z
R1 = R1 - R2
x = R1
```

但如果 `y` 已经在某个寄存器中，就不必重新 `LD`；如果 `z` 已经在寄存器中，也不必重新加载；如果 `x` 后面不再使用，甚至可以不写回内存。

### 2. 数组读取：`b = a[i]`

假设每个数组元素宽度是 8 字节：
```
LD   R1, i
MUL  R1, R1, #8
LD   R2, a(R1)
ST   b, R2
```

解释：

```
R1 = i
R1 = i * 8
R2 = contents(a + R1)
b = R2
```

### 3. 数组写入：`a[j] = c`

```
LD   R1, c
LD   R2, j
MUL  R2, R2, #8
ST   a(R2), R1
```

含义：

```
R1 = c
R2 = j * 8
contents(a + R2) = R1
```
这里非常重要：数组访问并不只是“取变量”，它包含**地址计算**。代码生成器必须知道元素宽度、数组基地址、下标值、寻址模式。

## 指令代价：为什么“最优代码“很难
程序和指令的代价可以用不同指标衡量：
- 编译时间
- 目标程序大小
- 运行时间
- 能耗
每条指令有固定代价，通常设为
`1 + 操作数寻址模式代价`

例如
`LD R0, R1` 代价为1，因为寄存器到寄存器很便宜
`LD R0, M` 代价为2，因为访问内存更贵

但真实机器更复杂。访问内存可能因为缓存命中/未命中而差距很大；不同指令有不同延迟；浮点、整数、向量指令代价不同；分支预测失败代价很高。

## 目标代码中的地址：IR名字如何变成真实地址？
`x = y + z` 中有变量名

但目标代码必须知道x、y、z在哪里，可能在：
- 静态区
- 栈帧
- 堆
- 寄存器
- 常量池
- 对象内部偏移
- 数组元素地址
代码生成器需要把IR中的名字转换成目标代码中的地址

### 静态分配
静态分配是指编译时就确定某个过程或变量的位置
```
staticArea + offset(x)
```
可以直接作为变量x的地址

**优点** 简单、访问快
**缺点** 限制大

如果每个过程只有一个固定活动记录，那么递归就无法自然支持
因为递归调用同一个过程时，不同调用需要不同的局部变量副本，静态分配至右一份空间，会被覆盖

静态分配适用于：
- 全局变量
- 静态变量
- 编译时大小固定且生命周期贯穿整个程序的数据
- 不支持递归或不需要多活动实例的过程语言模型

### 栈式分配
现代语言中，过程调用通常采用栈式分配，每次调用过程时创建一个新的活动记录，也叫栈帧

由于活动记录的位置在运行时才知道，所以目标代码不能写死绝对地址，而是使用类似
`offset(SP)` 或 `offset(FP)`

其中SP是栈顶指针，FP 是帧指针

过程调用代码：
```
ADD  SP, SP, #caller.recordSize
ST   0(SP), #here + 16
BR   callee.codeArea
```
过程返回：
```
BR   *0(SP)
SUB  SP, SP, #caller.recordSize
```
核心思想：
- 调用者调整栈指针，给被调用者活动记录留空间
- 保存返回地址
- 跳转到被调用过程
- 被调用过程执行完后跳回返回地址
- 调用者恢复栈指针

### 基本快和流图
代码生成通常不会对整个程序一次性处理，而是先划分为 **基本快** ，再构造 **流图**

### 什么是基本快？
基本快是满足以下性质的一段连续三地址语句：
1. 只能从第一条语句进入
2. 除最后一条语句外，中间没有跳转出去
3. 一旦进入，块内语句会顺序执行到最后

基本块是一个“没有分叉、没有中途入口”的直线代码片段。

### 如何划分基本快？
常用算法是先找入口语句 **leader**

一条三地址语句是入口语句，如果：
1. 它是程序第一条语句；
2. 它是某个跳转语句的目标；
3. 它紧跟在某个跳转语句之后；

### 什么是流图？
基本块之间根据控制流关系连接，形成控制流图 CFG
如果基本快 B1 执行完可能进入 B2，则有边 `B1 -> B2`
常见情况：
- B1最后一条是 `goto L`，则连到 L 所在基本块
- B1最后一条是条件跳转 `if ... goto L`,则连到跳转目标块，也连到下一顺序块；
- B1 没有跳转，顺序流入下一个块
- return 、 halt 后没有后继块
流图是后续数据流分析、活跃变量分析、寄存器分配和优化的基础

## 基本块优化方法

### DAG表示基本块
基本块中的表达式可以构造成 DAG，有向无环图
比如
```
a = b + c
b = a - d
c = b + c
```
![[Pasted image 20260618105429.png]]
发现b+c重复出现，可以使用连线

DAG的作用：
1. 发现公共子表达式
2. 删除无用赋值
3. 重排计算顺序
4. 减少临时变量
5. 生成更好的代码
### 局部公共子表达式消除
如果一个表达式之前已经计算过，而且参与运算的变量没有被修改，那么可以复用之前的结果

例如：
```
t1 = b + c
a = t1
t2 = b + c
d = t2

则可以变为
t1 = b + c
a = t1
d = t1
```
在基本快内，这种判断相对容易，因为控制流是顺序的

### 死代码删除
如果一个变量的值后面不会再被使用，那么对它的赋值就是死代码

```
x = y + z
x = a + b
```
第一条对 x 的赋值没有任何作用，因为马上被第二条覆盖，且中间没有使用 x
可以删除，变为
```
x = a + b
```
要注意：如果右侧表达式有副作用，则不能随便删除，比如函数调用，可能抛异常的操作等

### 代数化简
```
x = y + 0      => x = y
x = y * 1      => x = y
x = y * 0      => x = 0
x = y - y      => x = 0
```
但浮点场景要谨慎，例如NaN、±0、溢出、舍入模式都可能影响语义

### 常量折叠
`x = 2 * 3 => x = 6`

### 复制传播
```
x = y
z = x + 1
如果 x 和 y 之间没有修改，则可以变成
z = y + 1
```
后续如果 `x = y` 没用了，还可以删除

## 活跃信息和待用信息
为了生成高质量代码，代码生成器需要知道变量后面是否还会用

### 活跃变量
如果变量x在某个程序点之后还会使用，并且在使用之前没用被重新赋值，那么x在该点上活跃的
否则就是死的

例如
```
x = a + b
y = c + d
z = y + 1
```
如果后面没有使用x，那么x = a + b这条赋值就是死代码

### 待用信息 next-use
待用信息表示某个变量下一次会在哪条语句被使用

```
1: a = b + c
2: d = a + e
3: f = b + d
```
在第 1 条之后：
- `a` 的下一次使用在第 2 条；
- `b` 的下一次使用在第 3 条；
- `c` 如果后面不用，则无待用。

这些信息对寄存器选择很关键。
如果寄存器不够，需要选择一个寄存器腾出来。优先选择：
1. 其中的变量已经死了；
2. 或者下一次使用最远；
3. 或者内存中已经有最新副本；
4. 或者写回代价最低。

## 寄存器描述符和地址描述符
代码生成器通常维护两类表

**寄存器描述符 Register Descriptor**
记录每个寄存器当前保存了哪些变量或临时值
```
R0: a, t1
R1: b
R2: empty
```
表示 R0 中的值同时是变量 a和临时变量 t1 的当前值

**地址描述符 Address Descriptor**
记录某个变量当前值在哪里
```
a: R0, memory
b: R1
c: memory
t1: R0
```
表示：
- a的当前值在R0 和内存中都有
- b的当前值只在R1，内存可能不是最新
- c当前只在内存中
- t1 只在 R0中
两个描述符配合使用，可以避免重复加载和不必要写回

## 基本代码生成算法
假设要处理三地址语句：
`x = y op z`
朴素做法：
```
LD R1, y
LD R2, z
OP R1, R1, R2
ST x, R1
```
更好的代码生成器会做：
1. 调用 getReg(x = y op z) 选择一个寄存器 Rx
2. 确保 y 在某个寄存器中
3. 确保 z 可作为操作数
4. 生成计算指令
5. 更新寄存器描述符和地址描述符
6. 必要时在基本块出口写回活跃变量

### getReg的基本思想
getReg 要为当前语句找一个合适的目标寄存器

若 y已在某个寄存器中，且之后不再使用
`x = y op z`
假设y在R1中，则可以直接复用R1：
`OP R1,R1,z`
然后让 R1 表示 x

若有空寄存器
则直接使用空寄存器

如果没有空寄存器，则选择某个寄存器，把其中活跃且内存没有最新副本的变量写回内存
`ST a, R2`
然后复用 R2

### 对 x = y 的处理
如果是复制语句 `x = y`
不一定要生成机器指令，可以只更新描述符，让x 和 y指向同一位置
`R1 : y`
处理 x = y后
`R1: x,y`

地址描述符：
```
x : R1
y : R1
```
可以省掉一次move

### 对数组和指针的处理
数组读写和指针访问会复杂很多，因为可能涉及到别名
例如：
`x = a[i]`
不能简单认为内存中其他变量不变，因为 `a[i]` 可能和某个指针指向的位置重叠

对于
`a[i] = y`
它会修改内存
代码生成器必须保守地认为某些内存位置可能被改变，尤其在没有精确别名分析时

## 基本块出口的处理
在基本块内部，某些变量的最新值可能只在寄存器里，没有写回内存
如果基本块结束时，后继基本快需要这些变量，而后继块不知道当前寄存器的状态，通常需要写回
```
B1:
	x = y + z
	goto B2
B2:
	a = x + 1
```
如果 x 在 B1 结束时只在 R1 中，而 B2 按照内存地址访问 x，那么 B1 结束前必须：
`ST x, R1`
否则 `B2` 读到的是旧值
不过如果编译器做全局寄存器分配，能保证 x 跨基本快仍在同一个寄存器中，就可以省掉这类写回，这就是局部寄存器分配和全局寄存器分配的区别

## 寄存器分配和寄存器指派

### 寄存器分配 Register Allocation
决定哪些变量或临时值放入寄存器
```
a, b, t1 放寄存器
c, d 放内存
```

### 寄存器指派 Register Assignment
决定具体放在哪个物理寄存器
```
a  -> R1
b  -> R2
t1 -> R3
```

两者相关但不完全相同，先决定哪些值值得放寄存器，再决定怎么映射到有限物理寄存器

## 局部寄存器分配
局部寄存器分配只在一个基本块内部考虑

**优点**
- 实现简单
- 不需要复杂全局分析
- 适合教学编译器或低优化等级

**缺点**：
- 基本快之间不能复用寄存器信息
- 每个基本快出口可能要频繁写回
- 循环中的变量可能反复 load/store
- 对跨块活跃变量处理不够好

比如循环中，每次迭代都需要从内存加载 i ，再写回，性能很差，更好的方式是让 i 长期保存在寄存器中

## 全局寄存器分配：图染色

### 核心思想
把寄存器分配问题转化为图染色问题
- 图中的节点：变量、临时值、虚拟寄存器
- 图中的边：两个值的生命期冲突，不能放在同一个物理寄存器
- 颜色：物理寄存器
如果两个变量在同一时刻都活跃，它们就不能共用一个寄存器，因此在冲突图中连一条边

### 冲突图
例如
```
a = ...
b = ...
c = a + b
```
在计算 c = a + b 时，a和b同时活跃，所以a和b冲突，不能分配同一个寄存器
如果有k个物理寄存器，就要尝试用k种颜色给冲突图染色，使得相邻节点颜色不同
如果无法染色，就需要溢出

### k-Coloring

图染色问题本身是 NP-Complete，因此真实编译器使用启发式算法

### Chaitin算法
Chaitin算法是经典的图染色寄存器分配算法

**核心步骤**
1. 构建冲突图
2. 找到度数小于k的节点，删除并压栈
3. 重复直到图为空
4. 从栈中弹出节点，给它选择一个没有被邻居使用的颜色
5. 如果某个阶段找不到合适节点，选择一个结点作为溢出候选
6. 插入 load/store 后重新构图和染色

> 如果某个节点的度数小于k，那么当它最后被放回图中时，最多有k-1个邻居已经染色，因此总能找到一种可用颜色

### 溢出 Spill
如果寄存器不够，就必须把某些变量放到内存中

溢出会增加 load/store 因此要尽量选择“溢出代价低“的变量，例如
- 使用次数少
- 不在循环内部
- 下一次使用很远
- 生命期很长导致冲突多但收益低
