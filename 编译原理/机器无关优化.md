[[编译原理]]
**机器无关优化** machine-independent optimization ，是编译器中端在中间表示 IR 上进行的优化，它不直接依赖x86 ARM Risc-v 等目标机器指令、而是基于程序语义、控制流、数据流进行等价变换

例如：
```c
x = a + b;
y = a + b;
```
只要能证明第二次a + b 之前 a 和 b 没变，就可以改为
```c
x = a + b;
y = x;
```
它只关心：程序语义是否保持不变

从编译器结构看，IR是前端和后端之间的桥梁。
IR 与具体源语言和目标机器都相对独立，用于承载程序语义、驱动优化，并作为目标代码生成输入；大多数中断优化，如常量传播、死代码删除、循环优化，都发生在IR层

## 代码优化的基本目标
1. 在目标代码中消除不必要的指令
2. 把一个指令序列替换成完成相同功能但更快的指令序列
3. 全局优化通常基于数据流分析技术，用算法收集程序相关信息

减少运行时间 ： 少算、少访问、少跳转、少循环内重复计算
减少代码体积 ： 删除无用代码、合并公共表达式
减少内存访问 ： 更多复用已有值
减少能耗 ： 通常与减少指令数量、减少访存有关
为后端创造条件：例如减少临时变量，降低寄存器压力

**优化的前提永远是语义等价**

## 优化的来源：为什么程序里会有冗余？
编译器只能通过一些相对低层的语义等价转换来优化代码，冗余运算主要来自两个方面：源程序本身的冗余，以及高级语言编程帮时带来的副产品。典型优化包括公共子表达式消除、复制传播、死代码消除、常量折叠等

**源程序中的冗余**
```c
x = a * b + c;
y = a * b + d;
```
这里 `a * b`重复出现，如果中间没有改变a或b，那么第二次可以复用第一次结果
```c
t = a * b;
x = t + c;
y = t + d;
```

**高级语言特性带来的冗余**
高级语言让代码更清晰，但编译成低层IR后，可能产生很多重复地址就算
```c
a[i][j].f=0;
a[i][j].k=1;
```
可以把重复的地址计算部分提取出来

**中间代码生成产生的临时变量**
三地址码为了简单、线性，会引入很多临时变量。
例如：因为三地址码每条指令右侧最多一个运算符，常用临时变量分解复杂表达式；这种形式便于生成和优化，但也可能产生冗余临时变量
```
t1 = a + b
t2 = t1
x = t1
```
## 核心优化一：全局公共子表达式消除
### 什么是公共子表达式
如果表达式E在某次出现之前必然已经被计算过，并且E的分量在该次计算之后一致没有被改变，那么E的本次出现就是公共子表达式。如果上一次E的值赋给了x，且 x 的值至今没有被修改过，那么可以直接使用 x，而不必重新计算 E

>如果 E 已经算过，并且 E 依赖的变量没变，那么 E 不用再算。

### 局部公共子表达式 vs 全局公共子表达式
**局部公共子表达式消除**只在一个基本快内做
因为基本快中没有分支，语句按顺序必然执行，所以比较容易判断

**全局公共子表达式消除**跨基本快，需要分析控制流
```
B1:
t1 = a + b
if cond goto B2 else B3

B2:
x = a + b
```
要判断 B2 中的 a + b是否公共，必须知道：从入口到 B2 点所有路径上，是否都已经计算过 a + b，并且 a、b都没有被修改过，这就需要数据流分析，尤其是 **可用表达式分析**

## 核心优化二：复制传播

### 什么是复制语句？
复制语句形如
`u = v`
意思是从这条语句之后，只要 u 和 v 都没被修改过，u 的值就等于 v
如果在某个位置上u 一定等于 v，就可以把 u替换成 v；有时可以彻底消除对u的使用，从而消除u=v这个赋值语句

### 复制传播通常和其他优化联动
公共子表达式消除经常会引入复制语句

实际编译器中优化通常不是只跑一次，而是多个pass反复执行，直到没有明显变化

## 核心优化三：死代码消除
### 什么是死代码：
如果一个变量在某个程序点上的值可能会在之后被使用，那么这个变量在该点是活跃的；否则就是死的。若对一个变量的赋值结果之后不会被使用，那么这个赋值就是没有用的死代码

### 死代码常常由前面的优化产生

### 不是所有“看起来无用”的语句都能删除

例如：
```c
x = f();
```
即使 x 后面没有使用，f()可能有副作用，因此不能删除

死代码消除依赖两个判断：
1. 赋值左部变量是否活跃
2. 右部表达式是否有副作用


## 核心优化四：常量折叠与常量传播
### 常量折叠：编译器直接算出结果
```c
x = 2 * 3 + 4;
```
优化为
```c
x = 10;
```
### 常量传播：把变量已知常量值传播到使用处
```c
a = 10
b = a + 3
```
如果能证明a在b = a + 3前没有改变，可以传播
```c
a = 10
b = 10 + 3
```
### 常量传播要考虑控制流
```c
if (flag)
    a = 10;
else
    a = 20;
b = a + 1;
```
flag不确定，a就不一定是同一个常量，不能简单替换

## 核心优化五：代码移动
### 什么是代码移动？
代码移动，尤其是 **循环不变代码外提** ，是循环优化中的重点

循环不变表达式：在循环的同义词运行的不同迭代中，表达式的值不变。把循环不变表达式移动到循环入口之前计算可以调高效率
```c
while (i <= limit - 2) {
    a[i] = a[i] + 1;
    i++;
}
```
循环体不计算`limit -2` 所以可以优化：
```c
t = limit - 2;
while (i <= t) {
    a[i] = a[i] + 1;
    i++;
}
```

### 代码移动为什么有效
循环体执行次数可能很多，循环外代码执行一次，循环内代码执行n次

### 代码移动到安全条件
表达式E能外提，一般需要满足：
1. E在循环中每次计算结果相同
2. E的操作数在循环中不被改变
3. 外提不会改变异常、副作用、内存访问语义
4. 外提位置必须支配原计算点，通常放在循环前置块preheader中
5. 如果原表达式不一定执行，外提后却一定执行，要确认不会引入异常或副作用
## 核心优化六：归纳变量和强度削减
### 什么是归纳变量
如果每次对x的赋值都使x的值增加某个常量c，那么x就是归纳变量，可以把对x的赋值改成增量操作，以削减计算强度；如果两个归纳变量步调一致，还可以删除其中一个
```c
for (i = 0; i < n; i++) {
    t = 4 * i;
    a[t] = ...
}
```
t 随 i 线性变化，每次 i 加一，t 加 4
可以优化为
```c
t = 0;
for (i = 0; i < n; i++) {
    a[t] = ...
    t = t + 4;
}
```
这就是强度削减

### 什么是强度削减
强度削减 strength reduction 把昂贵运算换成便宜运算

## 为什么需要数据流分析
所有优化都在问类似的问题：
- 某个表达式之前是否一定计算过？
- 某个变量的值之后是否还会被使用？
。。。

这些问题不止看一行代码，而要看控制流路径
机器无关优化基于数据流分析技术，数据流分析用于收集程序相关信息

### 基本快和流图
数据里分析通常在控制流图CFG上进行

**基本块**是满足以下条件的一段连续语句：
1. 只有第一个语句可以作为入口
2. 只有最后一个语句可以跳出
3. 中间没有分支入口和分支出口

**流图 / 控制流图 CFG**
- 节点：基本快
- 边：可能的控制流转移
- 如果执行完B1可能进入B2，则有边 B1 -> B2
数据流分析在 CFG 上传播集合信息

### 数据流分析的一般形式
对每个基本快B，通常定义：
```
IN[B] 进入基本块B前成立的信息
OUT[B] 离开基本块B后成立的信息
gen[B] B内生成的信息
kill[B] B内杀死的信息
```
`OUT[B] = gen[B] ∪ (IN[B] - kill[B])`

### 数据流分析为什么要迭代？
因为程序有循环
```
B1 → B2 → B3
     ↑    ↓
     └────┘
```
只有反复迭代，知道所有 `IN / OUT` 集合不再变化，也就是达到**不动点fixed point**
```
初始化每个基本块的 IN/OUT
while 有集合发生变化:
    按某种顺序遍历基本块
    根据数据流方程更新 IN/OUT
```

## 到达定值分析
### 什么是“定值”
定值 definition 指对变量的一次赋值

```
d1 : x = 1
d2 : y = x + 2
d3 : x + 3
```
d1 和 d3 都是变量 x 的定值

### 什么是“到达定值”
一个定值 d 到达程序点 p，意思是：
从 d 到 p 存在一条路径，并且这条路径上没有其他对同一变量的重新定值杀死 d

### 到达定值分析的作用
到达定值分析可以回答
`这里使用的变量 x ，可能来自哪些赋值`

### 到达定值的数据流方向
到达定值是 **前向分析**
因为定值从赋值点向后传播
`前驱块 OUT -> 当前块 IN -> 当前块 OUT`

### 到达定值是 may analysis
只要存在一条路径使某个定值能到达，就认为它可能到达
```
IN[B] = U OUT[P] P 是 B 的前驱
```

### 到达定值方程
```
gen[B] : 基本块B中产生、并且能到达B末尾的定值
kill[B] : 被B中定值杀死的其他定值
IN[B] : 进入B时可能到达的定值集合
OUT[B] : 离开B时可到达的定值集合

方程
IN[B]  = ∪ OUT[P]，P ∈ pred(B)
OUT[B] = gen[B] ∪ (IN[B] - kill[B])
```

## 活跃变量分析
### 什么是活跃变量？
如果一个变量在某个程序点上的值可能会在之后被使用，那么这个变量在这个点上活跃；否则这个变量就是死的

### 活跃变量分析的作用
活跃变量分析用于：
- 死代码消除
- 寄存器分配
- 构造变量冲突图
- 判断某个临时变量是否还需要保存
尤其是寄存器分配：如果两个变量在同一程序点同时活跃，它们不能放在同一个寄存器里。
全局寄存器分配通常需要基于live返回构建冲突图，再用图染色算法分配物理寄存器

### 活跃变量分析的数据流方向
活跃变量是**反向分析**
因为要知道“以后会不会用”，必须从后往前看
`后继块 IN → 当前块 OUT → 当前块 IN`

### 活跃变量分析是may analysis
只要某条后续路径可能使用变量，这个变量就必须被认为是活跃的。
`OUT[B] = ∪ IN[S]，S ∈ succ(B)`

### 活跃变量方程
```
use[B] 在B中先被使用、后被定义的变量
def[B] 在B中被定义的变量
IN[B] 进入B前活跃的变量
OUT[B] 离开B后活跃的变量

方程：
OUT[B] = ∪ IN[S]，S ∈ succ(B)
IN[B] = use[B] ∪ (OUT[B] - def[B])
```
## 可用表达式分析

### 什么是可用表达式
表达式e在程序点p可用，意为：
从入口到p的所有路径上，e都已经被计算过，并且自最后一次计算e后，e的操作数没有被修改

### 可用表达式分析的作用
- 全局公共子表达式消除
- 检测冗余计算
- 辅助代码移动
### 可用表达式分析的数据流方向
可用表达式是 **前向分析**

因为表达式从计算点向后变得可用

### 可用表达式是must analysis
表达式要在某点“可用”，必须所有路径都计算过它。
`IN[B] = ∩ OUT[P]，P ∈ pred(B)`

### 可用表达式方程
```
e_gen[B] : B中计算并在B末尾仍可用的表达式
e_kill[B] : B中由于操作数被重新定义而杀死的表达式
IN[B] : 进入B前可用的表达式
OUT[B] : 离开B后可用的表达式

方程：
IN[B]  = ∩ OUT[P]，P ∈ pred(B)
OUT[B] = e_gen[B] ∪ (IN[B] - e_kill[B])
```

## 流图中的循环优化
循环是优化的重要目标，因为循环体可能执行很多次，即使每轮只省一条指令，总收益也可能很多

### 如何在流图中识别循环
常用 **支配关系 dominator** 和 **回边 back edge** 来识别自然循环 
**支配关系**
如果从入口节点到结点n 的每条路径都必须经过结点d ，则称d支配 n，记作
`d dom n`
理解为 `想去 n,必须先过 d`
入口节结点支配所有结点

**回边**
如果 CFG 中有一条边
`n -> d`
并且 d 支配 n ，那么这条边就是回边

**自然循环**
由回边 n -> d 形成的自然循环包括：
1. 循环头 d
2. 回边起点 n 
3. 所有能够到达 n 且不经过 d 的结点
从回边尾部往前找，能倒着追到的结点，再加上循环头，这是这个循环

### 循环入口和前置块 preheader
**循环入口**： 进入循环的跳转都以这个入口为目标

**preheader 前置块**：
```
preheader:
    循环外提代码放这里
    goto header

header:
    循环条件判断
    ...
```

循环不变代码外提后，需要有一个放外提出来的依据，并且这个地方应该：
1. 在进入循环前执行
2. 每次进入循环只执行一次
3. 不属于循环体
4. 支配循环头

### 循环不变代码外提 LICM
LICM，全称 Loop-Invariant Code Motion
依赖于两个分析：
循环识别 / 定值分析或SSA信息，判断操作数是否在循环中改变

### 归纳变量优化

```c
for (i = 0; i < n; i++) {
    t = 4 * i;
    a[t] = 0;
}
```
`i` 是基本归纳变量，`t` 是派生归纳变量。
```c
t = 0;
for (i = 0; i < n; i++) {
    a[t] = 0;
    t = t + 4;
}
```
如果 `i` 除了控制循环之外不再使用，有时还可以把循环条件也改写为基于 `t`：
```c
t = 0;
limit = 4 * n;
while (t < limit) {
    a[t] = 0;
    t = t + 4;
}
```
这叫**归纳变量消除**。
不过是否能消除 `i`，要看源语言语义、溢出行为、循环退出后是否还使用 `i`。
