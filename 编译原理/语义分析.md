[[编译原理]]
**为什么要语义分析：CFG只能描述语法，不能描述全部语义**
上下文无关文法 Context - Free Grammar 简称CFG 很适合描述程序的结构
例如
E → E + T
E → T
T → id
它能说明 a+b 是一个表达式，但不能说明
```
a 是否声明?
a 是 int 还是 float?
+ 能不能作用在这两个类型上?
表达式结果放在哪里?
```

上下文无关文法只能描述语法，因此需要结合语义规则来完成语义分析与中间代码生成。
语法制导定义SDD正是为了解决这个问题：把属性和语义规则附着到文法符号和产生式上

`语法树+属性计算=语义分析`

## 语法制导定义 SDD 与语法知道翻译 SDT

### SDD ：Syntax-Directed Definition 语法制导定义

SDD是一种形式化描述方法
`SDD = CFG + 属性 + 语义规则`
它把属性附加到文法符号上，并把语义规则附加到产生式上

例如：
```
E → E1 + T
E.type = if E1.type == T.type then T.type else float
E.place = newTempPlace()
```
这里
```
E.type 表示表达式 E 的类型
E.place 表示表达式 E 的结果存放位置
```
如果有
```
F → id
F.type = lookupIDTable(id.lexValue) -> type
F.place = lookupIDTable(id.lexValue) -> address
```
那么语义分析就通过符号表查出标识符的类型和地址

### 属性：语义分析的核心信息载体

属性 Attribute  是与文法符号相关联的值，用来存储语义信息。常见属性包括
```
type 类型
place 值存放位置
addr 地址
code 生成的中间代码
val 表达式值
width 类型宽度
offset 相对地址偏移量
entry 符号表条目
```
例如
`E.val = E1.val + T.val`
这表示父节点E的值依赖于子节点E1和T的值，属性之间可能存在依赖关系，因此需要确定属性计算顺序

### 综合属性和继承属性
**综合属性**
综合属性从子节点向父节点传递，适合表达
```
表达式的值
表达式的类型
表达式生成的代码
表达式结果位置
```
例子
```
E → E1 + T (E.val = E1.val + T.val)
E → T (E.val = T.val)
T → num (T.val = num.lexval)
```

其中 `val` 就是综合属性，计算方向是自底向上。综合符的综合属性通常由词法分析获得，例如`digit.lexval`

**继承属性 Inherited Attribute**
继承属性从父节点或左兄弟节点向当前节点传递，适合表达
```
声明语句中的类型环境
当前作用域符号表
数组或记录的上下文
已经累计的中间结果
控制流目标标签
```
例如
```c
int a,b,c;
```
变量a，b，c的类型来自左边的int，这就是典型继承属性

### 依赖图与属性求值顺序
属性不能随便算，比如：
```
E.val = E1.val + T.val
```
必须先算出E1.val 和 T.val ，才能算 E.val

课件引入依赖图 Dependency Graph 来表示属性实例之间的信息流，如果属性 a2 的计算依赖属性 a1 ，就画一条从 a1 到 a2 的边。属性值应按照依赖图的拓扑序计算；如果依赖图中存在环，属性计算无法进行

例如：
```
A → B
A.s = B.i
B.i = A.s + 1
```
A.s 依赖 B.i ，而 B.i 又依赖 A.s ，形成循坏依赖，因此这个SDD不能计算

### S属性SDD 与 L属性SDD
**S属性SDD**
S属性 SDD 只包含综合属性
特点：
- 所有信息都从子节点流向父节点
- 天然适合自底向上计算
- 可以和LR语法分析器结合
- 归约时计算属性
```
E → E1 + T { E.val = E1.val + T.val }
E → T      { E.val = T.val }
```
在LR分析中，当右部已经归约完成时，右部符号的属性已经在线上，可以计算左部属性
S属性SDD 可以和 LR 语法分析器一起实现，栈中的状态可以附加属性值，归约时按照语义规则计算新符号的属性

**L属性SDD**
L属性SDD允许综合属性，也允许受限的继承属性

对产生式：
`A -> X1 X3 ... Xn`
计算 `Xi.` 时，只能依赖
```
A 的继承属性
Xi 左边符号 Xj 的继承属性或综合属性
Xi 自身不成环的属性
```
继承属性只能从父节点或左侧兄弟传来，不能依赖右侧兄弟

保证了从左到右扫描时，所需信息已经可用。
S属性文法和L属性文法本质上是对属性依赖的限制，设计目的就是保证可行的无环计算顺序

### 消除左递归时，语义规则如何保持等价
语法分析中常常要消除左递归，例如
`E -> E + T | T`
原语义规则可能是
```
E -> E1 + T {E.val = E1.val + T.val}
E -> T {E.val = T.val}
```
消除左递归后
```
E -> T E'
E' -> + T E' | ε
```
原来的 E -> E + T 是不断向左累加，现在变成右递归或迭代结构，语义规则也要改写
```
给 A' 引入继承属性 inh，用来保存已经累积的结果
给 A' 引入综合属性 syn，用来返回最终结果
```
对表达式加法
```
E -> T E'{
	E'.inh = T.val
	E.val = E'.syn
}
E' -> + T E1'{
	E1'.inh = E'.inh + T.val
	E'.syn = E1'syn
}
E' -> ε{
	E'.syn = E'.inh
}
```
直观理解：
```
E'.inh 保存当前已经算出的部分和
每遇到一个 + T,就把 T.val 加进去
遇到 ε 时，说明结果，把inh 作为最终syn返回
```

### 语义规则、语义动作与副作用
语义规则 Semantic Rules 逻辑层面的属性计算公式
语义动作 Semantic Actions 实现层面的可执行代码

语义动作可能有副作用Side Effect 例如：
```
修改符号表
生成中间代码
输出错误信息
修改全局计数器
```
副作用必须受控，受控副作用需要满足：
```
动作依赖的信息已经计算完成
执行顺序确定
不会引入循环依赖或破坏属性求值结果
```

例如：
```
D -> T id
动作：addType(id.name, T.type)
```
这个动作必须在T.type已经计算完之后执行，否则就可能把错误类型写入符号表
在有回溯的解析器中直接执行副作用可能导致符号表状态和解析状态不同步，需要回滚或缓冲

### 逆波兰表达式RPN ： 语法制导翻译的入门例子
Reverse Polish Notation  也叫后缀表达式
中缀表达式 `a + b * c`
后缀表达式 `a b c * +`
语义规则可写为
```
E → E1 + T { E.postfix = E1.postfix || T.postfix || "+" }
E → E1 - T { E.postfix = E1.postfix || T.postfix || "-" }
E → T      { E.postfix = T.postfix }

T → T1 * F { T.postfix = T1.postfix || F.postfix || "*" }
T → T1 / F { T.postfix = T1.postfix || F.postfix || "/" }
T → F      { T.postfix = F.postfix }

F → (E)    { F.postfix = E.postfix }
F → id     { F.postfix = id.name }
```
于是
```
b * c → b c *
a + b * c → a b c * +
```
语义分析或翻译可以借助文法结构，把程序结构一步步转换成更适合机器处理的表示
