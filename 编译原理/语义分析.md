[[编译原理]]
**为什么要语义分析：CFG只能描述语法，不能描述全部语义**
上下文无关文法 Context - Free Grammar 简称CFG 很适合描述程序的结构
例如
E → E + T
E → T
T → id
它能说明 a+b 是一个表达式，但不能说明
```
a 是否声明?
a 是 int 还是 float?
+ 能不能作用在这两个类型上?
表达式结果放在哪里?
```

上下文无关文法只能描述语法，因此需要结合语义规则来完成语义分析与中间代码生成。
语法制导定义SDD正是为了解决这个问题：把属性和语义规则附着到文法符号和产生式上

`语法树+属性计算=语义分析`

## 语法制导定义 SDD 与语法知道翻译 SDT

### SDD ：Syntax-Directed Definition 语法制导定义

SDD是一种形式化描述方法
`SDD = CFG + 属性 + 语义规则`
它把属性附加到文法符号上，并把语义规则附加到产生式上

例如：
```
E → E1 + T
E.type = if E1.type == T.type then T.type else float
E.place = newTempPlace()
```
这里
```
E.type 表示表达式 E 的类型
E.place 表示表达式 E 的结果存放位置
```
如果有
```
F → id
F.type = lookupIDTable(id.lexValue) -> type
F.place = lookupIDTable(id.lexValue) -> address
```
那么语义分析就通过符号表查出标识符的类型和地址

### 属性：语义分析的核心信息载体

属性 Attribute  是与文法符号相关联的值，用来存储语义信息。常见属性包括
```
type 类型
place 值存放位置
addr 地址
code 生成的中间代码
val 表达式值
width 类型宽度
offset 相对地址偏移量
entry 符号表条目
```
例如
`E.val = E1.val + T.val`
这表示父节点E的值依赖于子节点E1和T的值，属性之间可能存在依赖关系，因此需要确定属性计算顺序

### 综合属性和继承属性
**综合属性**
综合属性从子节点向父节点传递，适合表达
```
表达式的值
表达式的类型
表达式生成的代码
表达式结果位置
```
例子
```
E → E1 + T (E.val = E1.val + T.val)
E → T (E.val = T.val)
T → num (T.val = num.lexval)
```

其中 `val` 就是综合属性，计算方向是自底向上。综合符的综合属性通常由词法分析获得，例如`digit.lexval`

**继承属性 Inherited Attribute**
继承属性从父节点或左兄弟节点向当前节点传递，适合表达
```
声明语句中的类型环境
当前作用域符号表
数组或记录的上下文
已经累计的中间结果
控制流目标标签
```
例如
```c
int a,b,c;
```
变量a，b，c的类型来自左边的int，这就是典型继承属性

### 依赖图与属性求值顺序
属性不能随便算，比如：
```
E.val = E1.val + T.val
```
必须先算出E1.val 和 T.val ，才能算 E.val

课件引入依赖图 Dependency Graph 来表示属性实例之间的信息流，如果属性 a2 的计算依赖属性 a1 ，就画一条从 a1 到 a2 的边。属性值应按照依赖图的拓扑序计算；如果依赖图中存在环，属性计算无法进行

例如：
```
A → B
A.s = B.i
B.i = A.s + 1
```
A.s 依赖 B.i ，而 B.i 又依赖 A.s ，形成循坏依赖，因此这个SDD不能计算
