#### 1. A、B：代码区
A、B 位于代码区，不是变量存储单元，而是程序指令所在区域。
所以可以写成源代码层面的指令：

|位置|内容|
|---|---|
|A|代码：`int i = 100;`|
|B|代码：`int j = 20;`|

#### 2.C：静态数据区
```c
int m = 0;
```
全局变量具有静态存储期，放在静态数据区。
全局变量属于静态分配，编译时即可确定存储位置；而局部名字通常在栈中分配。

#### 3. D、E、F、G：sub 函数栈帧
题目给出的栈帧结构是：
```
高地址
局部变量
BP 的值
返回地址
返回值
实参
低地址
```
`sub` 函数中：
```c
int sub(int a, int b) {
    int res = 0;
    res = a - b;
    return res;
}
```

|位置|内容|解释|
|---|---|---|
|D|局部变量 `res = 80`|第 12 行已经计算完成|
|E|保存的旧 BP 值|用于函数返回时恢复调用者的 BP|
|F|返回地址|返回到 `main` 中调用 `sub(i, j)` 之后继续执行的位置|
|G|返回值槽：`null`|第 13 行 `return res;` 还没执行|

这里的 E 不是当前 BP 寄存器本身，而是**栈帧中保存的调用者 BP 值**。当前 BP 寄存器的位置另算。

### SP 和 BP 的位置
题设规定：

> BP 指向栈底，SP 指向栈顶。  
> 栈从低地址向高地址增长。

当前正在执行 `sub`，所以当前活动记录是 `sub` 的栈帧。
过程调用和返回由控制栈管理，每个活跃活动对应一个活动记录；当前未结束的调用会保留在栈上。

由于栈从低地址向高地址增长，所以：
```
高地址
↑
SP 指向 sub 栈帧顶部，也就是 D 的上方
D: res = 80
E: 保存的旧 BP
F: 返回地址
G: 返回值 null
b = 20
a = 100
BP 指向 sub 栈帧底部，也就是 a 的下方
↓
低地址
```

|项目|答案|
|---|---|
|SP|指向 D 的上方，即 sub 栈帧顶部|
|BP|指向 a 的下方，即 sub 栈帧底部|

### 最关键的陷阱：引用计数无法处理循环垃圾

从**可达性**角度看，删除 `X → A` 后，已经没有任何根引用能够到达 A、B、C、D、E、F、G、H、I。

也就是说，如果使用的是**标记-清除 Mark-Sweep** 这类追踪式垃圾回收，从根 X 出发遍历，最终所有对象都不可达，所以这些对象都应该被回收。

但是本题考的是**引用计数**。引用计数只看“当前还有没有指针指向我”，不看“这些指针本身是否从根可达”。因此：

```
D → G → H → D
```

构成循环引用。
所以 D、F、G、H、I 虽然已经从根不可达，但它们互相引用，引用计数不为 0，因此不会被引用计数法删除。这就是引用计数算法的经典缺陷：**循环垃圾无法自动回收**。
