[[编译原理]]
词法分析将源代码字符流切成了Token序列：
```c
position = initial + rate * 60
```
词法分析得到：

```
<id,1> <=,> <id,2> <+,> <id,3> <*,> <number,4>
```

语法分析器的任务就是判断该 Token序列是否符合语言的语法规则，并构造出语法结构

类型检查、语义分析、中间代码生成等有时也会和语法分析交错完成

```
词法分析：字符流 -> token序列
语法分析：token序列 -> 语法树 / AST
语义分析：语法树 + 符号表 -> 类型检查、作用域检查
```

语法分析不仅要知道Token串是否合法，还需要知道它如何被理解

这涉及到 **文法、推导、语法树、二义性、优先级、结合性** 等概念

## 语法和文法

**语法 Syntax** 
语法树语言中“规则结构”等总称，它关心的是
```
哪些符号组合是合法的？
哪些符号组合是不合法的？
```

**文法 Grammar**
文法是描述语法的**形式化工具**。

**上下文无关文法CFG Context Free Grammar**
上下文无关：推导过程不依赖左右邻居

`G = (V , ∑ , P , S)`
```
V : 非终结符集合
∑ : 终结符集合
P : 产生式集合
S : 开始符号
```

产生式形如
`A -> α`

其中 `A` 是一个非终结符，`α` 是终结符和非终结符组合成的串
CFG的关键为：**产生式左部只能是单个非终结符**

**终结符 Terminal**
终结符是不能再继续展开的基本符号，通常对应词法分析产生的 Token
例如
`id + * ( ) if else number`

**非终结符 Non-terminal**
非终结符表示某类语法结构，可以继续展开

例如
```
E 表达式
T 项
F 因子
Stmt 语句
Decl 声明
E -> E + T
E -> T
T -> T * F
T -> F
```
在这里，`E,T,F` 是非终结符,`+ *`是终结符

#### 为什么叫上下文无关
假设产生式
`A -> γ`
无论其左边右边是什么，都可以把A 替换为  γ 
例如
`α A β`
只要有规则`A -> γ` 就可以推出 `α γ β`

这里 `α` 和 `β` 就是 `A` 的上下文。但 CFG 不关心这个上下文，只关心 `A` 自己能不能替换。因此叫**上下文无关**。

### 推导、句型、句子、语言
这些是语法分析最容易混的概念

### 推导 Derivation
推导就是从开始符号出发，不断用产生式替换非终结符，最终得到某个串的过程

例如文法
`E → -E | E + E | E * E | (E) | id`
从 `E` 推导出 `-(id)`：

```
E
⇒ -E
⇒ -(E)
⇒ -(id)
```

从开始符号不断替换，就可以得到文法的不同句型

### 句型 Sentential Form
如果从开始符号 `S` 能推导出 `α`，那么 `α` 就是一个句型。

句型可以包含非终结符，也可以包含终结符。
例如
```
E + T
id + T
id + id
```
都可能是某个文法的句型

### 句子 Sentence
如果一个句型中不再包含非终结符，只包含终结符，那么它就是句子

例如 `id + id * id` 就是一个句子

### 语言 Language
一个文法`G`能生成的所有句子的集合，叫做这个文法定义的语言，记作`L(G)`

### 最左推导、最右推导和语法树
**最左推导**
每一步都替换当前句型中**最左边的非终结符**
`E ⇒ E + T ⇒ T + T ⇒ F + T ⇒ id + T ⇒ id + F ⇒ id + id`

**最右推导**
每一步都替换成当前句型中**最右边的非终结符**
`E ⇒ E + T ⇒ E + F ⇒ E + id ⇒ T + id ⇒ F + id ⇒ id + id`

**语法分析树 Parse Tree / CST**
语法分析树是推导过程的树型表示
基本规则：
```
根节点是开始符号
内部节点是非终结符
叶子节点可以是终结符、非终结符或ε
每个内部节点表示某个产生式的一次应用
```

例如
```
E → E + T
E → T
T → F
F → id
```
表达式
`id + id`

对应的树：
```
        E
      / | \
     E  +  T
     |     |
     T     F
     |     |
     F    id
     |
    id
```

### 语法树和推导的关系
一个语法树可能对应多个推导顺序。因为你可以先展开左边，也可以先展开右边
但是一颗语法树唯一对应一个最左推导，也唯一对应一个最右推导

### CST 和 AST 的区别

语法分析树也叫 **具体语法树CST** ，完整保留文法推导细节
但实际编译器中更常用 **AST抽象语法树** 。 AST去掉了冗余语法细节，只保留语义相关结构。
例如：
```
a + b * c
```

CST 可能非常复杂：

```
E → E + TT → T * FF → id...
```

AST 则可以简化成：

```
      + 
     / \
    a   *
       / \
      b   c
```

所以：

```
CST 关注“怎么推导出来”；
AST 关注“程序真正是什么意思”。
```
## 文法二义性 Ambiguity
如果一个文法可以为同一个句子生成两棵不同的语法树，那么这个文法就是二义的
例如
```
E -> E + E
E -> E * E
E -> id
```
对于
`id + id * id`
可能有两种结构：
```
id + (id * id)
```
也可能是：
```
(id + id) * id
```
这就是二义性。

程序不能允许这种不确定性

### 表达式二义性的解决方式
通常通过文法结构体现优先级和结合性

```
E → E + T | T
T → T * F | F
F → (E) | id

E 处理加法
T 处理乘法
F 处理括号和标识符
```

### dangling else 问题
```c
if (x)
	if(y)
		A;
	else
		B;
```
对于else到底是匹配外层if还是内层if？
通常语言标准规定：
`else匹配最近的尚未匹配的if`

这就是通过语言规则消解二义性

## 正则表达式和上下文无关文法的关系
词法分析用正则表达式比较多，语法分析用上下文无关文法比较多

因为
```
正则表达式适合描述简单、线性的词法结构
上下文无关文法适合描述嵌套、递归的语法结构
```

每个正则语言都可以用CFG描述；
但不是每个CFG语言都可以用正则表达式描述

## 文法与处理：消除二义性、左递归、左公因子

### 左递归 Left Recursion

如果一个非终结符 A 可以推导出以自己开头的串：
`A -> ...  -> Aα`
如果直接存在`A -> Aα | β` 则称为立即左递归 
例如
`E -> E + T | T`

**为什么自顶向下分析不能有左递归**
递归下降中左递归会形成死循环，因此必须先消除

**消除公式**
如果有
`A → Aα1 | Aα2 | ... | Aαm | β1 | β2 | ... | βn`
其中 `β` 不以 `A` 开头，那么改写为：

```
A  → β1 A' | β2 A' | ... | βn A'
A' → α1 A' | α2 A' | ... | αm A' | ε
```

例子：
`E -> E + T | T`
改为
```
E -> T E'
E' -> + T E' | ε
```

### 间接左递归
不一定是直接写成 `A -> Aα` 才叫左递归
例如：
```
A → B a
B → A b | c
```

在这有
`A ⇒ B a ⇒ A b a`
这就是间接左递归

消除间接左递归通常要
```
1. 给终结符排序 A1,A2,...,An
2. 按顺序替换前面非终结符的产生式
3. 把间接左递归转化成直接左递归
4. 再用直接左递归消除方法处理
```

### 左公因子 Left Factoring
如果两个产生式有相同前缀
`A -> αβ1 | αβ2`
自顶向下分析看到 α 时无法确定使用哪一条规则
因此需要提取左公因子：
```
A  → α A'
A' → β1 | β2
```

## 自顶向下语法分析
自顶向下就是从开始符号出发，一步步展开，尝试生成输入串

```
从根节点S开始
不断选择产生式展开非终结符
直到叶子节点正好等于输入 Token 序列
```

在构造语法树时：
从根到叶
从大结构到小结构

### 递归下降分析 Recursive Descent Parsing
递归下降分析时最直观的自顶向下方法

核心思想：
每个非终结符对应一个函数

例如文法：
```
E -> T E'
E' -> + T E' | ε
T -> id
```
写成：
```
parseE():
	parseT()
	parseEPrime()

parseEPrime():
	if lookahead == '+':
		match('+')
		parseT()
		parseEPrime()
	else:
	// ε

parseT():
	match(id)
```
递归下降分析框架由一组过程组成，每个非终结符对应一个过程；
程序从开始符号对应的过程开始执行，遇到非终结符就递归调用对应过程，遇到终结符就匹配输入

### 递归下降的问题：回溯
如果一个非终结符有多个候选式，分析器可能不知道该选哪个

```
S -> c A d
A -> a b | a
输入:
c a d 
分析A时如果先尝试 A -> a b
匹配 a成功，但是匹配b失败，于是回退，再尝试 A -> a
```
这就叫 **回溯**
回溯的问题：
- 效率低
- 实现复杂
- 可能重复扫描输入

实际希望做到**无回溯**

## 预测分析法 Predictive Parsing
预测分析法是一种无回溯的自顶向下语法分析方法
其核心为：
```
根据当前栈顶非终结符 A 和当前输入符号 a，
唯一确定应该使用哪条产生式
```
即构造一张预测分析表
`M[A, a] = A → α`

预测分析法的要求是文法必须是 LL(1) 文法；如果有多个可能的产生式，预测分析法无能为力。它只看一个前瞻符号，就要决定使用哪条产生式

**LL(1)的含义**
> 从左到右读输入，每一步构造最左推导，只向前看一个 Token 就能决定用哪条产生式。
```
第一个 L：Left-to-right，从左到右扫描输入
第二个 L：Leftmost derivation，构造最左推导
1：只看 1 个前瞻符号
```

### 预测分析的基本流程
1. 预处理文法
   - 消除左递归
   - 提取左公因子
   - 转换为 LL(1) 文法

2. 构造预测分析表
   - 行：非终结符
   - 列：终结符和 $
   - 表项：应使用的产生式

3. 分析输入
   - 使用一个栈保存符号
   - 根据栈顶符号和当前输入符号查表
   - 直到栈空且输入读完

## FIRST 集
FIRST 集用于回答
> 某个符号串最开始可能出现什么终结符？

`FIRST(α) = α 能推导出的串的首终结符集合`

如果 `α` 能推出空串，还要包含 `ε`。

### 基本规则

**终结符**
`FIRST(a) = {a}`

**空串**
`FIRST(ε) = { ε }`

**非终结符**
如果
`A -> α`
则将
`FIRST(α) - {ε}` 加入 `FIRST(A)`
如果 `α` 能推出 `ε`，则把 `ε` 加入 `FIRST(A)`。

**符号串**
对于
`X1 X2 ... Xk`
先看`X1`
`把 FIRST(X1) - {ε} 加入`
如果 X1 能推出 ε ，再看 X2

## FOLLOW集
FOLLOW集用于回答
> 某个非终结符后面可能紧跟什么终结符？

形式上：
`FOLLOW(A) = 在某些句型中可能出现在 A 右边的终结符集合`

如果 A 能出现在句子末尾，那么 $ 属于 FOLLOW(A)

### 基本规则

**开始符号**
`$ ∈ FOLLOW(S)`
因为开始符号推导出的完整句子后面就是结束输入标记

**A → α B β**
那么
`FIRST(β) - {ε} 加入 FOLLOW(B)`
因为 `B` 后面接着 `β`，所以 `β` 的开头可能跟在 `B` 后面。

**β 能推出 ε**
如果 `A → α B β` 且 `β ⇒* ε`
那么 `FOLLOW(A) 加入 FOLLOW(B)`
因为 `β` 可以消失，所以 `B` 后面可能直接跟着原来 `A` 后面的符号。

## SELECT 集与LL(1) 判断

对于产生式
A -> α

SELECT 集回答：
> 当前输入符号是什么时，可以选择这条产生式？

公式：
```
SELECT(A → α) = FIRST(α) ∪ FOLLOW(A) 如果 ∈ FIRST(α)  
SELECT(A → α) = FIRST(α) 如果 ε ∉ FIRST(α)
```

也就是说：
如果α不能推出空串，看 FIRST(α)
如果α能推出空串，还要看 FOLLOW(A)

因为当右部能为空时，是否使用这条空产生式，要看A后面允许跟什么符号

### LL(1)判别条件
对于同一个非终结符 A 的不同候选式：
A -> α1
A -> α2
...
A -> αn

它们的SELECT集必须两两不相交
即
SELECT(A -> αi) ∩ SELECT(A → αj) = ∅
如果有交集，说明看到同一个输入符号时，分析器不知道选哪条产生式，预测分析表会冲突，所以不是LL(1)

## 构造 LL(1) 预测分析表
对于每条产生式
`A -> α`

1. 对所有 a ∈ FIRST(α) - {ε}
   填 M[A, a] = A → α

2. 如果 ε ∈ FIRST(α)
   对所有 b ∈ FOLLOW(A)
   填 M[A, b] = A → α

如果一个表格中要填入两条不同产生式，就产生冲突，说明文法不是LL(1)

## LL(k) 与 LL(\*)  自顶向下分析的扩展

LL(1) 只看一个前瞻符号，有时不够

例如
```
stmt → ID = expr
stmt → ID ( args )
```
因此需要再看第二个符号才能判断

这就是LL(k)的思想：向前看 k 个符号
但k增大后，分析表和预测逻辑会变得复杂
LL(\*)思想：不固定k，而是用 DFA 识别任意长度的前缀来预测分支；ANTLR中就使用了类似思想
