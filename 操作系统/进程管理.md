[[处理器管理]]
#进程管理
## 进程及其状态

为什么要提出**进程**？
程序本身只是存放在磁盘上的一段静态代码，例如一个`.exe`文件，一个Linux可执行文件、一个Java程序文件

当程序真正运行起来的时候，操作系统不仅要知道它的代码在哪里，还要知道它的运行状态（Pc，内存，文件管理，资源占用，状态等）

因此，操作系统需要为“正在运行的程序”建立一个管理实体，即 **进程**。

### 进程的定义

> 进程是一个具有一定独立功能的程序关于某个数据集合的一次运行活动，是操作系统进行资源分配和处理器调度的一个独立单位

**进程是程序的一次运行活动**
程序是静态的，进程是动态的

```
程序：qq.exe
进程：你当前正在运行的QQ
程序 + 运行状态 + 资源 + CPU现场 = 进程
```

**进程依赖某个数据集合**
同一个程序处理不同数据，可能形成不同进程
代码可以相同，但是数据集合不同，运行活动也不同

**进程是资源分配和处理器调度单位**
传统操作系统中，进程既是
**资源分配单位** ： 内存、文件、设备等资源分配给进程
**处理器调度单位**：CPU在多个进程之间切换

在多线程环境下，情况有所变化：进程更多作为资源分配单位，线程成为CPU调度多基本单位。

#### 进程的形式化定义

一个进程包括五个实体部分：
```
P：操作系统管理运行程序的数据结构
C：运行程序的内存代码
D：运行程序的内存数据
R：通用寄存器信息
PSW：程序状态字信息
```

## 进程的三态模型

进程状态包括：
运行态 就绪态 等待态/阻塞态

**运行态**
运行态指进程正在占用CPU执行
在单核CPU上，某一时刻通常只有一个进程处于运行态
在多核CPU上，可以有多个进程或线程同时处于运行态，因为多个CPU核心可以同时执行

**就绪态**
就绪态指进程已经具备运行条件，但暂时没有CPU

**等待态**
等待态是指因为等待某个事件而暂时不能运行

### 三态转换关系

```
就绪态 ➡️ 运行态 ： 被调度程序选中
运行态 ➡️ 就绪态 ： 时间片用完 / 被抢占
运行态 ➡️ 等待态 ： 等待 I/O / 等待资源 / 等待信号
等待态 ➡️ 就绪态 ： 等待事件完成
```

## 进程挂起

### 为什么需要挂起？
操作系统无法完全预知进程数量和资源需求。如果系统运行中出现内存不足、资源紧张、负载过高等情况，继续让所有进程都驻留内存可能导致性能下降甚至死锁

解决方法为剥夺某些进程的内存和其他资源，把它们掉入OS管理的对换区，使其不参加进程调度，等合适时再调入内润、恢复资源、参与运行，这就是进程挂起

### 挂起态和等待态的区别

等待态进程占有已申请到的资源处于等待，而挂起态进程没有任何资源

### 挂起就绪态和挂起等待态

**挂起就绪态**：进程逻辑上已经就绪，但不在内存中，不能参与CPU调度

**挂起等待态**：进程既在等待状态，又被唤出内存

一般选择等待态进程进入挂起等待态，也可以选择就绪态进程进入挂起就绪态；等待事件结束后，挂起等待态进入挂起就绪态；一般选择挂起就绪态进程予以恢复

## 进程上数据描述

### PCB 进程控制块

PCB Process Control Block 进程控制块，是操作系统用于记录和刻画进程状态及环境信息的数据结构
借助PCB，OS可以全面管理进程的物理实体，刻画进程执行现状，控制进程执行

没有PCB，操作系统就不知道系统中有哪些进程，也不知道它们的状态、资源、优先级和运行现场

**PCB的三类信息**

**标识信息**
标识信息用于唯一标识进程
包括：系统分配的进程标识号PID
进程组标识号 用户定义的进程名 进程组名

**现场信息**
现场信息用于保存进程运行时的CPU现场

包括：用户可见寄存器内容 控制与状态寄存器内容
PC IR PSW 用户栈指针 核心栈指针

**控制信息**

用于支持进程管理和调度

调度相关信息：状态、等待时间或原因、优先级
进程组成信息：代码和数据地址、外存映像地址
队列指引元：进程队列指针、父子兄弟进程制作
通信相关信息：消息队列、信号量、锁
进程特权信息：内存访问权限、处理器特权
处理器使用信息：占用处理器、时间片、CPU使用时间、记账信息
资源清单信息：正占用资源、已使用资源

### 进程映像

某一时刻进程的内容及其执行状态集合，也称进程的内存映像
包括PCB、进程程序块、进程数据块、核心栈等

**PCB** 保存进程标识信息、状态信息和控制信息
**进程程序块** 进程执行的程序空间，也就是代码段
**进程数据块** 进程处理的数据空间，包括数据、用户栈、可修改程序等
**核心栈** 进程中内核模式下运行时使用的堆栈
当进程发生系统调用、中断或异常进入内核时，会使用核心栈保存内核调用过程、现场信息等

### 进程上下文

进程执行需要环境支持，包括CPU现场和Cache中的执行信息，OS中的进程物理实体和支持进程运行的环境合成进程上下问

**用户级上下文**
包括用户程序块 用户数据区 用户栈 用户共享内存
这些时进程中用户态运行时直接使用的内容

**寄存器上下文**
包括PSW 栈指针 通用寄存器 程序计数器PC 条件码等
这些决定进程执行到哪里、当前CPU现场时什么

**系统级上下文**
包括PCB 内存区表 核心栈
与内核管理相关的数据结构

进程切换的本质就是上下文切换
如果上下文保存不完整，进程恢复时就无法正确执行

## 进程管理的实现
进程不是只靠理论模型存在，操作系统内部需要一套软件模块和数据结构来实际管理进程
其中最核心的为
```
队列管理
进程管理原语
进程调度
```

### 进程队列模型

为什么需要队列 系统中同时有很多进程 操作系统需要把这些进程组织起来，于是使用队列

**就绪队列**
就绪队列中存放所有已经具备运行条件、只等待CPU运行的进程
调度程序会从就绪队列中选择一个进程运行

**等待队列**
等待队列中存放正在等待某个事件的进程
通常不同事件有不同的等待队列

```
磁盘 I/O 等待队列：P2 → P6
键盘输入等待队列：P4
网络数据等待队列：P7 → P9
信号量 S 等待队列：P10
```
当对应事件完成时，OS从相应等待队列中取出进程，把它们转入就绪队列

**队列如何组织？**
进程队列可以按需组织为先进先出或优先队列，队列中的进程可以通过PCB中队列指引元采用单/双指引元或索引连接

PCB中会有一些指针资源，用来把进程传入不同队列

```
PCB(P1).next → PCB(P2)
PCB(P2).next → PCB(P3)
```
这样操作系统不需要移动整个进程，只需要移动PCB指针，就能完成入队、出队

### 进程控制原语
原语是由若干指令构成的、完成特定功能的程序，在执行上具有不可分割性，也就是原子性。原语可以通过关中断实现

**为什么需要原语？**
进程控制会修改很多关键数据结构，如果执行到一半被中断，系统数据结构可能出现错误
因此操作必须保证原子性

**进程创建原语**
流程：进程表加一项，申请PCB并初始化，生成标识，建立映像，分配资源，移入就绪队列

**进程撤销原语**
流程：进程撤销需要从队列中移除、归还资源、撤销标识、回收PCB、移除进程表项

**进程阻塞原语**
进程阻塞发生在运行进程因为等待事件而不能继续运行时
流程：保存当前进程现场、修改PCB状态为等待态、记录等待原因、把PCB加入对应等待队列，调用调度程序选择其他进程运行

**进程唤醒原语**
当等待事件完成时，操作系统要唤醒等待进程
流程：从等待队列移出、修改PCB、移入就绪队列
如果该进程优先级高于运行进程，可能触发抢占

**进程挂起和激活原语**
进程挂起原语：修改进程状态、把进程从原队列移到挂起队列、收回内存等资源、把进程映像换出到对换区
进程激活流程：分配内存、把进程从外存调入内存、修改状态、加入相应队列

进程挂起会修改状态并出入相关队列，收回内存等资源送至对换区；进程激活则分配内存，修改状态并出入相关队列

## 进程切换和模式切换

模式切换：CPU在用户态和内核态之间切换
进程切换：CPU从一个进程切换到另一个进程

### 模式切换
模式切换又称处理器状态切换，包括：
用户模式 ➡️ 内核模式
内核模式 ➡️ 用户模式

用户模式到内核模式由中断、异常、系统调用触发；
内核模式到用户模式由OS执行中断返回指令触发

中断装置完成正向模式切换，包括处理器模式转为内核模式、保存当前进程PC/PSW到核心栈、转向中断/异常/系统调用处理程序；中断返回指令完成逆向模式转换，包括弹出PSW/PC、处理器模式转为用户模式

### 进程切换

进程切换时从正在运行的进程中收回处理器，让待运行进程占有吹起运行，其本质上被中断运行进程与待运行进程的上下文切换

**为什么进程切换开销大**

因为进程切换不是简单的改一个变量，而是需要很多工作
调度算法不能过于频繁地切换，否则系统时间会大量消耗在管理上，而不是执行用户程序

**流程**
```
1. 中断 / 异常 / 系统调用触发正向模式切换，并压入 PSW / PC
2. 保存被中断进程的现场信息
3. 处理具体中断 / 异常
4. 把被中断进程的系统堆栈指针 SP 保存到 PCB
5. 调整被中断进程的 PCB 信息，例如进程状态
6. 把被中断进程的 PCB 加入相关队列
7. 选择下一个占用 CPU 运行的进程
8. 修改被选中进程的 PCB 信息，例如进程状态
9. 设置被选中进程的地址空间，恢复存储管理信息
10. 恢复被选中进程的 SP 到处理器寄存器
11. 恢复被选中进程的现场信息进入处理器
12. 中断返回指令触发逆向模式转换，并弹出 PSW / PC
```

```
中断进入内核
      ↓
保存旧进程上下文
      ↓
处理事件
      ↓
修改旧进程状态和队列
      ↓
调度新进程
      ↓
恢复新进程上下文
      ↓
返回用户态运行新进程
```

### 进程切换发生的时机
进程切换一定发生在中断、异常或系统调用处理过程中
常见情况包括：阻塞式系统调用、虚拟地址异常、时间片中断、I/O中断后发现更高优先级进程、终止系统调用、不能继续执行的异常等

**阻塞式系统调用**

```
进程A 执行 read()
需要等待磁盘
A 进入等待态
调度进程 B
```

**时间片中断**

```
进程 A 时间片用完
时钟中断进入内核
A：运行态 ➡️ 就绪态
调度 B
```
分时操作系统中最常见的切换原因

**I/O中断唤醒高优先级进程**

```
进程 A 正在运行
进程 B 原来等待磁盘
磁盘 I/O 完成中断发生
B 被唤醒，进入就绪态
如果 B 优先级高于 A
A 被抢占
B 开始运行
```

**异常导致终止**

```
进程 A 执行非法指令
产生异常
OS 终止 A
调度其他进程运行
```

**缺页异常**
缺页异常有两种情况：
如果页面可以调入，当前进程可能等待磁盘 I/O，发生切换
如果页面已经可以快速处理，也可能处理完后恢复原进程

```
进程 A 访问缺页
      ↓
缺页异常
      ↓
OS 调页
      ↓
A 可能进入等待态
      ↓
调度 B
```

### 中断处理不一定导致进程切换
有些中断或异常不会引起进程状态转换，不会引起进程切换，只是在处理完成后把控制权交回被中断进程

### 模式切换与进程切换
|对比项|模式切换|进程切换|
|---|---|---|
|含义|CPU 在用户态和内核态之间切换|CPU 从一个进程切换到另一个进程|
|是否换进程|不一定|一定|
|触发原因|中断、异常、系统调用、中断返回|调度程序决定更换运行进程|
|开销|相对较小|较大|
|是否保存完整上下文|通常保存必要状态|必须保存和恢复完整进程上下文|
|例子|read 系统调用进入内核后又返回原进程|时间片到，A 切换为 B|
