[[处理器管理]]
当系统中有多个进程或线程都想使用CPU时，操作系统应该让谁运行、运行多久、什么时候换人？

## 什么是处理器调度
**处理器调度**就是操作系统按照某种策略，把CPU分配给某个可运行进程或线程的过程
在单核CPU上，同一时刻只能有一个进程或线程真正占用CPU，可是系统中通常同时存在很多进程

这些进程看起来“同时运行”，实际上是操作系统通过快速切换CPU使用权，让它们交替执行
所以处理器调度要解决的问题是：

```
谁可以进入系统？
谁可以留在内存？
谁现在获得CPU？
谁应该被抢占？
一个进程应该运行多久？
```

## 处理器调度的层次

```
高级调度 / 长程调度 / 作业调度
中级调度 / 中程调度 / 平衡负载调度
低级调度 / 短程调度 / 进程调度
```

### 高级调度：决定能不能进入系统
**高级调度**又叫 **长程调度** 或 **作业调度**

决定能否加入到执行的进程池中。批处理系统中，高级调度会从后备作业队列中选择作业进入主存，并为作业运行前做准备、运行后做善后工作
分时系统中，高级调度可能决定是否接受中断用户连接、命令能否被系统接纳并构成进程、新建态进程是否加入就绪队列

**批处理系统中的高级调度**
在批处理系统中，用户提交作业后，作业不一定立刻进入内存运行
可能有一个后备作业队列：
`后备作业队列:J1,J2,J3,J4,...`
高级调度负责从中选择一部分进入内存

选择时可能考虑：
- 作业优先级
- 预计运行时间
- 资源需求
- I/O 需求
- 系统负载
- 公平性

例如：系统可能不希望同时放入太多CPU密集型作业，否则I/O 设备利用率很低；也不希望同时放入太多I/O密集型作业；否则CPU可能空闲

**分时系统中的高级调度**
在分时系统中，高级调度表现为：
```
是否允许一个用户登录？
是否允许一个命令创建进程？
系统是否还有足够资源接纳新进程？
```

例如：系统负载过高时，可能暂时拒绝新的登录或限制新任务创建

**高级调度的特点**
高级调度频率较低
因为作业进入系统不是每毫秒都发生的事情，它更多影响系统整体负载和作业吞吐量

`高级调度：控制系统入口`

### 中级调度：决定“能不能留在内存”
**中级调度** 又叫 **中程调度** 或 **平衡负载调度**

核心问题：
```
哪些进程应该驻留在内存中？
哪些进程应该被换出道外存挂起？
```
中程调度决定主存中的可用进程集合；引入中级调度是为了提高内存利用率和作业吞吐量，它决定哪些进程被允许驻留主存参与竞争处理器及其他资源，并通过把部分进程换出主存进入挂起状态来平顺系统负载

**为什么需要中级调度**
假设系统中有很多进程都在主存中：
`P1,P2,P3,P4,...,P100`
如果内存不足，系统会频繁换页，导致大量时间花在磁盘的换入换出上，CPU反而没多少时间真正执行程序，这种现象严重时叫 **抖动**

中级调度的作用就是：
```
把一部风进程换出内存
降低内存压力
减少系统抖动
提高整体吞吐量
```

**中级调度和挂起状态**
中级调度和 **挂起状态** 密切相关
它可能导致：
```
就绪态 -> 挂起就绪态
等待态 -> 挂起等待态
挂起就绪态 -> 就绪态
挂起等待态 -> 等待态 或 挂起就绪态
```

一个等待 I/O 的进程暂时用不到 CPU，也许可以先换出内存
`等待态 -> 挂起等待态`
等它等待的事件完成后：
`挂起等待态 -> 挂起就绪态`
之后系统负载降低，再把它调回内存：
`挂起就绪态 -> 就绪态`

**中级调度的特点**
中级调度频率介于高级调度和低级调度之间
它不是每次CPU切换时都发生，但比高级调度更频繁

`中级调度：控制内存驻留集合`

### 低级调度：决定“现在谁用CPU”
**低级调度** 又叫 **短程调度、进程调度** 或 **处理器调度**

它的核心问题时：
`当前CPU应该分配给哪个就绪进程或内核级进程？`

低级调度按照某种原则把处理器分派给就绪态进程或内核级线程。进程调度程序又称分派程序，是操作系统中实现处理器调度的程序，是操作系统最核心部分

**低级调度的对象**
在传统进程模型中，调度对象是：`就绪态进程`
在支持内核级线程的系统中，调度对象通常是：`内核级线程`

**低级调度的主要功能**
```
记住进程或内核级线程的状态
决定某个进程或内核级线程什么时候获得处理器
决定它占用处理器多长时间
把处理器分派给进程或内核级线程
收回处理器
```

**低级调度的特点**
低级调度的频率最高
当发生以下事件时，都可能触发低级调度：
```
当前进程终止
当前进程阻塞
时间片用完
高优先级进程到达
I/O 完成唤醒进程
中断或异常处理后需要重新选择运行对象
```
低级调度：控制CPU当前归属

|调度层次|别名|决定什么|主要对象|频率|
|---|---|---|---|---|
|高级调度|长程调度、作业调度|作业或进程能否进入系统|后备作业、新建进程|低|
|中级调度|中程调度、平衡负载调度|哪些进程驻留内存|挂起进程、内存中进程|中|
|低级调度|短程调度、进程调度|当前谁使用 CPU|就绪进程或内核级线程|高|
核心：
高级调度：进不进系统
中级调度：留不留内存
低级调度：用不用CPU

## 处理器调度算法的评价指标

### 资源利用率
资源利用率指CPU 、 I/O 设备等系统资源被有效利用等程度
期望目标：
```
CPU 不长时间空闲
I/O 设备不长时间空闲
CPU 和 I/O 尽可能并行工作
```

如果系统只运行CPU密集型进程，CPU很忙，但I/O设备可能空闲
如果系统只运行I/O密集型进程，进程经常等待I/O，CPU可能空闲
好的调度算法应该让CPU密集型任务和I/O密集型任务合理搭配，提高整体资源利用率

### 响应时间
**响应时间**是交互式系统中特别重要的指标

通常指：
`用户发出请求到系统首次响应之间的时间`

例如你点击一个按钮，系统多久有反应；你在中断输入命令，多久开始输出
对于交互式系统，响应时间越短，用户体验越好
所以分时系统、桌面系统、移动系统都非常重视响应时间

### 周转时间
**周转时间**指作业从提交给系统开始，到执行完成并获得结果为止的时间间隔

`周转时间 = 完成时间 - 到达时间`
例如
```
进程 0 秒到达，10秒完成
周转时间=10-0=10秒
```

平均周转时间越短，说明作业整体完成得越快

### 带权周转时间
`带权周转时间 = 周转时间 / 服务时间`
其中服务时间就是进程真正需要CPU运行的时间

带权周转时间反映了进程等待和运行之间的相对比例

### 吞吐量
**吞吐量**指单位时间内系统完成的进程或作业数量

`每分钟完成20个作业`
吞吐量越高，说明系统整体处理能力越强

### 公平性
公平性指每个用户、每个进程都应该获得合理的CPU份额，不能长期得不到运行机会
例如如果调度算法总是选择短作业，长作业可能一直等不到CPU，这就是 **饥饿**

## 抢占式调度与非抢占式调度

### 非抢占式调度
**非抢占式调度**指一个进程一旦获得CPU，除非它主动放弃CPU，否则调度程序不会强行剥夺它
它只有在以下情况下才会换进程
```
进程运行结束
进程主动阻塞
进程主动让出CPU
```
非抢占式调度中没有“运行态 -> 就绪态“的跳转，也就是说，进程不会因为时间片用完或更高优先级进程到达而被强行抢占

**非抢占式调度的风险**
- 一个进程可能长时间占用CPU
- 交互式响应变差
- 高优先级任务无法及时运行
- 系统实时性差
- 如果进程死循环，其他进程可能长期得不到 CPU

### 抢占式调度
**抢占式调度**指操作系统可以强行收回当前进程的CPU，转而调度另一个进程

触发抢占的情况包括：
```
时间片用完
更高优先级进程到达
中断唤醒了更重要的任务
实时任务需要立即运行
```

抢占式调度适合：
```
分时系统
交互式系统
实时系统
现代通用操作系统
```

## 优先数调度算法
**基本思想**
优先数调度算法根据分派给进程的优先数决定下一个运行进程

```
每个进程有一个优先级 / 优先数
调度程序选择优先级最高的进程运行
```

有的系统数字越大优先级越高
有的系统数字越小优先级越高

### 抢占式优先级调度
如果新到达进程优先级高于当前运行进程，当前进程会被抢占

```
P1 正在运行，优先级 3
P2 到达，优先级 8
如果数字越大优先级越高
则P2 抢占 P1 
```
这种方式响应高优先级任务快，适合实时任务

### 非抢占式优先数调度
当前进程一旦运行，就继续运行到结束或阻塞
即使高优先级进程到达，也要等待当前进程主动让出CPU

有点是实现简单、切换少
缺点是响应差

### 优先数的确定准则
```
进程负担任务的紧迫程度
进程的交互性
进程使用外设的额度
进程使用处理器的服务时间长短
进程进入系统的时间长短
```

**紧迫程度**
实时任务、系统关键任务通常优先级更高

**交互性**
交互式进程通常优先级较高

**外设使用额度**
频繁使用 I/O 的进程往往 CPU 使用时间短，应该尽快让它运行，以便它很快再次发出 I/O 请求，使CPU和I/O 并行
可以提高系统整体资源利用率

**CPU服务时间长短**
短作业或短进程优先可以降低平均周转时间

### 优先级调度的问题：饥饿

如果系统一直有高优先级进程到达，低优先级进程可能长期得不到CPU

解决方法之一：**老化** **aging**
> 进程等待时间越长，逐渐提高它的优先级

## 与服务时间相关的调度思想
计算时间短优先
剩余计算时间短优先
响应比高者优先
先来先服务

## FCFS 先来先服务
First Come First Served 先来先服务
当某个进程就绪时，加入就绪队列
当前正在运行的进程停止执行时，选择在就绪队列中时间最长的进程运行

队列结构通常是FIFO：
对头出队运行，队尾入队等待

FCFS通常是 **非抢占式调度**
进程一旦获得CPU，就运行到结束或阻塞

**FCFS的优点**
简单
公平直观
容易实现
开销小

**FCFS的缺点**
FCFS会导致短进程可能不得不等待很长时间，偏袒计算为主的进程，I/O多的进程不得不等待计算为主的进程做完

这叫 **护航效应** **convoy effect**
一个长CPU进程排在前面，后面一堆短I/O进程都要等它，导致
```
短作业响应慢
I/O 设备空闲
平均周转时间变大
交互体验差
```
## SPN 最短进程优先
Shortest Process Next 最短进程优先
也叫 SJF Shortest Job First 短作业优先

SPN是非抢占式调度，选择所需时间最短的进程，短进程会越过长进程优先获得调度

**SPN的优点**
平均周转时间通常较短
短作业响应快
理论上在所有进程同时到达且服务时间已知时效果很好

**SPN的缺点**
长作业饥饿：只要持续不断地提供更短的进程，长进程就有可能饿死

另外，操作系统通常无法准确知道进程未来还需要多少CPU时间
实际系统中常通过历史行为估计服务时间

### SRT 最短剩余时间优先
SRT Shortest Remaining Time最短剩余时间优先，可以看成SPN的抢占式版本

 SRT是抢占式调度，调度器总是选择预期剩余时间更短的进程；当新进程加入就绪队列时，如果它比当前运行进程剩余时间更短，就可能抢占当前进程

**SRT的优点**
比 SPN 更及时照度新到达的短进程
平均周转时间通常更短
响应短任务很快

**SRT的缺点**
长时间更容易饥饿
需要知道或估计剩余运行时间
抢占频繁，调度开销较大

### 最高响应比优先

HRRN Highest Response Ratio Next 最高响应比优先

`响应比 = (等待时间 + 预计处理器服务时间) / 预计处理器服务时间 `
也可以写成

$响应比 = 1 + 等待时间 / 预计服务时间$

调度时选择响应比最高的进程

HRRN 通常是 **非抢占式调度**
每次 CPU 空闲时，计算就绪队列中所有进程的响应比，选择最高者运行

**HRRN为什么能兼顾短作业和长作业**？
对于短作业：
```
服务时间少
等待时间 / 服务时间 较大
响应比容易高
```
所以短作业容易被优先调度

对于长作业
```
虽然服务时间大
但等待时间不断增加
响应比也会逐渐变大
```
所以长作业不会无限期等待

**既照顾短作业，又缓解长作业饥饿**

**HRRN的优点**
- 兼顾短作业和等待时间
- 缓解长作业饥饿
- 平均周转时间较好

**HRRN的缺点**
- 需要估计服务时间
- 每次调度都要计算响应比
- 通常不适合非常频繁的低级调度
## RR 时间片轮转调度算法
RR Round Robin 时间片轮转调度算法
RR根据各个进程进入就绪队列的时间先后，轮流占有CPU一个时间片；
当时间片发生中断时，当前正在运行的进程就置于就绪队列中，然后给予FCFS策略选择下一个就绪进程运行

核心思想：
```
每个进程轮流运行一个时间片
用完时间片还没结束，就回到就绪队列尾部
```

RR是 **抢占式调度**
因为时间片到期时，操作系统通过是时钟中断收回 CPU

**RR不需要预知CPU服务时间**
以RR为基本策略的调度算法不需要预知就绪进程所需CPU时长

**时间片长度的影响**
时间片长度要选择合适
过长则退化为先来先服务
过短则调度开销大

并且有单时间片、多时间片、动态时间片等设计

实际系统中，时间片通常会结合进程优先级、交互性、历史CPU使用情况动态调整

**RR的优点**
公平性好
响应时间较好
不需要预知服务时间
适合分时系统
实现相对简单

**RR的缺点**
平均周转时间不一定最优
时间片选择困难
上下文切换开销困难较大
对短任务不如 SPN/SRT 敏感

## Feedback：多级反馈队列调度
基本思想：
- 建立多个不同优先级的就绪进程队列
- 多个就绪队列之间按优先数调度
- 高优先级就绪进程分派的时间片短
- 单个就绪队列中进程优先数和时间片相同
- 单个队列内部通常按 FCFS 或 RR 调度

**多级反馈队列的结构**
可以表示为
```
RQ0 ： 最高优先级，时间片最短
RQ1 ： 中等优先级，时间片较长
RQ2 ： 低优先级，时间片更长
...
```
调度规则通常为：
```
优先选择高优先级队列
统一队列内部按FCFS 或 RR
新进程进入最高优先级队列
进程用完时间片还没完成，则降低优先级
如果进程主动阻塞，可能保持或提高优先级
```

**为什么叫反馈**？
因为系统会根据进程的运行行为调整它所在的队列
```
短进程很快完成，不会降很多级
长 CPU 密集型进程多次用完时间片，会逐渐降级
I/O 密集型或交互型进程经常需要主动阻塞，可能保持较高优先级
```
所以调度系统通过进程过去的行为“反馈”出它的类型
```
CPU 密集型 -> 降到低优先级
交互性 / I/O 密集型 -> 保持较高优先级
短任务 -> 很快完
```

### 分级原则
**外设访问**
频繁访问外设的进程通常CPU使用时间短，可以给予较高优先级，让它们快速发出 I/O 请求，提高CPU和I/O 并行度

**交互性**
交互式任务需要快速响应用户输入，应放在较高优先级队列

**时间紧迫程度**
实时任务或软实时任务应优先处理

**系统效率**
需要考虑整体吞吐量、资源利用率和上下文切换开销

**用户立场**
需保证用户体验和公平性，不能让某些用户任务长期得不到响应

**q**
$q=1$ 代表各级队列的时间片长度均为1
另有 $q=2^i$ 代表优先级越低，时间片越长

**多级反馈队列的优点**
- 不需要预知服务时间
- 兼顾短任务和长任务
- 交互响应好
- 能区分 CPU 密集型 和 I/O 密集型进程
- 适合现代通用操作系统

**多级反馈队列的缺点**
- 参数复杂
- 队列数量、时间片大小、升降级规则难以设计
- 低优先级进程可能饥饿
- 需要老化机制防止长期等待
- 实现复杂度高于 FCFS / RR

## 彩票调度算法
基本思想：
为进程开发针对系统各种资源的彩票
当调度程序需要决策时，随机选择一张彩票
持有该彩票的进程获得系统资源

合作进程之间可以进行彩票交换

假设系统中有三个进程：

```
P1 有 10 张彩票
P2 有 30 张彩票
P3 有 60 张彩票
```

总共 100 张彩票。

每次调度时随机抽一张。

那么长期来看：

```
P1 获得约 10% CPU
P2 获得约 30% CPU
P3 获得约 60% CPU
```

**彩票调度的优点**
实现比例共享比较自然
支持灵活资源分配
可以通过彩票数量表达优先级
随机性避免某些固定模式问题

**彩票调度的缺点**
短期结果有随机波动
实时性较弱
需要管理彩票分配和转移
