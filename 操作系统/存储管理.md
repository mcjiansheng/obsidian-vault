[[操作系统]]

如何把用户看到的地址空间，安全、高效、灵活地映射到政治贮存甚至辅存上

**逻辑地址** 用户编程时使用的地址空间
逻辑地址从0开始编号，用两种形式：
一维逻辑地址，即普通“地址”
二维逻辑地址，即“段号：段内地址”

**物理地址** 程序真正执行时访问主存所使用的地址空间
CPU执行指令时最终按照物理地址访问主存

## 存储管理的主要模式
四类：单连续存储管理、段式存储管理、页式存储管理、段页式存储管理

主要差别：
- 程序段逻辑地址空间是一维还是二维
- 主存空间是按连续分区分配，还是按固定大小页框分配

### 主存复用的两种基本方式
多道程序设计要求多个程序共享主存

**按照分区复用**
将主存划分成多个固定或可变大小的分区，一个程序或程序段占用一个分区

**按照页框复用**
把主存划分成多个固定大小的页框，一个程序或程序段占用多个页框

### 单连续存储管理
**单连续存储管理**是最简单的模式
定义为：一维逻辑地址空间的程序，占用一个主存固定分区或可变分区

特点：一个进程的代码、数据、堆栈等整体被装入一个连续的主存区域
```
逻辑地址空间：
0 -------------------------------- n

主存：
[ 操作系统 ][ 进程A连续区 ][ 进程B连续区 ][ 空闲区 ]
```

优点：
简单，地址转换容易
`物理地址 = 基址 + 逻辑地址`

缺点：
进程必须放在一整块连续空间中
如果内存中虽然总空闲空间足够，但被分散成很多小块，就可能放不下一个大进程，这就是外部碎片问题

### 页式存储管理
**页式存储管理**适用于一维逻辑地址空间

一维逻辑地址空间大成组，占用多个主存页框，也叫页架

页式管理的核心思想：把进程的逻辑地址空间切成固定大小的“页”，把物理主存切成同样大小的“页框”。进程的各个页不必连续存放，可以分散装入不同页框

```
进程逻辑空间：
页0 | 页1 | 页2 | 页3

物理主存：
页框5 放 页2
页框8 放 页0
页框12 放 页3
页框20 放 页1
```
页式管理的地址一般分成
`逻辑地址 = 页号 + 页内位移`

地址转换依靠页表完成
```
页号 → 查页表 → 页框号
物理地址 = 页框号 + 页内位移
```

页式管理的优点是可以离散分配，避免外部碎片；
缺点是引入页表，地址转换更复杂，并可能产生内部碎片

页式管理是现代虚拟内存的基础，因为“页”很适合做部分装入、缺页中断、页面置换

### 段式存储管理
**段式存储管理**面向的是二位逻辑地址空间
段式程序设计是把一个程序设计成多个段，例如代码段、数据段、堆栈段等

段式存储管理的定义：段式二位逻辑地址空间的程序，占用多个主存可变分区

段式地址形式：
`逻辑地址 = 段号 : 段内地址`

例如：
```
代码段：120
数据段：80
逻辑段：40
```

段式管理更符合程序员对程序结构的理解，程序本来就可以自然分为代码段、全局数据段、堆段、栈段、共享库段等

**优点**
是支持模块化、共享和保护。例如多个进程可以共享同一个只读代码段。

**缺点**
段长不固定，主存仍然按可变分区分配，所以可能产生外部碎片

特别注意：用户可以自己应用“段覆盖技术”扩充内存空间使用量，但这属于程序设计技术，不是操作系统存储管理的功能

### 段页式存储管理
**段页式存储管理**结合了段式和页式
定义：段式二位逻辑地址空间的程序，占用多个主存页框区

`逻辑地址 = 段号 + 段内页号 + 页内位移`

先按段组织程序，再把每个段分页。这样既保留段式管理的逻辑结构，又利用页式管理的离散分散能力

```
程序：
代码段 → 分成若干页
数据段 → 分成若干页
堆栈段 → 分成若干页

主存：
所有页分散装入多个页框
```
**优点**
综合性强：逻辑上便于共享和保护，物理上便于离散分配和虚拟存储

**缺点**
地址转换复杂，需要段表和页表同时参与

|模式|逻辑地址空间|主存分配方式|是否连续|典型优点|典型问题|
|---|---|---|---|---|---|
|单连续存储管理|一维|固定/可变分区|每个进程连续|简单、开销小|外部碎片，扩展差|
|页式存储管理|一维|页框|不要求连续|离散分配，适合虚存|页表开销，内部碎片|
|段式存储管理|二维，段号:段内地址|可变分区|每段连续|符合程序逻辑，便于共享保护|外部碎片|
|段页式存储管理|二维，段内再分页|页框|不要求连续|兼顾段式逻辑与页式物理优势|地址转换复杂|
## 存储管理的功能

**地址转换、存储保护、主存分配、主存共享、虚拟存储器**

### 地址转换：从逻辑地址到物理地址

地址转换也叫**重定位**
定义：把逻辑地址转换为绝对地址

**静态重定位**
静态重定位是再在程序装入内存时完成地址转换，由装入程序执行，早期小型操作系统通常使用

例如，程序原来认为自己从地址 0 开始运行：

```
逻辑地址：0, 100, 200, 300
```

如果装入主存的起始位置是 10000，那么装入时就把地址全部改成：

```
物理地址：10000, 10100, 10200, 10300
```

这样程序运行时不再需要频繁转换，执行简单。但缺点是程序一旦装入，位置就不方便改变，也不适合虚拟存储、动态调入调出。

**动态重定位**
动态重定位是在CPU执行程序时进行地址转换
从效率出发，动态重定位依赖硬件地址转换机构

CPU每次访问逻辑地址`X`时，硬件自动计算
`物理地址 = 基址寄存器 + 逻辑地址`

**优点**
灵活，进程可以被移动，页面可以被调入不同页框，程序也可以只装入一部分运行

现代操作系统基本依赖动态重定位

### 主存空间的分配和去配
进程装入主存时，存储管理软件要进行主存分配，并设置表格记录主存空间分配情况；
当进程撤离或主动归还主存资源时，系统要收回其占用的全部或部分空间，并调整主存分配表

这里的”分配“不是随便找一块内存，而是要考虑：
进程需要多少空间；当前哪些区域空闲；是否需要连续空间；如果空间不足，是否要换出某些进程或页面；分配后如何记录；释放时如何合并空闲区

不同管理模式下，分配单位不同：
```
连续分区：分配单位是分区
页式管理：分配单位是页框
段式管理：分配单位是段对应的可变分区
段页式管理：分配单位是页框，但逻辑上按段组织
```

### 主存共享
主存共享有两层含义：
- 多个进程共享主存储器资源，即多到程序设计让若干程序同时进入主存，各自占用一定存储空间
- 多个进程共享主存中的某些程序，例如协作进程共享共同的程序块或数据块

### 存储保护
存储保护是为了避免主存中的多个进程相互干扰

私有主存区中的信息可读可写；
公共区中的共享信息根据授权访问；
非本进程信息不可续写。
并且这一功能需要软硬件协同完成，CPU检查是否允许访问，不允许则产生地址保护异常，由OS处理

所以现代 OS 必须依靠地址空间隔离和访问权限控制。例如页表项中通常有读、写、执行、用户态/内核态、有效位等保护信息。

## 虚拟存储器概述

解决什么问题？
**程序可以比实际主存更大**
因为不必把整个程序一次性装入主存，只需要把当前要执行的部分放入主存

**每个进程可以拥有独立地址空间**
进程A和进程B都可以认为自己从地址0开始，但它们映射到不同的物理页框

**提高主存利用率和多道程序度**
虚拟存储器下，程序看到的是虚拟地址空间。这个空间通常很大、连续、从0开始
例如一个64位系统中，进程理论上可以看到非常大的地址空间

物理地址空间则是真实主存大小决定的，例如机器只有16GB内存，那么真实可直接访问的物理主存空间就是有限的

虚拟存储的关键是建立映射：
```
虚拟页 0 → 物理页框 8
虚拟页 1 → 不在主存，在磁盘
虚拟页 2 → 物理页框 3
虚拟页 3 → 物理页框 20
```
如果访问的虚拟页不在主存，就会发送**缺页中断**，操作系统把需要的页从辅存调入主存

### 虚拟存储的基本原理

虚拟存储器依赖两个思想：
**局部性原理**和**按需调入**

**局部性原理**
程序在一段时间内倾向于反复访问某一小部分代码和数据，比如循环、函数调用、数组局部访问等都会导致访问集中在某些区域

**按需调入**
不需要一开始把整个程序装入内存，而是在访问某页或某段时，如果它不在主存，再由操作系统调入

### 虚拟存储器的优势和代价

**优势**
- 扩大程序可用地址空间
- 提高多道程序度
- 支持进程隔离
- 支持共享库和共享内存
- 支持按需装入，减少不必要I/O

**代价**：
- 地址开销转换增加
- 页表/段表需要占用内存
- 缺页会造成巨大性能损失
- 需要页面置换算法
- 需要硬件和操作系统紧密配合

## 存储管理的硬件支持
地址转换与存储保护需要硬件支撑

### 为什么需要硬件支撑
因为程序执行时，每条访存指令都涉及地址转换。如果每访问一次内存都让操作系统软件介入，性能会无法接受

### 基址寄存器与限长寄存器
处理器发出逻辑地址后，会与限长寄存器比较；
如果越界，则产生中断，如果合法，则与基址寄存器相加得到物理地址

```
若 逻辑地址 < 限长：
    物理地址 = 基址 + 逻辑地址
否则：
    越界中断
```

那么访问越界，硬件触发地址保护异常，交给操作系统处理。

这种机制适合连续分区存储管理。它同时完成了两件事：

```
地址转换：逻辑地址 + 基址 → 物理地址存储保护：逻辑地址不能超过限长
```

### 页表硬件支持

在页式存储管理中，仅靠基址和限长已经不够。因为一个进程的页面可能分散在不同页框中，所以需要页表。

```
逻辑地址 = 页号 + 页内位移
页号 → 查页表 → 页框号
物理地址 = 页框号 + 页内位移
```
页表项通常还包含保护信息，例如：

```
有效位：该页是否在主存
访问位：最近是否被访问
修改位：是否被写过
权限位：可读/可写/可执行
页框号：映射到哪个物理页框
```

这些信息既用于地址转换，也用于保护和页面置换。

### 快表 TLB

页表查询本身要访问内存。如果每次访问数据之前都要先访问页表，那么一次内存访问可能变成两次甚至更多次内存访问，性能会明显下降。
因此现代 CPU 通常使用 TLB，也叫快表，保存最近使用的页表项。

```
先查 TLB
    命中：直接得到页框号
    未命中：查页表，再把结果放入 TLB
```
TLB 的理论依据仍然是局部性原理：程序最近访问过的页，很可能马上还会访问。

### 段表硬件支持
在段式管理中，逻辑地址是：
```
段号 : 段内地址
```
地址转换依靠段表。段表项通常包含：
```
段基址段长访问权限共享标志存在位修改位
```
转换过程是：
```
根据段号查段表检查段内地址是否小于段长检查访问权限物理地址 = 段基址 + 段内地址
```
如果段内地址超过段长，则产生越界中断。如果访问权限不符，例如试图写只读代码段，则产生保护中断。

### 段页式硬件支持

段页式结合段表和页表。转换过程更复杂：

```
逻辑地址 = 段号 + 段内页号 + 页内位移
第一步：用段号查段表
第二步：得到该段对应页表的位置
第三步：用段内页号查页表
第四步：得到页框号
第五步：页框号 + 页内位移 = 物理地址
```

它的优点是：逻辑上按段保护和共享，物理上按页框离散分配。缺点是转换链路长，所以更依赖 TLB 等硬件加速。
