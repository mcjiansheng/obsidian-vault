[[并发程序设计]]

死锁是并发系统中非常重要的问题。前面学临界资源、互斥、PV 操作时，我们其实已经接触过死锁的影子。

**死锁定义**
一组进程因争夺资源陷入永远等待的状态。例子是 P0 和 P1 都需要 S、Q 两类资源，P0 先申请 S 再申请 Q，P1 先申请 Q 再申请 S，若每类资源只有一个，就可能互相等待。

如果一组进程中的每个进程都在等待某个事件，而这个事件只能由同组中另一个正在等待的进程产生，那么这组进程就陷入死锁。

### 死锁与饥饿的区别
**死锁**是多个进程形成环形等待，谁都无法继续。  
**饥饿**是某个进程长期得不到资源，但系统整体仍在运行。

死锁是多个进程都获得了部分资源，还想得到其他进程占有的资源，最终所有进程陷入死锁；饥饿是一个进程由于其他进程总是优先于它而被无限期拖延。

死锁一定是严重错误；饥饿则通常是调度策略不公平导致的。

### 死锁产生的资源背景
要理解死锁，必须先理解资源。

操作系统中的资源可以分为：

**可重用资源**：使用后不会消失，例如 CPU、内存、打印机、文件锁、互斥锁。  
**消耗性资源**：使用后会消失，例如消息、信号、中断事件。  
**可抢占资源**：可以从进程手中夺走，例如 CPU、部分内存页。  
**不可抢占资源**：不能强行夺走，否则会出错，例如打印机、互斥锁、正在写的文件。

死锁主要发生在多个进程竞争**不可抢占的互斥资源**时。

资源使用的一般模式是：

```
请求资源 → 使用资源 → 释放资源
```

如果请求资源时资源不可用，进程就进入等待状态。多个等待状态互相形成环，就可能死锁。

## 死锁产生的四个必要条件
系统形成死锁的四个必要条件：互斥条件、占有和等待条件、不剥夺条件、循环等待条件。

### 互斥条件 mutual exclusion
互斥条件指系统中存在临界资源，进程必须互斥地使用这些资源。

如果所有资源都可以共享，就不会因为互斥资源竞争产生死锁。但现实中很多资源天然不可共享，所以互斥条件通常难以完全破坏。

### 占有和等待条件 hold and wait
占有和等待指进程已经占有了一些资源，同时又去请求新的资源；当新资源得不到满足时，它不释放已经占有的资源。

### 不剥夺条件 no preemption
不剥夺条件指资源一旦被某进程占有，就不能被其他进程强行夺走，只能由资源占有者主动释放。

如果资源可剥夺，系统可以强行回收资源，从而打破等待。但对于很多 I/O 资源、锁资源、事务资源来说，剥夺代价很大，甚至不可行。

### 循环等待条件 circular wait
循环等待指存在一个进程等待链：
```
P0 等待 P1 占有的资源；
P1 等待 P2 占有的资源；
...
Pn 等待 P0 占有的资源。
```
形成环后，每个进程都在等下一个进程释放资源，而下一个进程也在等待别人释放资源，所以整个环上的进程都无法继续。

### 四个条件之间的关系
这四个条件是死锁的**必要条件**。也就是说，发生死锁时，这四个条件一定同时成立。

但要注意：
在每类资源只有一个实例的情况下，资源分配图中出现环通常意味着死锁；  
在每类资源有多个实例的情况下，出现环不一定死锁，但没有环一定没有死锁。

## 死锁的防止
既然死锁必须同时满足四个必要条件，那么只要破坏其中任意一个条件，就可以保证系统不发生死锁。

### 破坏互斥条件
方法是让资源可以同时访问，而不是互斥使用。

例如：
打印机本身是独占设备，但可以用 SPOOLing 技术把打印任务先写入磁盘缓冲队列，再由打印进程逐个输出。对用户进程来说，好像大家都能“同时使用打印机”。

但这个方法不能解决所有资源。例如互斥锁、写文件、数据库事务中的排他锁，本质上仍需要互斥。

所以，破坏互斥条件适用范围有限。

### 破坏占有和等待条件
方法是：不允许进程“拿着一部分资源再等另一部分资源”。

常见策略有两种：

第一，静态分配。  
进程运行前一次性申请它需要的全部资源，全部满足才运行，否则一个都不给。

第二，申请新资源前必须释放已有资源。  
也就是说，进程只有在不占有任何资源时，才能申请其他资源。

这种策略可以防止死锁，但缺点明显：

资源利用率低；  
进程可能长期等不到全部资源；  
很多程序无法预先知道全部资源需求；  
降低系统并发度。

### 破坏不剥夺条件
方法是允许系统强行剥夺资源。

例如：
如果进程 P 已经占有资源 A，但申请资源 B 失败，那么系统可以要求 P 释放 A，然后 P 等待以后重新申请。

这种方法适合可保存和恢复状态的资源，例如 CPU、内存页。  
但不适合很多不可恢复资源，例如打印机、正在写入的文件、持锁修改到一半的数据结构。

如果强行剥夺锁资源，可能导致共享数据处于不一致状态。

### 破坏循环等待条件
这是比较常用的方法。核心思想是给所有资源编号，规定进程必须按资源编号递增顺序申请资源。

层次分配策略：把系统中所有资源排成顺序，例如 `r1, r2, ..., rm`，规定进程不得在占用资源 `ri` 后再申请编号更小的 `rj`；按这种策略分配资源时，系统不会发生死锁

## 死锁的避免
死锁防止比较保守，它通过强制规则破坏死锁条件。死锁避免则更灵活：系统并不是绝对禁止某些申请方式，而是在每次资源分配前判断“如果这次分配出去，系统是否仍然安全”。

### 安全状态与不安全状态
课件给出了系统安全性的定义：在某一时刻系统是安全的，当且仅当存在一个进程序列，使得每个进程在该序列中都能获得完成所需资源，执行结束后释放资源，从而让后续进程继续完成。

**安全状态**：存在某种进程完成顺序，使所有进程最终都能完成。  
**不安全状态**：找不到这样的保证顺序，系统可能走向死锁。  
**死锁状态**：已经有进程集合互相等待，无法继续。

注意：不安全状态不等于死锁。  
它只是说明系统已经没有“保证安全完成”的调度方案。

### 银行家算法的基本思想
银行家算法是经典死锁避免算法。它把操作系统看成银行家，把资源看成资金，把进程看成客户。
银行家不会把钱随便借出去，而是要确保即使借出这笔钱，银行仍然有能力让所有客户最终完成并归还资金。

进程申请资源时，系统先假装分配；  
然后运行安全性测试；  
如果系统仍然安全，则真正分配；  
如果不安全，则拒绝分配，让进程等待。

### 银行家算法的数据结构
设系统有 n 个进程，m 类资源。

常用数据结构：
```
Available[m]
```
表示每类资源当前可用数量。

```
Max[n][m] 或 Claim[n][m]
```
表示每个进程最多可能需要多少资源。

```
Allocation[n][m]
```
表示当前已经分配给每个进程的资源数量。

```
Need[n][m]
```
表示每个进程还需要多少资源。

公式：`Need = Max - Allocation`

系统需要检查三件事：

第一，`Request_i <= Need_i`。  
否则说明进程申请量超过了它声明的最大需求，是错误请求。

第二，`Request_i <= Available`。  
否则说明当前资源不够，进程必须等待。

第三，假装分配后系统是否安全。  
如果安全，则正式分配；否则撤销试探分配，让进程等待。

### 安全性测试算法
银行家算法中的安全性测试程序：先令 `currentavail` 等于当前可用资源，把所有进程放入 `rest` 集合，然后不断寻找一个进程 Pk，使得它剩余需求 `claim[k,*] - allocation[k,*]` 小于等于当前可用资源；如果找到，就假设它完成并释放资源；最后如果所有进程都能移出 `rest`，则系统安全。

```c
Work = Available;
Finish[i] = false for all i;

while (exists i such that Finish[i] == false && Need[i] <= Work) {
    Work = Work + Allocation[i];
    Finish[i] = true;
}

if (Finish[i] == true for all i)
    system is safe;
else
    system is unsafe;
```
`Work` 表示当前可用于推进进程完成的资源；  
如果某个进程的 `Need` 小于等于 `Work`，说明它可以完成；  
它完成后会释放已经占有的资源，所以 `Work` 增加；  
如果最后所有进程都能完成，就存在安全序列。

## 死锁的检测和解除
死锁防止：从规则上禁止死锁条件成立。  
死锁避免：每次分配前判断是否安全。  
死锁检测和解除：平时允许资源自由申请，系统定期检查是否已经死锁；如果发现死锁，再采取措施解除。

死锁检测和解除不对资源分配加任何限制，也不采取死锁避免措施，而是定时运行死锁检测程序判断系统是否已经出现死锁；如果检测到死锁，再采取措施解除。
### 单实例资源的检测：等待图
如果每类资源只有一个实例，可以用等待图检测。

节点是进程；  
边 `Pi → Pj` 表示 Pi 正在等待 Pj 占有的资源。

如果等待图中存在环，则发生死锁。

### 多实例资源的检测
如果每类资源有多个实例，仅仅看有没有环不够。因为即使存在等待环，某类资源还有额外实例释放后，也可能打破等待。

多实例资源通常使用类似安全性检测的算法。
```c
Available[m]     // 当前可用资源
Allocation[n][m] // 当前已分配资源
Request[n][m]    // 当前各进程正在请求的资源

Work = Available;

for each process i:
    if Allocation[i] == 0:
        Finish[i] = true;
    else:
        Finish[i] = false;

while (exists i such that Finish[i] == false && Request[i] <= Work) {
    Work = Work + Allocation[i];
    Finish[i] = true;
}

if (exists i such that Finish[i] == false)
    those processes are deadlocked;
else
    no deadlock;
```
死锁检测算法与死锁避免算法类似，但二者关注点不同：死锁避免是在资源分配前判断是否会进入不安全状态；死锁检测则是在系统运行中检查是否已经有进程无法继续推进。

### 死锁解除方法
检测到死锁之后，必须解除死锁。

#### 方法一：撤销进程
可以直接终止死锁进程。

做法有两种：
直接撤销所有死锁进程；  
一次撤销一个进程，每撤销一个就重新检测是否解除死锁。

选择撤销哪个进程时，可以考虑：
进程优先级；  
已经运行时间；  
还需运行时间；  
占有资源数量；  
回滚代价；  
是否交互进程；  
是否关键系统进程。

这种方法简单粗暴，但可能造成数据丢失或计算浪费。

#### 方法二：剥夺资源
从某些进程手中强行收回资源，分配给其他进程，使等待环被打破。

关键问题是：
选择剥夺哪个资源；  
选择牺牲哪个进程；  
如何保存和恢复进程状态；  
如何避免同一个进程反复被剥夺导致饥饿。

资源剥夺适合可回滚资源，不适合不可恢复资源。

#### 方法三：进程回滚
如果系统支持检查点 checkpoint，可以把某个进程回滚到之前的安全状态，释放它后来占有的资源。

例如数据库事务系统中，如果事务死锁，可以回滚某个事务，让它释放锁，其他事务继续执行。

## 混合策略
实际操作系统往往不会只使用一种死锁处理策略，而是混合使用。

对于可共享资源，尽量消除互斥；  
对于必须互斥但可排序的资源，采用资源有序分配，防止循环等待；  
对于可预知最大需求的资源，使用银行家算法避免死锁；  
对于复杂、低概率或代价较高的资源，采用检测和解除；  
对于极少发生且影响可控的场景，系统可能选择忽略，让应用层处理。
