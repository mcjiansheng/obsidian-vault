# 并发程序/并发进程
**提高系统资源利用率和系统吞吐率**

**把一个任务拆成若干个能重叠推进的进程，使 CPU、I/O 设备、内存等资源尽可能同时工作。**

### 顺序程序设计：并发的对照物
并发程序破坏了很多“默认安全假设”。

顺序程序设计中，程序是实现算法的操作或指令序列，每个程序在处理器上严格按序执行，这叫程序执行的**内部顺序性**；而把一个具体问题设计成一个程序，或者设计成若干严格顺序执行的程序序列，这叫程序执行的**外部顺序性**。

顺序程序设计有四个重要特性：
- **顺序性**：指令严格按程序规定的顺序执行。
- **封闭性**：程序运行时好像独占系统资源，外界不会随便干扰它。
- **确定性**：同一输入下，结果与执行速度、执行时间段无关。
- **可再现性**：同一输入下，执行轨迹可重复。

### 进程的并发执行
多道程序设计允许多个程序同时进入内存，它们竞争处理器，获得运行机会。操作系统允许在一个时间段内存在多个正在运行的进程，这就是**多个进程并发执行**。
对于按照顺序程序设计思想编写、且互不干扰的进程，操作系统应当保证它们并发执行时仍然像独占计算机一样不受影响，这类进程称为**无关的并发进程**。

**并发 Concurrency**：多个进程在时间上重叠推进。单处理器系统中，宏观上看多个进程都在运行，微观上任一时刻只有一个进程占用 CPU。
**并行 Parallelism**：多个进程在物理上真正同时执行。通常需要多核 CPU 或多处理器支持。

### 并发程序设计的概念
把一个程序分成若干个可同时执行的程序模块，每个程序模块与其处理的数据共同组成一个进程

```c
// 进程 i：输入
while (true) {
    input;
    send;
}

// 进程 p：处理
while (true) {
    receive;
    process;
    send;
}

// 进程 o：输出
while (true) {
    receive;
    output;
}
```
这里的 `send` 和 `receive` 表示进程之间需要通信或同步。输入进程读完数据后，处理进程才能处理；处理进程处理完后，输出进程才能输出。它们虽然并发执行，但不是完全随意执行，而是存在**制约关系**。

并发程序设计的三个特性是：
- **并行性 / 并发性**：多个进程可重叠执行，提高效率。
- **共享性**：多个进程共享软件资源或硬件资源。
- **交往性**：多个进程之间存在制约，需要协调。
**并发不是简单地“同时跑”，而是“同时推进但要协调”。**

## 并发进程的制约关系
并发进程分为两类：**无关的并发进程**和**交往的并发进程**。

**无关的并发进程**
无关的并发进程分别在不同变量集合上运行，一个进程的执行不会影响另一个进程的结果。

**交往的并发进程**
交往的并发进程共享某些变量或资源，一个进程的执行可能影响其他进程的结果。

### Bernstein 条件：判断两个程序能否安全并发
设 $R(p_i)$ 表示程序 $p_i$ 执行期间**引用/读取**的变量集合
$W(p_i)$ 表示程序 $p_i$ 执行期间 **改变/写入**的变量集合
两个程序 $p_1$​、$p_2$​ 若满足：
$(R(p1​)∩W(p2​))∪(R(p2​)∩W(p1​))∪(W(p1​)∩W(p2​))=∅$
则它们并发执行与时间无关，可以安全并发。

这个公式本质上检查三类冲突：
**读-写冲突**
$R(p1​)∩W(p2​)$
如果 `p1` 要读某个变量，而 `p2` 会写这个变量，那么谁先谁后会影响 `p1` 读到的值。

**写-读冲突**
$R(p2​)∩W(p1​)$
同理，如果 `p2` 要读某个变量，而 `p1` 会写这个变量，也会产生顺序依赖。

**写-写冲突**
$W(p1​)∩W(p2​)$
如果两个程序都写同一个变量，最后结果取决于谁后写。

### 与时间有关的错误
对于交往的并发进程，执行速度和推进顺序不可预知。如果程序设计不当，就会出现**与时间有关的错误**。

两类表现：**结果错误**和**永远等待**

#### 结果错误：机票售票问题
假设某航班余票数 `Aj = 1`，两个售票进程同时售票：
```c
// T1
int X1 = Aj;
if (X1 >= 1) {
    X1--;
    Aj = X1;
    输出一张票;
}

// T2
int X2 = Aj;
if (X2 >= 1) {
    X2--;
    Aj = X2;
    输出一张票;
}
```
```
T1 读取 Aj = 1
T2 读取 Aj = 1
T1 判断有票，卖出一张，Aj = 0
T2 判断有票，卖出一张，Aj = 0
```
结果：两个人都拿到票，但系统只减少了一张票。
这个错误不是算法本身在顺序执行时有问题，而是**并发执行时共享变量 Aj 没有被互斥保护**。

#### 永远等待：主存管理问题
核心变量是：
```c
int X = memory;  // 当前可用主存容量
```
一个进程执行 `borrow(B)` 申请内存，另一个进程执行 `return(B)` 归还内存。如果申请进程判断 `B > X` 后准备进入等待队列，但还没真正进入等待队列时，另一个进程已经归还内存并执行了“唤醒等待进程”的操作，那么这次唤醒会丢失。
后果是：申请进程随后进入等待队列，但以后可能没有进程再归还内存，于是它永远等待。
这类问题本质上叫 **lost wakeup，丢失唤醒**。

### 竞争关系与协作关系
并发进程之间有两种基本关系：**竞争关系**和**协作关系**。

#### 竞争关系
多个进程争用同一资源，例如打印机、共享变量、缓冲区、文件表、内存分配表等。一个进程获得资源后，其他进程必须等待。
竞争关系带来两个典型问题：
**死锁 Deadlock**：一组进程各自占有部分资源，又等待其他进程占有的资源，最终全部等待。
**饥饿 Starvation**：某个进程因为其他进程总是优先于它，导致长期得不到资源。

#### 协作关系
多个进程为了完成同一任务而分工合作。例如：
`输入进程 -> 处理进程 -> 输出进程`
处理进程必须等输入进程提供数据，输出进程必须等处理进程产生结果。这种关系不是争抢资源，而是存在先后配合。

### 互斥与同步
#### 互斥 Mutual Exclusion
互斥是解决竞争关系的手段。
若干进程要使用同一共享资源时，任何时刻最多允许一个进程使用该资源，其他进程必须等待，直到占有资源的进程释放资源。

`互斥 = 不能同时用`

#### 同步 Synchronization
同步是解决协作关系的手段。
两个以上进程基于某个条件协调活动。一个进程的执行依赖另一个进程的消息或信号；如果消息或信号未到达，就需要等待，直到被唤醒。

### 互斥是特殊的同步
进程互斥关系是一种特殊的进程同步关系。因为互斥本质上也是在协调顺序：当一个进程正在使用临界资源时，其他进程必须等它退出后才能进入

# 临界区管理
### 临界资源与临界区
**临界资源**：一次只能被一个进程使用的共享资源。  
**临界区**：并发进程中访问临界资源的程序段。

例如：
```c
int tickets = 1;  // 共享变量，代表余票数
```
售票代码：
```c
if (tickets > 0) {
    tickets--;
    print_ticket();
}
```
这里 `tickets` 是临界资源，访问并修改 `tickets` 的代码段就是临界区。

临界资源是资源。
临界区是代码。

### 相关临界区与不相关临界区
如果两个临界区访问的是同一个临界资源，就是**相关临界区**，必须互斥进入。
如果两个临界区访问不同临界资源，则不相关，可以并发进入。
### 临界区管理的三个要求
- **一次至多允许一个进程停留在相关临界区内。**
- **一个进程不能无限地停留在临界区内。**
- **一个进程不能无限地等待进入临界区。**

#### 互斥进入
同一临界资源对应的相关临界区中，任意时刻最多只能有一个进程进入。
#### 有限停留
进入临界区的进程不能永远不出来。否则其他进程永远等不到资源。
#### 有限等待
等待进入临界区的进程不能无限期等待，否则就会出现饥饿。

### 临界区的一般结构
```c
while (true) {
    // 剩余区 remainder section
    ...

    // 进入区 entry section
    entercritical(R);

    // 临界区 critical section
    使用临界资源 R;

    // 退出区 exit section
    exitcritical(R);

    // 剩余区 remainder section
    ...
}
```

- **进入区**：判断能否进入临界区，并完成加锁。
- **临界区**：真正访问共享资源。
- **退出区**：释放资源，唤醒或允许其他进程进入。
- **剩余区**：与临界资源无关的代码。

### 临界区嵌套与死锁风险
```c
// 进程 P1
region x do {
    ...
    region y do {
        ...
    }
}

// 进程 P2
region y do {
    ...
    region x do {
        ...
    }
}
```
```
P1 进入 region x，占有 x
P2 进入 region y，占有 y
P1 想进入 region y，但 y 被 P2 占有
P2 想进入 region x，但 x 被 P1 占有
```
于是 P1 等 P2，P2 等 P1，形成死锁。

解决这类问题的常见原则是：
```
所有进程按相同顺序申请多个临界资源。
```
例如统一规定：必须先申请 `x`，再申请 `y`。这样就不会出现一个进程拿着 `x` 等 `y`，另一个进程拿着 `y` 等 `x` 的环路等待。


## 临界区管理实现的尝试
### 错误尝试一：先检查别人，再声明自己
```c
bool inside1 = false;
bool inside2 = false;

// P1
while (inside2);   // 如果 P2 在临界区，就等待
inside1 = true;
临界区;
inside1 = false;

// P2
while (inside1);   // 如果 P1 在临界区，就等待
inside2 = true;
临界区;
inside2 = false;
```
问题在于：**检查和声明不是原子操作**。
“先看有没有人，再说我要进去”是不安全的。
因为你看的那一瞬间和你声明的那一瞬间之间，别人可能也通过了检查。

### 错误尝试二：先声明自己，再检查别人
```c
bool inside1 = false;
bool inside2 = false;

// P1
inside1 = true;
while (inside2);
临界区;
inside1 = false;

// P2
inside2 = true;
while (inside1);
临界区;
inside2 = false;
```
这次看似更谨慎：先声明“我要进”，再检查别人。

但可能执行：
```
P1 设置 inside1 = true
P2 设置 inside2 = true
P1 检查 inside2 == true，于是等待
P2 检查 inside1 == true，于是等待
```
结果两个进程都在等待对方退出，但谁也没有进入临界区，谁也不会把自己的 `inside` 改回 `false`。
这就是一种软件层面的死锁。

### Peterson 算法
```c
bool inside[2] = {false, false};
int turn;

// P0
inside[0] = true;
turn = 1;
while (inside[1] && turn == 1);
临界区;
inside[0] = false;

// P1
inside[1] = true;
turn = 0;
while (inside[0] && turn == 0);
临界区;
inside[1] = false;
```
#### inside 数组表示“我想进入”

```c
inside[i] = true;
```

表示进程 `Pi` 有进入临界区的意愿。

#### turn 表示“如果冲突，让谁先”

```c
turn = 1;
```

在 `P0` 中表示：如果 `P0` 和 `P1` 都想进入，那么 `P0` 礼让 `P1`。

```c
turn = 0;
```

在 `P1` 中表示：如果二者都想进入，那么 `P1` 礼让 `P0`。

核心逻辑：
如果对方想进，并且当前轮到对方，那我等。
否则我进。

#### Peterson 算法的局限
- 主要用于两个进程的理论模型。
- 使用忙式等待，等待进程会一直占用 CPU 检查条件。
- 依赖读写操作的顺序可见性，在现代 CPU 和编译器优化环境下，需要内存屏障或原子语义支持。
- 不适合直接作为通用操作系统同步机制。
互斥的困难不在“写一个 if 判断”，而在于并发执行时检查、声明、等待、退出必须被严密协调。

## 临界区管理实现的硬件方式
它们共同目标是：  
**借助硬件保证某些关键操作的原子性。**

### 关中断
最简单的互斥方法是：

```
关中断;
临界区;
开中断;
```

为什么有效？

在单处理器系统中，进程切换通常依赖时钟中断。如果关中断后，当前进程不会因为时钟中断被切走，就可以连续执行完临界区。

#### 适用场合
关中断适合操作系统内核中非常短小的临界区，例如修改某些内核数据结构时。
操作系统原语可以采用这种思路，但临界区指令长度必须短小精悍，才能保证系统效率。

#### 缺点
1. **不适合用户程序**：用户程序如果能随意关中断，会破坏整个系统。
2. **影响系统响应性**：关中断期间，外部事件无法及时响应。
3. **不适合多处理器互斥**：在多核系统中，关掉当前 CPU 的中断，并不能阻止其他 CPU 上的进程访问共享数据。
4. **临界区不能太长**：否则系统中断延迟过大。

### 测试并建立指令 Test-and-Set，TS
```c
bool TS(bool &x) {
    if (x) {
        x = false;
        return true;
    } else {
        return false;
    }
}
```
**测试 x 的值和修改 x 的值由硬件一次完成，中间不能被打断。**
```c
bool s = true;

process Pi() {
    while (!TS(s));   // 上锁
    临界区;
    s = true;         // 开锁
}
```
- `s = true` 表示锁空闲。
- 某个进程执行 `TS(s)` 时，如果 `s == true`，则把 `s` 改成 `false`，并返回 `true`，表示加锁成功。
- 其他进程再执行 `TS(s)` 时，发现 `s == false`，返回 `false`，只能继续循环等待。
- 退出临界区时，执行 `s = true`，释放锁。

#### TS 的优点
1. 硬件保证原子性。
2. 可用于多处理器系统。
3. 实现简单，常用于自旋锁 spinlock 的基础。

#### TS 的缺点
TS 的等待方式是：

```c
while (!TS(s));
```

这叫**忙式等待 busy waiting**。等待进程并没有阻塞，而是一直占用 CPU 反复测试锁。

所以如果临界区很短，忙等可以接受；如果临界区很长，就会严重浪费 CPU。

#### 对换指令 SWAP
```c
void SWAP(bool &a, bool &b) {
    bool temp = a;
    a = b;
    b = temp;
}
```
硬件保证 `SWAP(a, b)` 是原子操作。

```c
bool lock = false;

process Pi() {
    bool keyi = true;

    do {
        SWAP(keyi, lock);
    } while (keyi);     // 上锁

    临界区;

    SWAP(keyi, lock);  // 开锁
}
```
初始：

```
lock = false
```

表示锁空闲。

某个进程进入时：

```c
keyi = true
SWAP(keyi, lock)
```

交换后：

```c
keyi = false
lock = true
```

因为 `keyi == false`，跳出循环，进入临界区。

其他进程也尝试交换时，由于 `lock == true`，交换后 `keyi` 仍为 `true`，于是继续循环等待。

退出临界区时，再交换一次，把 `lock` 恢复成 `false`。

#### TS 和 SWAP 的共同问题：忙式等待
TS 和 SWAP 都是忙式等待，效率低。
1. **浪费 CPU 时间**：不能进入临界区的进程仍然占用 CPU 循环测试。
2. **把同步责任推给进程**：如果每个进程都自己写加锁逻辑，程序容易出错，系统可靠性下降。

# 信号量与 PV 操作
**忙式等待**：进程进不了临界区时，不进入阻塞态，而是一直执行类似
```c
while (不能进入临界区) {
    反复测试;
}
```
这会浪费 CPU 时间，而且把“如何判断能否进入临界区”的责任交给各个竞争进程，会增加编程负担、降低系统可靠性。
通用解决方案是：**信号量与 PV 操作**。

不能继续执行的进程，不再忙等，而是进入等待队列；
当条件满足时，由 V 操作唤醒它。

### 信号量的概念
信号量 semaphore 是 Dijkstra 在 1965 年提出的同步工具。
P 来自荷兰语 Proberen，含义接近“检测、尝试”；V 来自 Verhogen，含义接近“增加”。
信号量可以使进程在某一特殊点停止，直到接收到对应的信号；复杂的进程协作可以通过合适的信号量结构表达。

**记录型信号量**
```c
typedef struct semaphore {
    int value;       // 信号量的整数值
    queue waitq;     // 等待该信号量的进程队列
} semaphore;
```

这里的 `value` 不只是一个普通整数，它有明确含义：
value > 0：表示当前还可用的资源数。
value = 0：表示资源刚好用完，但暂时没有进程等待。
value < 0：表示已经有进程等待，|value| 等于等待进程数。

若信号量为正，其值等于还可以执行 P 操作而不阻塞的次数，也等于可用资源数；若为负，其绝对值等于该信号量队列中等待的进程个数；通常 P 表示请求资源，V 表示释放资源，在一定条件下 P 表示阻塞，V 表示唤醒。

### P 操作与 V 操作的语义
P 操作将信号量减 1，如果结果小于 0，则调用 P 的进程进入该信号量的等待队列；V 操作将信号量加 1，如果结果不大于 0，则唤醒一个等待该信号量的进程，使其进入就绪态。

```c
void P(semaphore &s) {
    s.value--;
    if (s.value < 0) {
        block(current_process, s.waitq);
    }
}
void V(semaphore &s) {
    s.value++;
    if (s.value <= 0) {
        wakeup(one_process_from(s.waitq));
    }
}
```
```
P 后判断 s.value < 0。
V 后判断 s.value <= 0。
```
**P 和 V 是原语操作**:原语是在 CPU 内核态、关中断环境下执行的一段指令序列，具有原子性；对信号量只能通过 P/V 原语访问，不能直接读写 `value` 或操作等待队列。

### 信号量与进程状态转换
信号量把进程状态和资源申请联系起来。

当进程执行 `P(s)` 时，如果资源可用，它继续运行；如果资源不可用，它进入等待态 / 阻塞态。以后某个进程执行 `V(s)` 释放资源时，阻塞队列中的一个进程被唤醒，进入就绪态，等待 CPU 调度。

## 信号量的三种基本用法
### 用于互斥：二元信号量 / 互斥信号量
如果一个临界资源一次只能被一个进程使用，可以设：
```c
semaphore mutex = 1;
```

每个进程访问临界区时:
```c
P(mutex);
/* critical region 临界区 */
V(mutex);
```
达纯粹互斥关系时，信号量初值为 1，且同一个信号量的 P 操作和 V 操作处于同一侧进程中；但这种结构不适用于同步关系。

### 用于同步：表达先后关系
同步不是“只能一个进入”，而是“某件事必须等另一件事发生”。
`P1 执行完 S1 后，P2 才能执行 S2。`

```c
semaphore s = 0;
process P1() {
    S1;
    V(s);     // 告诉 P2：S1 已完成
}

process P2() {
    P(s);     // 等待 S1 完成
    S2;
}
```
为什么同步信号量初值通常是 0？
因为一开始事件还没有发生。若 P2 先执行 `P(s)`，则 `s` 从 0 变成 -1，P2 阻塞；当 P1 执行完 `S1` 后执行 `V(s)`，P2 被唤醒。



互斥：P 和 V 通常在同一个进程中成对出现。
同步：P 和 V 通常分布在不同进程中，一个进程 V 发信号，另一个进程 P 等信号。

若某个进程必须等多个前驱完成，就对多个信号量执行 P；若某个进程完成后要通知多个后继，就对相关信号量执行多次 V。

### 用于资源计数：一般信号量
如果系统中有 `k` 个同类资源，例如 `k` 个缓冲区、`k` 个空座位、`k` 个可用连接，则可以设：
```c
semaphore resource = k;
```
每申请一个资源：
```c
P(resource);
```
每释放一个资源：
```c
V(resource);
```
这时信号量不是简单的锁，而是资源计数器。

例如缓冲区有 `k` 个空位：
```c
semaphore empty = k;
```
生产者放入一个产品前必须执行：
```c
P(empty);
```
表示消耗一个空缓冲区；消费者取走一个产品后执行：
```c
V(empty);
```
表示释放一个空缓冲区。

## PV 操作典型问题
### 售票问题：互斥访问共享数据
假设多个售票进程共享余票数组 `A[m]`，其中 `A[j]` 表示第 `j` 个航班剩余票数。如果多个进程同时读写 `A[j]`，可能出现超卖。
每个航班一把锁：
```c
semaphore s[m];

for (int j = 0; j < m; j++) {
    s[j] = 1;
}
```

### 哲学家就餐问题：PV 使用不当会死锁
信号量能解决互斥，但使用不当也会导致死锁。五个哲学家围坐，每两人之间一把叉子，每个哲学家必须同时拿到左右两把叉子才能吃饭。

问题：如果五个哲学家都先拿起右手叉子，然后都等待左手叉子，就会发生死锁。

常见改进有三类：
第一，最多允许 4 个哲学家同时尝试取叉子：
第二，奇偶哲学家取叉顺序不同。比如偶数先右后左，奇数先左后右。这样破坏循环等待。
第三，要么同时获得两把叉子，要么一把也不拿。这属于更高级的 AND 型资源申请思想。

核心：理解
```
连续执行两个 P 操作之间存在“缝隙”。
进程可能在拿到第一个资源后被切换，从而造成环路等待。
```

### 生产者—消费者问题
有 `n` 个生产者和 `m` 个消费者，共享一个有 `k` 个缓冲单元的有界缓冲区。缓冲区未满时生产者可以放入产品；缓冲区不空时消费者可以取走产品。

```c
semaphore sput = 1; // 空缓冲区数
semaphore sget = 0; // 可取产品数
int B;

生产者
process producer() {
    while (true) {
        produce(product);

        P(sput);     // 等待空缓冲区
        B = product;
        V(sget);     // 增加一个产品
    }
}


消费者
process consumer() {
    while (true) {
        P(sget);     // 等待产品
        product = B;
        V(sput);     // 释放空缓冲区

        consume(product);
    }
}
```
这里没有额外互斥锁，因为只有一个生产者、一个消费者，且单缓冲区的放取顺序已经由 `sput/sget` 控制。

### 多个生产者、多个消费者、多个缓冲区
此时问题变复杂。多个生产者会同时竞争 `putptr`，多个消费者会同时竞争 `getptr`，所以除了同步信号量，还需要互斥信号量。

```c
int B[k];
int putptr = 0;
int getptr = 0;

semaphore sput = k; // 空缓冲区数
semaphore sget = 0; // 产品数
semaphore mutex = 1; // 互斥访问缓冲区指针和缓冲区

生产者
process Producer_i() {
    while (true) {
        produce(product);

        P(sput);      // 先确认有空位
        P(mutex);     // 再进入临界区

        B[putptr] = product;
        putptr = (putptr + 1) % k;

        V(mutex);
        V(sget);      // 通知消费者有产品
    }
}

消费者
process Consumer_j() {
    while (true) {
        P(sget);      // 先确认有产品
        P(mutex);     // 再进入临界区

        product = B[getptr];
        getptr = (getptr + 1) % k;

        V(mutex);
        V(sput);      // 通知生产者有空位

        consume(product);
    }
}
```
这里有一个非常重要的顺序：

```
生产者必须先 P(sput)，再 P(mutex)。
消费者必须先 P(sget)，再 P(mutex)。
```
不要先拿 `mutex` 再等 `sput/sget`。否则可能出现：生产者拿着 `mutex` 等空位，但消费者因为拿不到 `mutex` 无法取产品释放空位，系统死锁。

优化版本：用两个互斥信号量 `s1` 和 `s2` 分别保护 `putptr` 和 `getptr`，使生产者之间互斥、消费者之间互斥，但生产者和消费者在不同指针上可以更并行。

### 苹果—桔子问题
桌上有一只盘子，每次只能放一个水果。爸爸只放苹果，妈妈只放桔子；儿子只取桔子，女儿只取苹果。

这个问题的本质是“一个缓冲区 + 两类产品 + 指定消费者”。

### 管程
PV 操作虽然强大，但有明显缺点：
```
P/V 分散在各个进程中，容易写错。
P/V 顺序错误可能死锁。
忘记 V 会导致永久阻塞。
误用信号量初值会导致同步错误。
共享变量和同步操作没有被统一封装。
```
引入管程的原因：把分散在各进程中的临界区集中起来管理；防止进程有意或无意的违法同步操作；便于用高级语言书写程序。

所以管程的目标是：
`把共享数据、访问共享数据的过程、同步互斥机制封装在一个模块中。`
PV 操作像“手动加锁”，管程像“把锁和资源封装到类里”。

### 管程的定义
管程是由局部于自己的若干公共变量及其说明，以及所有访问这些公共变量的过程组成的软件模块。管程具有共享性、安全性、互斥性。

可以把管程理解成一种特殊的类：
```c
monitor MonitorName {
    // 局部共享变量
    // 条件变量
    // 初始化语句

    procedure p1(...) {
        ...
    }

    procedure p2(...) {
        ...
    }
}
```
管程外的进程不能直接访问管程内部数据，只能调用管程提供的过程。

例如生产者消费者问题中，外部进程不直接操作 `B[k]`、`in`、`out`、`count`，只能调用：
```c
append(x);
take(x);
```
这样可以把错误限制在管程内部，外部进程更不容易破坏同步规则。

### 管程的基本结构
```
局部变量说明
条件变量说明
初始化语句
管程内可被外部调用的过程/函数列表
管程内过程/函数体
```
![[截屏2026-06-25 18.48.56.png]]
管程最重要的隐含规则是：
`任一时刻，最多只有一个进程在管程内部执行。`
这就是管程的互斥性。进程调用管程过程时，如果已有其他进程在管程内，它必须在入口队列等待。

### 条件变量 condition variable
管程内部不仅需要互斥，还需要同步。为此引入条件变量。
条件变量是出现在管程内的一种数据结构，只能在管程中访问，对管程内所有过程是全局的，只能通过 `wait()` 和 `signal()` 两个原语控制。

#### wait(c)
当前进程因条件不满足而阻塞在条件变量 c 上；
同时释放管程，使其他进程可以进入管程。

例如生产者发现缓冲区满：
```c
if (count == k) {
    wait(notfull);
}
```

消费者发现缓冲区空：
```c
if (count == 0) {
    wait(notempty);
}
```

#### signal(c)
如果有进程在条件变量 c 上等待，则唤醒一个；
如果没有进程等待，则该 signal 不保存。

这一点非常重要。条件变量不是信号量，`signal(c)` 不会像 `V(s)` 一样累计资源数。如果没人等，这次 signal 就丢失。

## Hoare 管程
### signal 后谁继续执行？
管程中有一个核心问题：如果进程 A 在管程中执行 `signal(c)` 唤醒进程 B，那么此时 A 和 B 都可能想在管程中继续执行。可是管程要求同一时刻只有一个进程在管程内。

方法一：执行 signal 的进程等待，让被唤醒进程立即进入管程执行。
方法二：被唤醒进程等待，直到 signal 进程退出管程。

Hoare 管程采用第一种办法；Hansen 管程采用折中方案，规定 signal 必须是过程体最后一个操作。

Hoare 管程语义可以概括为：
```
signal 后，被唤醒进程立即获得管程执行权；
发出 signal 的进程暂时等待。
```
因此在 Hoare 管程里，被唤醒进程恢复时，它等待的条件仍然成立，所以课件例子中常用 `if` 判断条件。

### Hoare 管程用 PV 实现
Hoare 方法使用 P/V 操作实现管程中过程的互斥调用，以及共享资源的互斥使用管理。
每个管程需要三个层次的数据：

#### mutex
```c
semaphore mutex = 1;
```

用于保证管程互斥。进程进入管程过程前执行：

```c
P(mutex);
```

退出管程时执行：

```c
V(mutex);
```

但由于 Hoare 管程中有 `signal` 立即交接机制，所以退出时不一定直接 `V(mutex)`。

#### next 和 next_count
```c
semaphore next = 0;
int next_count = 0;
```

`next` 用来阻塞发出 `signal` 的进程。

在 Hoare 管程中，A 唤醒 B 后，A 不能继续占着管程，而是到 `next` 上等待。等 B 退出管程或再次等待时，再唤醒 A。课件指出，`next_count` 用来记录在 `next` 上等待的进程数

#### x_sem 和 x_count
每个条件变量 `x` 对应：
```c
semaphore x_sem = 0;
int x_count = 0;
```
```
x_sem：阻塞等待条件 x 的进程。
x_count：记录有多少进程正在等待条件 x。
```
当申请资源得不到满足时，执行 `P(x_sem)` 阻塞；释放资源时，需要通过 `x_count` 判断是否有人等待，并用 `V(x_sem)` 唤醒。

#### Hoare 管程 enter 和 leave
```c
void enter(InterfaceModule &IM) {
    P(IM.mutex);
}
// 含义：调用管程过程前，先获得管程互斥权。

void leave(InterfaceModule &IM) {
    if (IM.next_count > 0)
        V(IM.next);
    else
        V(IM.mutex);
}
/*
退出管程时，如果有因 signal 而等待的进程，则优先唤醒它；
否则开放管程，让入口队列中的新进程进入。
*/
```
#### Hoare 管程 wait
```c
void wait(semaphore &x_sem, int &x_count, InterfaceModule &IM) {
    x_count++;

    if (IM.next_count > 0)
        V(IM.next);
    else
        V(IM.mutex);

    P(x_sem);

    x_count--;
}
```
1. x_count++：登记当前进程正在等待条件 x。
2. 释放管程：
   如果 next 上有人等待，先唤醒 next；
   否则释放 mutex，让其他管程调用者进入。
3. P(x_sem)：当前进程阻塞在条件 x 上。
4. 被 signal 唤醒后，x_count--，重新进入管程继续执行。

#### Hoare 管程 signal
```c
void signal(semaphore &x_sem, int &x_count, InterfaceModule &IM) {
    if (x_count > 0) {
        IM.next_count++;
        V(x_sem);
        P(IM.next);
        IM.next_count--;
    }
}
```
1. 如果 x_count > 0，说明条件变量 x 上有人等待。
2. 当前 signal 进程先把 next_count++，表示自己即将进入 next 队列。
3. V(x_sem)：唤醒一个等待条件 x 的进程。
4. P(IM.next)：signal 进程自己阻塞，让被唤醒进程立即运行。
5. 将来被唤醒后，next_count--。

signal 不是简单通知，而是立即交接管程执行权。

# 进程通信
在前面的 PV 操作和管程中，我们已经学习了并发进程之间的**互斥**和**同步**。但信号量本质上是一种比较低级的通信方式，它主要表达“能不能进入”“要不要等待”“是否释放资源”，并不擅长表达复杂数据交换。
交往进程可以通过信号量实现互斥和同步，但这属于低级通信方式；当进程需要交换更多信息，例如把数据传送给另一个进程时，就需要引入高级通信方式，也就是进程通信机制。进程交互必须满足两个基本要求：一是同步，二是通信；消息传递提供了这两个功能，典型原语是 `send` 和 `receive`

**同步**解决的是“什么时候做”的问题。  
例如消费者必须等生产者生产出数据后才能消费。

**通信**解决的是“传什么数据”的问题。  
例如生产者不仅要通知消费者“有数据了”，还要把具体数据传过去。

所以，进程通信比单纯同步更高级。它不仅能协调执行顺序，还能传递具体信息。

### 进程通信的基本模型：消息传递
进程通信主要按**消息传递**来讲。所谓消息传递，就是进程之间不直接共享变量，而是通过操作系统提供的机制发送和接收消息。

```c
send(destination, message);
receive(source, message);
```
其中：

`send` 表示发送消息；  
`receive` 表示接收消息；  
`message` 是被传递的数据；  
`destination/source` 表示发送或接收对象。

消息传递有一个重要特点：通常由**内核**介入。用户进程不能随便访问另一个进程的地址空间，所以发送方的数据一般要经过内核缓冲、复制或队列管理，再交给接收方。![[截屏2026-06-26 17.12.09.png]]
### 消息格式
![[截屏2026-06-26 17.43.23.png]]
**消息类型**：说明这是什么消息，例如普通数据、控制命令、错误通知、结束标志等。  
**目标 ID**：说明消息发给谁。  
**源 ID**：说明消息来自谁。  
**消息长度**：说明消息体有多长，方便系统分配缓冲和解析。  
**控制消息**：用于附加控制信息，例如优先级、权限、状态码等。  
**消息内容**：真正要传输的数据。


## 直接通信
 直接通信是指发送进程直接把消息发送给某个进程。也就是说，通信双方通过进程名或进程标识直接建立联系。
直接通信分为两种：**对称直接寻址**和**非对称直接寻址**。

### 对称直接寻址
对称直接寻址要求发送方和接收方都明确命名对方。

```c
send(P, message);       // 发送消息到进程 P
receive(Q, message);    // 接收来自进程 Q 的消息
```
发送者必须知道接收者是谁；  
接收者也必须知道发送者是谁。

这种方式的优点是关系明确、语义清楚。缺点是进程之间耦合度高：如果发送者或接收者的名字变化，另一方也必须修改。

### 非对称直接寻址
非对称直接寻址只要求发送方命名接收方，而接收方不必指定发送方。
```c
send(P, message);       // 发送消息到进程 P
receive(id, message);   // 接收来自任意进程的消息，并把发送者 ID 存入 id
```
例如：
```c
// 客户端进程
send(Server, request);

// 服务器进程
receive(client_id, request);
```
服务器并不提前指定“只能接收某个客户端”的消息，而是从任意客户端接收请求，同时通过 `client_id` 得知是谁发来的。

这种方式非常适合服务器模型。例如 Web 服务器、数据库服务器、打印服务进程都可能同时接收很多客户端进程的请求。

### 直接通信的特点
直接通信的核心特点是：**通信对象是进程本身**。

优点是简单、直观、无需额外命名对象。  
缺点是进程之间绑定较紧，通信关系不够灵活。

## 间接通信
间接通信不是把消息直接发给某个进程，而是发到一个中间对象中。这个中间对象通常称为**信箱 mailbox**，也可以理解为消息队列。

消息不是直接从发送者发送到接收者，而是发送到由临时保存消息的队列组成的共享数据结构中，这些队列通常称为信箱；一个进程向合适的信箱发送消息，另一个进程从信箱中获得消息。
### 信箱结构
信箱可分为信箱头和信箱体两部分。信箱头指出信箱容量、信件格式、存放信件位置的指针等；信箱体用来存放信件，并分成若干区，每个区可容纳一封信。
```
信箱 mailbox
├── 信箱头
│   ├── 信箱容量
│   ├── 当前消息数量
│   ├── 消息格式
│   ├── 队列头指针
│   └── 队列尾指针
└── 信箱体
    ├── 消息槽 1
    ├── 消息槽 2
    ├── ...
    └── 消息槽 n
```
信箱本质上是一个有同步控制的缓冲队列。
### end 和 receive 的阻塞逻辑
发送信件时，如果指定信箱未满，则将信件送入信箱，并释放等待该信箱中信件的等待者；否则发送者进入等待信箱状态。接收信件时，如果指定信箱中有信，则取出一封信，并释放等待信箱的等待者；否则接收者进入等待信件状态。

### 间接通信的优点
间接通信相比直接通信，最大的优点是**解耦**。

发送者不必知道具体由哪个进程接收消息，只要知道信箱名。  
接收者也不必知道具体由哪个进程发送消息，只要从信箱取消息。
