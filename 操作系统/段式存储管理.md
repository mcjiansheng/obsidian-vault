[[存储管理]]
### 为什么要引入段式存储管理
分页是信息的物理单位，与源程序逻辑结构无关，用户不可见；页长由系统确定，页面只能从页大小整数倍地址开始
但程序本身并不是天然按固定大小页面组织的。一个程序从逻辑上通常包括：
主程序段
子程序段
函数库代码段
全局数据段
堆段
栈段
数组段
工作区段
这些部分在含义、权限、共享方式、增长方向上都可能不同。例如代码段通常只读可执行，数据段可读可写，栈段需要动态增长，共享库代码段可以被多个进程共享。

所以段式管理的出发点是：
**按照程序的逻辑结构来划分地址空间。**

每个程序可以由若干段组成，每段都可以从 0 开始编址，段内地址连续；分段存储器的逻辑地址由两部分组成：**段号:单元号**。

## 段式存储管理
### 段式程序设计的基本思想
在段式程序设计中，一个程序不再被看成单一的一维地址空间，而是被看成多个逻辑段组成的集合。

```
段0：主程序段
段1：子程序 X
段2：子程序 Y
段3：数组 A
段4：工作区段
段5：堆栈段

每个段内部从 0 开始编址：
段0：0, 1, 2, ..., n
段1：0, 1, 2, ..., m
段2：0, 1, 2, ..., k

所以段式逻辑地址不是单纯的一个数，而是二维地址：
逻辑地址 = 段号 : 段内地址
```

### 段式存储管理的实现基础
**段式存储管理基于可变分区存储管理实现，一个进程要占用多个分区**。
也就是说，每个段在主存中占用一个连续分区，但整个进程的多个段之间不要求连续。

连续分区管理：
一个进程整体占用一个连续分区

段式管理：
一个进程由多个段组成，每个段占用一个连续分区
不同段之间可以离散存放

主存：
`[ OS ][ 进程A代码段 ][ 空闲 ][ 进程B数据段 ][ 进程A数据段 ][ 进程A栈段 ]`

进程 A 的各段不必相邻，但每一段内部必须连续。

这意味着段式管理比单连续分区灵活，但仍然存在可变分区管理的外部碎片问题。因为段长不固定，每段都要求连续空间。

### 段表
为了知道每个段放在哪里，系统需要为每个进程建立一张**段表**。

每个段占用一个段表项，段表项包括：**段始址、段限长，以及存储保护、可移动、可扩充等标志位**。

|字段|作用|
|---|---|
|段号|标识第几个段|
|段基址 / 段始址|该段在主存中的起始物理地址|
|段长 / 段限长|该段的长度，用于越界检查|
|保护位|控制该段是否可读、可写、可执行|
|共享位|标识该段是否共享|
|修改位|标识该段是否被修改|
|可扩充位|标识该段能否增长|
|可移动位|标识该段是否可被移动或换出|

### 段式地址转换过程
```
逻辑地址 = 段号 S + 段内地址 B
```

第一步：从段表控制寄存器中得到当前进程段表
第二步：根据逻辑地址中的段号 S 查段表
第三步：得到该段的段基址和段长
第四步：检查段内地址 B 是否小于段长
第五步：如果 B 合法，物理地址 = 段基址 + B
第六步：如果 B 越界，产生越界中断

```
若 B < L[S]：
    物理地址 = Base[S] + B
否则：
    越界中断
    
其中
S：段号
B：段内偏移量 / 单元号
Base[S]：第 S 段的物理起始地址
L[S]：第 S 段长度
```

### 段的共享
段式管理非常适合共享，因为段本来就是具有逻辑意义的单位。

段的共享可以通过**不同进程段表中的项指向同一个段基址**实现；对共享段必须进行保护，例如规定只能读不能写，不满足保护条件则产生保护中断。

共享段最典型的例子是：
- 共享库代码段
- 只读公共函数段
- 多个进程共同访问的共享数据段
### 段式管理的保护能力
段式管理天然适合做保护，因为每个段有明确含义。

代码段：只读、可执行、不可写
数据段：可读、可写、不可执行
栈段：可读、可写、可扩充
共享库段：只读、可执行、可共享

系统可以在段表项中设置不同权限。访问时硬件检查权限：
如果写代码段 → 保护中断
如果执行数据段 → 保护中断
如果访问段外地址 → 越界中断

这比纯分页更符合程序结构，因为分页只是固定大小物理块，单独一个页不一定对应完整逻辑模块。

### 段式存储管理的优缺点
**优点**：
符合程序的逻辑结构
用户可见，便于模块化程序设计
便于共享，例如共享代码段和共享库段
便于保护，不同段可设置不同访问权限
段可动态增长，适合堆、栈等结构

**缺点**:
每段必须连续存放，仍然存在外部碎片
段长不固定，主存分配和回收复杂
地址转换需要查段表，有额外开销
段移动、紧缩需要动态重定位支持

## 段式虚拟存储管理
**基本思想**
段式虚拟存储管理是“分段”和“虚拟存储”的结合。

**把进程的所有分段都存放在辅存中，进程运行时先把当前需要的一段或几段装入主存，在执行过程中访问到不在主存的段时，再把它们动态装入。**
段的调进调出由 OS 自动实现，对用户透明；这不同于段覆盖技术，段覆盖技术是用户控制的主存扩充技术，OS 不感知。

段覆盖技术：
程序员自己安排哪个段何时覆盖哪个段。

段式虚拟存储：
操作系统自动判断缺段、调段、换段，用户程序不用关心。

### 段式虚拟存储如何运行
普通段式管理要求每个被访问的段都已经在主存中。段式虚拟存储则允许：
部分段在主存
部分段在辅存
访问不在主存的段时产生缺段中断

当程序调用子程序 Y 时，发现段 2 不在主存，于是触发缺段中断，操作系统把段 2 从辅存调入主存，然后重新执行被中断的指令。

### 段表的扩充
段式虚拟存储需要比普通段式管理更复杂的段表项。
段表扩充字段：**特征位、存取权限、扩充位、标志位、主存始址、限长、辅存始址**。

其中，特征位用于表示不在内存、在内存、共享段；
存取权限用于表示可执行、可读、可写；
扩充位表示固定长或可扩充；
标志位表示未修改、已修改、不可移动。

### 缺段中断
当程序访问一个不在主存中的段时，会产生**缺段中断**。这类似页式虚拟存储中的缺页中断，但调入单位从“页”变成了“段”。

CPU 访问逻辑地址 (S, B)
        ↓
查段表
        ↓
检查段 S 是否在主存
        ↓
若在主存：
    检查权限
    检查 B 是否越界
    形成物理地址
若不在主存：
    产生缺段中断
    OS 从辅存调入该段
    更新段表
    重新执行指令

### 段式虚拟存储中的连续空间问题
**段式虚拟存储虽然是虚拟存储，但调入一个段时仍然需要一块连续主存空间。**
因为段内部是连续编址的，段表只记录一个段基址和段长。所以一个段必须完整放入一块连续主存区。

### 可扩充段
段式虚拟存储比页式更适合表达“动态增长”的逻辑对象，比如栈段、堆段。
课件段表扩充字段中包含“扩充位”，用于标识固定长或可扩充。地址转换图中也包含对 “S 段可扩充” 和 “与 S 段末端相邻的空闲区长度是否满足要求” 的判断。

### 段式虚拟存储的优缺点
**优点**
逻辑结构清晰，符合程序员视角
以段为单位共享和保护更自然
支持动态链接、共享库、动态增长段
可实现比主存更大的逻辑地址空间

**缺点**
调入调出单位是段，段大小不固定
需要连续主存空间，仍有外部碎片
大段调入调出开销大
主存紧缩和段移动成本高
地址转换和段表维护复杂

## 段页式存储管理
### 为什么需要段页式
段式管理和页式管理各有优点，也各有问题。

页式管理：
```
优点：物理分配方便，无外部碎片，适合虚拟存储。
缺点：页是物理单位，不反映程序逻辑结构，用户不可见。
```

段式管理：
```
优点：符合程序逻辑，便于共享和保护。
缺点：每段要求连续空间，会产生外部碎片。
```

**分段是信息的逻辑单位，由源程序逻辑结构决定，用户可见；段长可按用户需要规定，段起始地址可从任意主存地址开始；分段方式中，源程序经连接装配后地址仍保持二维结构。**
**而分页是物理单位，与源程序逻辑结构无关，用户不可见，页长由系统确定。**

段页式的目的就是：
**逻辑上按段组织，物理上按页存放。**

### 段页式的基本思想
段页式存储管理中，一个进程先按逻辑结构分成若干段：
代码段
数据段
堆段
栈段
共享库段

然后每个段再被划分成若干页。主存仍然被划分成固定大小的页框。每个段中的页可以离散地装入主存页框。
```
进程逻辑空间：
段0：页0、页1、页2
段1：页0、页1
段2：页0、页1、页2、页3

物理主存：
页框3  ← 段0页1
页框7  ← 段2页0
页框9  ← 段1页1
页框12 ← 段0页0
```

这样就同时获得两方面优势：
```
段：体现程序逻辑结构，便于共享和保护
页：实现离散分配，避免外部碎片
```

### 段页式逻辑地址结构
段页式逻辑地址通常由三部分组成：
逻辑地址 = 段号 S + 页号 P + 页内位移 D

S：访问哪个段  
P：访问该段中的第几页  
D：访问该页中的哪个字节/单元

### 段表和页表
段页式中，每个进程有一张段表。段表中的每个表项不再直接给出该段在主存中的基址，而是通常给出该段对应页表的位置、页表长度以及段级保护信息。

每个段又有自己的页表。页表记录该段内部的页到物理页框的映射。

### 段页式地址转换过程
CPU 给出逻辑地址 (S, P, D)
        ↓
根据段号 S 查段表
        ↓
检查段号是否合法，检查段级权限
        ↓
得到该段页表始址和页表长度
        ↓
检查页号 P 是否超过该段页表长度
        ↓
根据 P 查该段页表
        ↓
得到物理页框号 F
        ↓
物理地址 = F × 页大小 + D

```
逻辑地址 = (S, P, D)

段表[S] → 该段页表地址 PT_S、页表长度 Len_S
如果 P ≥ Len_S：越界中断
页表_S[P] → 页框号 F
物理地址 = F × page_size + D
```
### 段页式中的快表
段页式地址转换比页式更复杂，因为至少涉及：
查段表
查页表
访问实际数据

如果没有快表，访存次数会明显增加。因此段页式系统通常也使用快表 TLB。
快表可以缓存：
```
(段号, 页号) → 页框号
```

这样 CPU 访问 `(S, P, D)` 时，先查快表：
```
快表命中：
    直接得到页框号 F
    物理地址 = F × 页大小 + D

快表未命中：
    查段表
    查页表
    更新快表
```

### 段页式虚拟存储中的缺段与缺页
段页式虚拟存储中可能发生两类中断：
**缺段中断**：该段的页表或段信息还不在主存
**缺页中断**：段存在，但访问的页不在主存

**缺段**：连这个逻辑段的管理信息或段整体都没准备好
**缺页**：段已经存在，但段中的某一页没调入主存

段页式可以把段作为逻辑保护和共享单位，把页作为调入调出单位。这样比纯段式更高效，因为不必一次调入整个大段。

### 段页式的优缺点
**优点**
逻辑结构清晰：保留段式的模块化视角
便于保护：按段设置读、写、执行权限
便于共享：多个进程段表可指向共享段或共享页表
无外部碎片：物理内存按页框分配
适合虚拟存储：可以按页调入调出
比纯段式减少大段调入调出的开销

**缺点**
地址转换复杂，需要段表和页表
表项开销更大
一次访存可能涉及多次查表
实现复杂，强依赖快表
段表、页表、快表切换和维护成本高
