## 处理器

### 处理器与寄存器

#### 用户程序可见寄存器
—— CPU内部给程序临时存放数据和地址的小型高速存储空间

用户程序可以直接使用的寄存器

作用：让程序员减少访问主存储器的次数，从而提高指令执行效率；
可以被所有程序使用，包括应用程序和系统程序

主要包括 **数据寄存器** 和 **地址寄存器**

**数据寄存器** 也叫通用寄存器

主要用来保存计算过程中的数据

**example**

```
a = b + c
执行伪过程：
把 b 放入 R1
把 c 放入 R2
R3 = R1 + R2
把 R3 写会 a
```

**地址寄存器** 用于保存内存地址而非普通数据

主要包括 **索引寄存器、栈指针、段地址寄存器** 等

用途：

1 **索引寄存器**
用于数组访问

对于 `arr[i]` CPU需要知道数组起始地址和偏移量，索引寄存器可用于帮助计算目标元素地址

2 **栈指针寄存器**
用于指向当前栈顶

函数调用、局部变量、返回地址、参数传递等都和栈有关

3 **段地址寄存器**
在某些体系结构中，内存被分成代码段、数据段、栈段等，段地址寄存器用于保存段等基地址

#### 控制与状态寄存器

用于控制CPU操作、记录CPU状态等一类寄存器

主要被具有特权等操作系统程序使用，用于控制程序执行。包括PC IR CC 标志位等

通常不能随便让普通用户程序修改

1 **PC Program Counter 程序计数器**
保存下一条将要取出等指令地址

2 **IR Instruction Register 指令寄存器**
保存当前正在执行或最近取出等指令

CPU执行等基本流程：
```
根据PC取出指令
⬇️
放入IR
⬇️
对IR中的指令译码
⬇️
执行指令
```
IR 是 CPU 当前动作的依据

3 **CC Condition Code 条件码**
用于记录CPU运算结果的状态

条件码是CPU为指令操作结果设置的位，用于标志正、负、零、溢出等结果

```c
if (a==b){
	...
}
```
CPU先执行指令 `a-b` 根据结果设置条件码
然后条件跳转指令更具标志决定是否跳转

4 标志位
例如 `中断位、中断允许位、处理器模式位、内存保护位等`

**中断允许位**
用于选择性屏蔽某些中断

**处理器模式位**
表示CPU当前位于用户态还是内核态

处理器模式位非常重要，因为他决定当前能否执行特权指令

**内存保护位**
用于帮助实现内存访问保护
比如用户程序不能访问内核地址空间，不能随便访问其他进程的内存空间

#### PSW 程序状态字
——当前程序运行状态的“总说明书” 或 “状态快照”
**Program Status Word** 用于记录当前程序运行状态的一组信息

可以表现为一组控制与状态寄存器，也可以是专门的PSW寄存器

PSW通常包含：`程序计数器PC 指令寄存器IR 条件码 中断字 中断允许/禁止位 中断屏蔽位 处理器模式位 内存保护位 调试控制信息等`

*在不同体系结构中，PSW的具体实现不同*

**PSW的重要性**
- PSW决定程序能否正确继续执行
- PSW是中断和模式切换的核心
- PSW是进程上下文的重要组成部分

### 指令与处理器模式

#### 机器指令

机器指令是计算机系统能够执行的基本命令，也是CPU执行的基本单位

机器指令由一个或多个字节组成，通常包括操作码字段、一个或多个操作数地址字段，以及一些表示机器状态的状态字或特征码；机器指令可以完成算数逻辑运算、数据传输和控制流跳转等操作

```
高级语言代码
⬇️编译 / 解释
汇编指令
⬇️汇编
机器指令
⬇️
CPU执行
```

一条机器指令一般包含
1. **操作码**
   说明这条指令要做什么
2. **操作数或操作数地址**
   说明这条指令操作谁
3. **状态信息或特征码**
   某些指令还会涉及机器状态，例如条件码、寻址方式、特权级别等

**主要类型**
1. 算数逻辑指令
   加减乘除、与或非...
2. 数据传输指令
   用于在寄存器、内存、I/O设备之间传送数据
3. 控制流指令
   用于改变程序执行顺序
#### 指令执行过程

`取指 ➡️ 解码 ➡️ 执行`

CPU根据PC取出指令，放入IR，并对指令译码，然后发出各种控制命令，执行一系列微操作，从而完成一条指令的执行

**取指**
根据PC中的地址，从内存或高速缓存Cache中取出指令，并放入IR
`IR <- Memory[PC]`

**解码**
解释IR中的指令，判断这条指令要做什么

**执行**
真正完成指令要求的操作
不同类型指令的执行动作不同

#### 指令执行周期

指令执行周期是CPU完成一条指令所经历的完整过程

```
取指 -> 解码 -> 执行
```

不考虑流水线时，CPU可能这样执行

```
时间 →
I1: 取指  解码  执行
I2:                取指  解码  执行
I3:                               取指  解码  执行
```
当I1执行时，取指部件和译码部件可能闲着，当I1正在解码时，执行部件可能闲着

#### 指令流水线
**指令流水线**就是把一条指令的执行过程拆成了多个阶段，让多条指令在不同阶段重叠执行

```
时间 →
I1: 取指  解码  执行
I2:       取指  解码  执行
I3:            取指  解码  执行
I4:                  取指  解码  执行
```
单条指令的执行时间不一定减少，但单位时间内完成的指令书增加

**为什么能提高效率？** CPU内部有多个功能部件，不同指令中同时可以占用不同部件，提高硬件利用率

**流水线可能遇到的问题** 
1. 结构相关：多个指令同时需要同一个硬件部件，可能会产生冲突
2. 数据相关：后一条指令可能依赖前一条指令的结果
3. 控制相关：跳转指令会改变PC，导致CPU不确定下一条应该取哪条指令

#### 特权指令与非特权指令

**为什么要区分？** 操作系统要保护核心资源：用户程序不能使用全部机器指令，哪些与计算机核心资源相关的特殊指令会被保护，例如启动I/O指令、置PC指令等；核心资源相关的指令只能有操作系统程序使用

**特权指令**
只能由操作系统内核使用的指令

**非特权指令**
所有程序都可以使用的普通指令

如果用户程序想做特权操作，只能通过 **系统调用** 请求操作系统代为执行

**example**
用户程序想读取文件
```c
read(fd,buf,size);
```

不能直接控制磁盘设备

```
用户程序调用 read()
⬇️
触发系统调用
⬇️
CPU 从用户态切换到内核态
⬇️
操作系统检查权限和参数
⬇️
内核执行真正的 I/O 操作
⬇️
返回用户态
```

#### 处理器模式

CPU当前运行权限级别的标志
决定当前正在运行的程序可以执行哪些指令、访问哪些资源

常见设计可以有0、1、2、3四种运行模式，分别对应不同保护级别：其中0模式可以执行全部指令，3模式只能执行非特权指令，其他模式可以执行规定的指令子集

现代操作系统中一般只使用0和3两种模式 分别对应内核态和用户态

**用户模式**
普通应用程序运行的模式
只能执行非特权指令，不能直接访问硬件设备和内核空间等

即使用户程序崩溃，通常也只是当前进程崩溃，不会导致整个操作系统崩溃

**内核模式**
操作系统内核运行的模式

#### 处理器模式切换
模式切换就是CPU中用户模式和内核模式之间转换

中断、异常或系统异常等事件会导致用户程序切换到OS内核，从而触发用户模式到内核模式到转换；内核处理完成后，调用中断返回指令，触发内核模式到用户模式到转换

`处理器模式决定当前程序能不能执行特权指令`

## 中断管理

### 中断

操作系统不是一直“主动运行”的，大多数时候，CPU正在执行某个用户程序或内核程序；当发生某些事件时，CPU才会暂停当前执行流，转去执行操作系统的处理程序

处理完成后，再返回原程序被中断处，或者调度其他程序执行

**广义中断**
广义上，只要某个事件打断了当前程序的正常执行，让CPU转去执行操作系统处理程序，都可以称为中断

**狭义中断**
狭义中断通常指 **来自处理器外部的、与当前正在执行的指令无直接关系的事件**

**中断的本质**
```
当前执行流被暂停
⬇️
保存现场
⬇️
CPU转入内核态
⬇️
执行中断处理程序
⬇️
恢复原执行流或调度新进程
```

### 异常
**异常时由当前正在执行的指令引起的中断事件**

常见异常包括地址异常、算数异常、处理器硬件故障等

**异常和中断的区别**

|对比项|狭义中断|异常|
|---|---|---|
|来源|CPU 外部设备或外部信号|CPU 执行当前指令产生|
|是否与当前指令有关|通常无关|直接有关|
|发生时机|通常异步|通常同步|
|例子|时钟中断、I/O 中断、键盘中断|除零、非法指令、缺页、越界|

**系统异常/陷入/系统调用**

执行陷入指令触发系统调用引起的中断事件，例如请求设备、请求I/O、创建进程等

系统异常通常是主动请求OS服务，与普通异常不同
用户程序不能直接访问硬件和内核资源，必须通过系统调用进入内核

**陷阱、故障、终止**

**故障 Fault** 通常是可以被修复等异常。处理后，可能重新执行引发故障等那条指令

example ： **缺页异常**
程序访问某个虚拟地址，但对应页面还没调入内存
```
访问虚拟地址
⬇️
发现页不在内存
⬇️
产生缺页异常
⬇️
OS把页面掉入内存
⬇️
重新执行原指令
```

**陷阱 Trap** 通常是有意触发等异常，常用于系统调用或测试断点

**终止 Abort** 是严重异常，通常无法恢复

### 中断源

中断源就是引发中断、异常或系统调用的来源

#### 处理器硬件故障中断事件
由处理器、内存、总线等硬件故障引起的中断事件

通常很严重，不一定能由操作系统自动恢复

#### 程序性中断事件
处理器执行机器指令时产生的异常
对不同程序性中断有不同的处理方式：除零、溢出等算数异常可以简单报告给用户；非法指令、用户态使用特权指令、地址越界、非法存取通常终止进程；虚拟地址异常则可能调整内存后重新执行指令。

**缺页异常** 虽然也是异常，但不是一定要杀死进程，而是正常机制等一部分

#### 自愿性中断事件
自愿性中断事件就是 **系统调用**

是处理器执行陷入指令请求OS服务引起的中断事件，又称系统调用。常见例子包括请求分配外设、请求I/O等。处理流程是陷入OS，保护现场、根据功能号查入口地址、跳转到具体处理程序

被称为“自愿性”是因为用户程序主动请求内核服务

#### I/O中断事件
来源于外部设备，用于报告I/O状态。
主要包括三类`I/O完成、I/O出错、I/O异常`

I/O完成时，操作系统会调整进程状态，释放等待进程，并把它加入就绪进程队列

```
进程A请求读磁盘
⬇️
磁盘很慢，A进入等待态
⬇️
CPU调度其他进程运行
⬇️
磁盘读完后发出I/O中断
⬇️
OS响应中断
⬇️
把A从等待队列移入就绪队列
```

#### 外部中断事件
由外围设备或外部信号引起

时钟/间隔时钟中断
设备报道与结束中断
键盘/鼠标信号中断
关起/重启动中断

**时钟中断** 尤其重要，因为它是分时系统实现时间片轮转的基础

**时间片流转**

```
进程A运行
⬇️
时钟中断到来
⬇️
OS进入内核
⬇️
发现A时间片用完
⬇️
A进入就绪队列
⬇️
调度进程B运行
```
如果没有时钟中断，一个用户进程可能长时间占用CPU，操作系统无法及时调度其他进程

### 中断系统
中断系统是计算机系统中响应和处理中断的系统

中断系统包括硬件子系统和软件子系统两部分：中断响应由硬件子系统完成，中断处理由软件子系统完成，体现了软硬件协同的设计思想

```
硬件负责发现中断、保存关键现场、挑战入口
软件负责分析原因、执行处理逻辑、恢复或调度
```


### 中断响应 vs 中断处理

#### 中断响应
中断响应偏硬件层面

中断响应过程发现中断源、发现中断寄存器中记录的中断、判断是否屏蔽、多个中断源时按优先级选择一个、中断当前程序执行、保存当前程序PSW/PC到核心栈、转向操作系统中断处理程序

#### 中断处理
中断处理偏软件层面

中断处理程序是OS处理中断事件的控制程序，主要任务是处理中断事件和恢复正常操作。处理过程包括保护未被硬件保护的处理器状态、通过PSW中断码字段识别中断源、查中断向量表执行中断处理子程序、恢复正常操作

#### 中断装置
中断装置是硬件层面的概念

计算机系统中发现并响应中断或异常的硬件装置称为中断装置。由于中断源多样，中断装置也有多种：处理器外的中断由中断控制器发现和响应；处理器内的异常由指令控制逻辑和实现线路发现和响应，对应机制称为陷阱；请求OS服务的系统异常由处理器执行陷入指令直接触发，对应机制称为系统陷阱

```
外部设备事件 ➡️ 中断控制器
指令执行异常 ➡️ CPU 内部异常检测逻辑
系统调用 ➡️ 陷入指令 / 系统陷阱
```

#### 中断控制器
中断控制器负责接收外部设备的中断请求，并协助CPU判断是否响应

中断控制器是CPU中的控制部件，包括中断控制逻辑线路和中断寄存器；外部设备向其发出IRQ，中断寄存器记录已发生的中断；指令处理结束前，CPU检查中断寄存器，如果有不被屏蔽的中断，则改变处理器内部操作顺序，引出OS中断处理程序

IRQ： Interrupt Request 中断请求
中断寄存器：记录哪些中断已经发生
中断屏蔽：判断某个中断是否暂时不响应
中断优先级：多个中断同时出现时决定先处理谁


#### 中断向量表
中断向量表是保存中断处理程序入口地址的一张表

中断向量表时包含中断服务程序地址的特定内存区域，这些服务程序是处理中断请求的代码

可以想象为：

|中断号|中断类型|处理程序入口|
|---|---|---|
|0|除零异常|divide_error_handler|
|14|缺页异常|page_fault_handler|
|32|时钟中断|timer_interrupt_handler|
|33|键盘中断|keyboard_interrupt_handler|
|80|系统调用|syscall_handler|
当CPU发现某个中断号时，就通过中断向量表找到对应处理程序

```
发生中断
⬇️
得到中断号 / 异常号
⬇️
查中断向量表
⬇️
跳转到对应中断服务程序
```

### 中断响应过程

1. 中断源发出中断请求
2. 中断处理器或CPU内部逻辑发现中断
3. 将中断信息记录到中断寄存器或异常状态中
4. CPU在合适时机检查中断
5. 判断该中断是否被屏蔽
6. 如果多个中断同时发生，按优先级选择一个
7. 暂停当前程序执行
8. 保存当前程序PC / PSW 到核心栈
9. CPU 切换到内核态
10. 根据中断向量表跳转到 OS 中断处理程序

### 中断处理过程

1. 保护未被硬件保护的处理器状态
2. 分析被中断进程PSW中的中断吗字段
3. 识别中断源
4. 查中断向量表，执行对应中断处理子程序
5. 恢复正常操作

恢复正常操作有两种情况：
一种是中断处理完成后，直接返回刚刚被中断的进程
另一种是需要中断当前进程运行，调整进程队列，启动低级调度，选择下一个进程执行。

#### 不引起进程切换的中断

键盘输入被缓冲，当前进程不受影响
某些简单时钟统计
某些可恢复的异常处理后继续执行

大致流程：
```
用户进程A运行
⬇️
发生中断
⬇️
进入内核
⬇️
处理中断
⬇️
恢复A现场
⬇️
返回A继续执行
```

#### 引起进程切换的中断

时间片中断，当前进程时间片用完
I/O完成中断，唤醒了更高优先级进程
当前进程执行阻塞式系统调用
当前进程发生不可恢复异常，被终止

大致流程：
```
流程A运行
⬇️
发生中断 / 异常 /系统调用
⬇️
进入内核
⬇️
保存A的上下文
⬇️
修改A的PCB和状态
⬇️
启动调度程序
⬇️
选择进程B
⬇️
恢复B的上下文
⬇️
返回用户态运行B
```

### 多中断管理

现实中可能多个中断同时发生，也可能在处理中断时又发生新的中断。因此需要多中断管理机制

#### 中断屏蔽

**中断屏蔽** 是指系统有选择地暂时不响应某些中断

当计算机检测到中断时，中断装置通过中断屏蔽位决定是否响应已发生中断，即有选择地响应中断

通常不能屏蔽所有中断，尤其是严重硬件故障这类不可屏蔽中断

#### 可屏蔽和不可屏蔽中断

**可屏蔽中断**
可以被CPU暂时屏蔽或延迟响应

例如： 普通 I/O 中断；键盘中断；某些设备中断

**不可屏蔽中断**
不能被普通屏蔽机制关闭，必须立即或尽快响应

例如：严重硬件故障；电源故障；内存校验错误；不可屏蔽中断通常对应系统级严重事件

#### 中断优先级
多个中断同时发生时，系统需要决定先响应哪个

中断优先级是当计算机同时检测到多个中断时，中断装置响应中断的顺序。不同类型OS有不同中断优先级：
处理器硬件故障、自愿性中断、程序性中断、时钟等外部中断、输入输出中断、重启动和关机中断

优先级设计通常考虑：
紧急程度；是否会影响系统安全；是否会丢失数据；是否实时性强；处理开销大小

#### 中断嵌套

**中断嵌套**是指CPU正在处理中断时，又响应了另一个中断

但是由于操作系统是性能攸关程序系统，且嵌套会增加开销，因此嵌套层数应有限制，例如三层；中断嵌套会改变中断处理次序，先响应的可能会后处理完成

### 中断与进程状态转换

中断管理不仅是“处理事件”，还和进程状态转换直接相关

运行进程因为等待事件、中断或系统调用进入内核；时间片用完或被抢占会让运行进程回到就绪态；等待事件结束会让等待态进程进入就绪态；return 可能 激活调度程序选择新进程运行；

**example**

时间片中断：
```
进程 A 运行
⬇️
时钟中断发生
⬇️
OS 判断 A 时间片用完
⬇️
A：运行态 ➡️ 就绪态
⬇️
调度 B 运行
```

I/O 请求系统调用
```
进程A 执行 read()
⬇️
系统调用进入内核
⬇️
发现需要等待磁盘
⬇️
A：运行态 ➡️ 等待态
⬇️
调度其他进程
```

I/O 完成中断
```
磁盘完成读操作
⬇️
发出 I/O 中断
⬇️
OS处理中断
⬇️
唤醒进程A
⬇️
A：等待态 ➡️ 就绪态
```

异常
```
进程 A 执行非法指令
⬇️
产生异常
⬇️
OS 进入异常处理程序
⬇️
终止 A
⬇️
调度其他进程
```

缺页异常
```
进程 A 访问缺页
⬇️
产生缺页异常
⬇️
OS 调页
⬇️
恢复 A 重新执行指令
```

### 中断与模式切换
中断、异常、系统调用都会让CPU从用户态进入内核态

中断、异常或系统异常等事件会导致用户程序向 OS 内核切换，触发“用户模式 ➡️ 内核模式”；OS 处理完成后，调用中断返回指令，例如 Intel 等 `iret` ,触发 “内核模式 ➡️ 用户模式”

中断处理等基本模式：

```
用户态程序运行
⬇️
中断 / 异常 /系统调用
⬇️
用户态 ➡️ 内核态
⬇️
执行中断处理程序
⬇️
内核态 ➡️ 用户态
```
普通用户程序不能直接处理硬件中断，必须由内核接管，这就是为什么中断管理是保护机制等一部分

### 重点概念对照表
|概念|含义|典型例子|
|---|---|---|
|广义中断|打断当前执行流并进入 OS 处理的事件|中断、异常、系统调用|
|狭义中断|来自处理器外部、与当前指令无关的事件|I/O 中断、时钟中断|
|异常|当前指令执行引起的事件|除零、越界、非法指令、缺页|
|系统异常 / 陷入|执行陷入指令请求 OS 服务|read、write、fork|
|中断源|引起中断的来源|设备、时钟、程序错误、系统调用|
|中断响应|硬件完成的响应动作|保存 PC/PSW，跳转入口|
|中断处理|OS 软件完成的处理逻辑|查向量表、执行处理程序|
|中断控制器|接收和管理外设中断请求的硬件|IRQ 控制|
|中断向量表|保存中断处理程序入口地址的表|中断号 → handler|
|中断屏蔽|暂时不响应某些中断|屏蔽低优先级设备中断|
|中断优先级|多个中断时决定响应顺序|硬件故障高于普通 I/O|
|中断嵌套|处理中断时响应另一个中断|高优先级中断打断低优先级中断|
