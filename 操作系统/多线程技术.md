[[处理器管理]]
#线程管理
## 传统单线程进程

在早期操作系统或传统进程模型中，一个进程通常只有一条执行路径，也就是 单线程结构进程
包括：
```
进程控制块 PCB
用户地址空间
用户栈
核心栈
一条执行序列
```
也就是说，一个进程既是 **资源分配单位** ，又是 **CPU调度执行单位**

### 单线程结构的问题

```
进程切换开销大
进程通信开销大
限制进程并发粒度
降低并行计算效率
```

**进程切换开销大**

进程切换需要保存和恢复大量上下文，包括：
```
PC 
PSW
通用寄存器
栈指针
PCB 状态
地址空间信息
页表 / 内存映射信息
打开文件表相关信息
```
且不同进程通常有不同地址空间，切换时可能影响TLB、Cache等硬件状态
因此进程切换相对昂贵

**进程通信开销大**

不同进程之间地址空间隔离，一个进程不能直接访问另一个进程的内存
这带来了安全性，但是也让通信更复杂

**并发粒度太粗**

如果只能以进程为单位开发，那么并发粒度就比较大
例如，一个浏览器如果每个小任务都创建进程，代价很高
但很多任务本来属于同一个应用内部，例如：
```
界面响应，网络下载，页面渲染，磁盘缓存，音视频播放
```
这些任务共享同一个应用资源，用进程划分太重，用线程更合适

**进程内不能并发**

把进程的两项功能分离开：
进程：作为资源分配和保护的独立单位
线程：作为调度和分派的基本单位

进程作为系统资源分配和保护的独立单位，不需要频繁切换
线程作为系统调度和分派的基本单位，可以轻装运行、频繁调度和切换
线程的出现减少了并发执行的时空开销，使并发粒度更细、并发性更好

## 多线程环境下的进程和线程

### 多线程结构进程
多线程结构下，一个进程可以包含多个线程
这些线程共享进程的资源
但是每个线程又有自己独立的执行状态

> 一个进程中有共享的用户地址空间和进程控制块，同时每个线程有自己的用户栈、内核栈、线程控制块和执行序列

### 多线程环境下的进程概念
在多线程环境中，进程不再主要表示”一条执行流“，而是表示：
**资源分配、资源保护和地址空间的单位**

多线程环境中，进程是操作系统中进行保护和资源分配的独立单位，具有容纳进程映像的虚拟地址空间，以及对进程、文件和设备的存取保护机制

### 多线程环境下的线程概念
线程是进程的一条执行路径，是调度的基本单位。处理器调度最终落在线程一级。同一个进程中的所有线程共享进程获得的主存空间和资源

线程具有：
```
线程执行状态
受保护的线程上下文
独立的程序指令计数器
执行堆栈
容纳局部变量的静态存储器
```

线程虽然共享进程资源，但它不是“没有自己状态的”。每个线程必须有自己的执行现场，否则多个线程无法中不同位置同时执行

每个线程需要有自己的栈和PC，用于记录局部变量、返回地址、函数调用信息和自己下一条要执行的指令

## 线程状态和线程调度

### 线程状态
线程状态包括：
运行、就绪、睡眠

**线程无挂起状态，挂起状态与资源相关，属于进程**
挂起通常意味着进程被换出内存、资源被剥夺、不参与调度。这个动作针对的是整个进程的地址空间和资源集合，而不是单个线程
因为同一进程内的线程共享地址空间，如果只把某一个线程的地址空间换出是不合理的，因为地址空间是进程级资源

|线程状态|含义|
|---|---|
|运行|当前线程正在 CPU 上执行|
|就绪|线程具备运行条件，等待 CPU|
|睡眠 / 阻塞|线程等待某个事件，暂时不能执行|

### 线程操作

|操作|含义|
|---|---|
|孵化|创建线程|
|封锁|线程因等待事件进入睡眠 / 阻塞|
|活化|等待事件完成，线程变为可运行|
|剥夺|当前线程被抢占|
|指派|调度程序选择某线程运行|
|结束|线程执行完成或被终止|

### OS 感知线程与 OS 不感知线程

**OS 感知线程环境**

如果操作系统知道线程的存在，那么处理器调度对象就是线程

OS 感知线程环境下，处理器调度对象是线程，进程没有三状态，或者说只有挂起状态

**OS 不感知线程环境**

如果操作系统不知道线程存在，那么 OS 调度对象仍然是进程

用户空间中的线程库负责在线程之间调度

OS 不感知线程环境下，处理器调度对象仍是进程，用户空间中的用户调度程序调度线程

## 多线程程序设计的优点

### 快速线程切换
统一进程内线程共享地址空间，切换时通常不需要切换整个进程资源环境

相比进程切换，线程切换通常少了：
```
地址空间切换
大量资源表切换
进程级保护结构切换
```
所以更轻量

### 减少系统管理开销
创建进程需要创建完整地址空间、资源表、PCB 等

创建线程通常只需要创建
```
线程控制块 TCB
线程栈
线程寄存器上下文
调度信息
```
因此线程创建和销毁开销更小

### 线程通信更容易
同一进程内线程共享内存，通信可以直接通过共享变量、队列、缓冲区等实现

```
shared_queue.push(task);
```
但也因此必须注意同步问题，否则会发生竞态事件

### 并行程度提高

在多核处理器上，多个线程可以同时在不同CPU核心上执行
可以显著提高计算密集型任务的吞吐量

### 节省内存空间
多个线程共享进程地址空间，不需要每个执行流都复制一套完整资源
相比创建多个进程，多个线程通常更节省内存

### 多线程技术的应用

### 前台和后台工作

```
前台线程：响应用户输入
后台线程：自动保存、拼写检查、索引构建
```

### C/S应用模式

服务器可以使用多个线程处理多个客户端请求

```
主线程监听连接
工作线程处理请求
日志线程记录访问
缓存线程维护缓存
```

### 加快执行速度

计算任务可以拆分给多个线程并行执行

```
图像处理
矩阵计算
视频编码
文件压缩
机器学习数据预处理
```

### 设计用户接口
GUI程序通常需要保持界面线程及时响应，同时把耗时任务放到后台线程

```
主线程：刷新界面
后台线程：下载文件
```

## KLT：内核级线程

### KLT Kernel-Level Threads 内核级线程
由操作系统内核直接管理和调度的线程

线程管理的所有工作由 OS 内核来做
OS 提供 API 供开发者使用 KLT
OS 直接调度 KLT

内核知道每个线程的存在。每个线程在内核中都有相应的数据结构，例如TCB 或线程描述符

### KLT 的工作方式
KLT模型中，线程操作通常需要内核参与

```
创建线程
销毁线程
线程阻塞
线程唤醒
线程调度
线程同步中的某些操作
```

例如一个 进程P 有三个线程：
```
P ： 
	T1
	T2
	T3
```
内核调度器可以直接调度 T1，T2, T3

### KLT的优点

**一个线程阻塞，不会阻塞整个进程**

**支持多处理器并行**
在多核CPU上，内核可以把同一进程的多个KLT调度到不同核心上执行

**内核自身也可以多线程化**
操作系统内核内部也可以使用多个内核线程完成不同任务

### KLT的缺点
应用程序线程中用户态运行，但线程调度和管理在内核实现；统一进程中控制权从一个线程传给另一个线程是，需要模式切换，系统开销较大

## ULT：用户级线程

**ULT User-Level Threads 用户级线程**
由用户空间线程库管理的线程

ULT是运行中用户空间的线程库，提供多线程应用程序的开发和运行支撑环境；
线程管理的所有工作都有应用程序完成，内核没有意识到线程的存在

```
OS 调度进程 P
⬇️
P 的内部线程库调度 U1 / U2 / U3
```

### ULT的优点
线程管理数据结构在用户空间，线程切换不需要内核模式，节省模式切换开销和内核资源；
允许应用按需求选择调度算法；
能运行在任何OS上，内核不需要做支持工作

**线程切换快**
因为ULT切换只在用户空间完成，不需要陷入内核

**调度策略灵活**
应用程序可以根据自身特点设计线程调度算法

**可移植性好**
由于内核不知道ULT的存在，理论上只要线程库能实现，就可以做不支持内核线程的系统上运行

### ULT的缺点

**不能充分利用多处理器**
因为OS调度的是进程，不是用户级线程
如果一个进程只有一个内核调度实体，那么及时进程内部有很多ULT，内核也只能把这个进程作为一个整体放到一个CPU上运行

不能实现真正多核并行

**一个线程阻塞，整个进程阻塞**
如果ULT执行了阻塞式系统调用，内核不知道这是某个ULT在阻塞，它只知道整个进程P阻塞了，因此整个进程都会被挂起，进程内其他ULT也不能运行

## Jacketing 技术

基本思想：把阻塞式系统调用改造成非阻塞式
当线程陷入系统调用时，执行 jacketing 程序；由 jacketing 程序检查资源使用情况，以决定是否执行进程切换或把控制权传递给另一个进程

### 大致流程

例如原本用户线程要执行阻塞式 read：

```
read(fd, buf, size);
```

经过 Jacketing 后，可能变成：

```
1. 先检查 fd 是否已有数据可读2. 如果不可读，不真正执行阻塞 read3. 把当前 ULT 标记为等待4. 切换到另一个 ULT5. 等数据准备好后，再恢复该 ULT
```

这样就避免整个进程被内核阻塞。

### 局限性
如果系统调用无法非阻塞化，或者内核机制不支持异步通知，那么仍然可能遇到阻塞问题

因此现代系统更常用KLT 或混合线程模型

## KLT 与 ULT 对比
ULT适用于解决逻辑并行性问题
KLT适用于解决物理并行性问题

**逻辑并行性**
逻辑并行性指程序结构上有多个任务可以交替推进，但不一定真的在多个CPU核心上同时执行

任务可以通过用户级线程或协程中CPU交替执行
ULT切换快，调度灵活，适合这种场景

**物理并行性**
指多个任务同时真的同时中多个CPU核心上运行
核心 1 执行线程A
核心 2 执行线程B
核心 3 执行线程C

KLT 更适合这种场景，因为内核知道每个线程，可以把她们分派到不同处理器上

## 多线程实现的混合策略

**结合KLT和ULT的优点，同时减少它们的缺点**

多线程实现的混合策略组合用户级线程和内核级线程设施。线程创建完全在用户空间完成， 线程调度和同步也中应用程序中进行；一个应用中的多个用户级线程映射到一个内核级线程上；程序员可以根据应用和机器调节内核级线程数量，以获得整体最佳效果

### 混合策略的基本结构

```
多个 ULT ➡️ 映射到 若干 KLT ➡️ 被 OS 调度到 CPU 上
```

```
用户空间：
  U1 U2 U3 U4 U5 U6

映射到：

内核空间：
  K1 K2 K3

CPU 调度：
  K1 → CPU0
  K2 → CPU1
  K3 → CPU2
```

ULT 数量可以很多 使得调度保持轻量和灵活
KLT 数量可以较少 其数量决定物理并行能力

### 混合策略的特点
- 组合用户级线程 / 内核级线程设施
- 线程创建完全在用户空间完成
- 线程调度和同步也在应用程序中进行
- 多个 ULT 映射到一些 KLT 上
- KLT 数目小于等于 ULT 数目
- 程序员可调节 KLT 数目以获得最佳结果
- 结合 ULT 与 KLT 的优点，减少缺点

### 混合策略解决了什么问题

**解决 ULT 的多核问题**
多个ULT可以映射到多个KLT，而KLT能被内核调度到多个处理器
混合模型可以实现物理并行

**缓解 ULT 的阻塞问题**
如果某个ULT绑定的KLT阻塞，其他KLT仍可能继续运行，其他ULT仍有机会执行

**缓解 KLT 到物理开销**
不需要每个用户级人物都对应一个内核线程
可以减少内核管理压力


### 混合策略下的线程状态

KLT 三态：由系统调度负责
ULT 三态：由用户调度负责
活跃态 ULT 代表绑定 KLT 的三态
活跃态 ULT 运行时可激活用户调度
非阻塞系统调用可使用 Jacketing 启动用户调度，调整活跃态 ULT

**KLT状态由OS管理**
KLT 是内核可见线程，因此它的状态由内核调度器管理
```
Running 运行态
Runnable 可运行态
Blocked 阻塞态
```

**活跃态ULT**
当前绑定到某个KLT、具备实际执行机会的ULT

不是所有ULT都同时绑定KLT。混合模型中，用户线程库会选择一部分ULT作为活跃线程，并映射到KLT上

## 三种线程模型总结

| 模型   | 映射关系             | 优点          | 缺点           |
| ---- | ---------------- | ----------- | ------------ |
| KLT  | 一个用户线程通常对应一个内核线程 | 支持阻塞隔离和多核并行 | 模式切换和内核管理开销大 |
| ULT  | 多个用户线程对应一个内核调度实体 | 切换快，灵活，可移植  | 阻塞问题，难以多核并行  |
| 混合模型 | 多个 ULT 映射到若干 KLT | 兼顾轻量调度和多核并行 | 实现复杂，调度协调困难  |

## 进程和线程的区别
|对比项|进程|线程|
|---|---|---|
|基本定位|资源分配和保护单位|调度和执行单位|
|地址空间|进程之间通常独立|同一进程内线程共享|
|拥有资源|拥有内存、文件、设备等资源|通常不独立拥有大量资源|
|切换开销|大|小|
|通信方式|IPC，开销较大|共享内存，通信方便|
|安全隔离|强|弱|
|崩溃影响|一个进程崩溃通常不影响其他进程|一个线程崩溃可能导致整个进程崩溃|
|创建销毁|开销较大|开销较小|
|并发粒度|粗|细|

## 多线程的问题

**竞态条件**
多个线程同时访问共享变量，如果没有同步，结果可能不确定

例如
```c
count ++;
```
底层：
```
读取count
加1
写会count
```
两个线程同时执行就可能丢失更新

**死锁**
多个线程互相等待对方持有的锁

**活锁**
线程没有阻塞，但一直在让步活重试，无法真正推进

**饥饿**
某些线程长期得不到 CPU 或锁资源

**调试困难**
多线程程序执行顺序不确定，Bug可能偶发，很难复现

## 总结

> **多线程技术的本质，是把传统进程中“资源分配”和“执行调度”两种功能分离：进程继续作为资源和保护单位，线程成为轻量级执行和调度单位。KLT 由内核管理，适合物理并行；ULT 由用户线程库管理，适合逻辑并发；混合线程模型把多个 ULT 映射到若干 KLT 上，试图同时获得用户级线程的轻量灵活和内核级线程的多处理器并行能力。**
