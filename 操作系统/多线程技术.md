## 传统单线程进程

在早期操作系统或传统进程模型中，一个进程通常只有一条执行路径，也就是 单线程结构进程
包括：
```
进程控制块 PCB
用户地址空间
用户栈
核心栈
一条执行序列
```
也就是说，一个进程既是 **资源分配单位** ，又是 **CPU调度执行单位**

### 单线程结构的问题

```
进程切换开销大
进程通信开销大
限制进程并发粒度
降低并行计算效率
```

**进程切换开销大**

进程切换需要保存和恢复大量上下文，包括：
```
PC 
PSW
通用寄存器
栈指针
PCB 状态
地址空间信息
页表 / 内存映射信息
打开文件表相关信息
```
且不同进程通常有不同地址空间，切换时可能影响TLB、Cache等硬件状态
因此进程切换相对昂贵

**进程通信开销大**

不同进程之间地址空间隔离，一个进程不能直接访问另一个进程的内存
这带来了安全性，但是也让通信更复杂

**并发粒度太粗**

如果只能以进程为单位开发，那么并发粒度就比较大
例如，一个浏览器如果每个小任务都创建进程，代价很高
但很多任务本来属于同一个应用内部，例如：
```
界面响应，网络下载，页面渲染，磁盘缓存，音视频播放
```
这些任务共享同一个应用资源，用进程划分太重，用线程更合适

**进程内不能并发**

把进程的两项功能分离开：
进程：作为资源分配和保护的独立单位
线程：作为调度和分派的基本单位

进程作为系统资源分配和保护的独立单位，不需要频繁切换
线程作为系统调度和分派的基本单位，可以轻装运行、频繁调度和切换
线程的出现减少了并发执行的时空开销，使并发粒度更细、并发性更好

## 多线程环境下的进程和线程

### 多线程结构进程
多线程结构下，一个进程可以包含多个线程
这些线程共享进程的资源
但是每个线程又有自己独立的执行状态

> 一个进程中有共享的用户地址空间和进程控制块，同时每个线程有自己的用户栈、内核栈、线程控制块和执行序列

### 多线程环境下的进程概念
在多线程环境中，进程不再主要表示”一条执行流“，而是表示：
**资源分配、资源保护和地址空间的单位**

多线程环境中，进程是操作系统中进行保护和资源分配的独立单位，具有容纳进程映像的虚拟地址空间，以及对进程、文件和设备的存取保护机制

### 多线程环境下的线程概念
线程是进程的一条执行路径，是调度的基本单位。处理器调度最终落在线程一级。同一个进程中的所有线程共享进程获得的主存空间和资源

线程具有：
```
线程执行状态
受保护的线程上下文
独立的程序指令计数器
执行堆栈
容纳局部变量的静态存储器
```

线程虽然共享进程资源，但它不是“没有自己状态的”。每个线程必须有自己的执行现场，否则多个线程无法中不同位置同时执行

每个线程需要有自己的栈和PC，用于记录局部变量、返回地址、函数调用信息和自己下一条要执行的指令

## 线程状态和线程调度

### 线程状态
线程状态包括：
运行、就绪、睡眠

**线程无挂起状态，挂起状态与资源相关，属于进程**
挂起通常意味着进程被换出内存、资源被剥夺、不参与调度。这个动作针对的是整个进程的地址空间和资源集合，而不是单个线程

|线程状态|含义|
|---|---|
|运行|当前线程正在 CPU 上执行|
|就绪|线程具备运行条件，等待 CPU|
|睡眠 / 阻塞|线程等待某个事件，暂时不能执行|
