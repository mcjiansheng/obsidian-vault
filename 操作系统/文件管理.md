[[操作系统]]
> **文件是操作系统提供给用户和程序的数据抽象；文件系统是操作系统中负责管理这些文件的模块。**

## 文件的概念
**为什么操作系统要引入“文件”？**
如果没有文件，用户要保存一段数据，就必须知道它放在磁盘的哪个柱面、哪个磁道、哪个扇区，甚至要自己考虑磁盘块如何分配、如何回收、如何防止别人误删。这显然不现实。

所以操作系统引入文件，把底层复杂的辅助存储器细节隐藏起来。

**文件系统的首要需求是按名存取**

### 文件的定义
文件是具有符号名的、在逻辑上具有完整意义的一组相关信息项的序列。

**具有符号名**
文件不是靠物理地址被用户识别，而是靠**文件名**识别。

文件名是用户和程序访问文件的入口。没有名字，用户就无法方便地引用文件。

**逻辑上具有完整意义**
文件不是任意堆放的一串二进制位，而是对用户或程序有意义的数据集合。

也就是说，文件是站在**用户视角和程序视角**看待数据的单位。

**一组相关信息项**
文件内部的数据项彼此相关。例如学生成绩文件中，每条记录可能包含学号、姓名、课程、成绩；源程序文件中，每一行代码都属于同一个程序；图片文件中，每部分二进制数据共同构成一张图像。

**序列**
文件中的信息通常具有某种顺序。比如文本文件按字节顺序排列，记录式文件按记录顺序排列，可执行文件按特定格式组织代码段、数据段、符号表等内容。

所以，文件既不是纯粹的“字节堆”，也不是纯粹的“磁盘块集合”，而是**用户逻辑意义和物理存储结构之间的桥梁**。

### 文件 document 与计算机文件 file 的区别
普通文件更偏向现实世界中的文档、档案、材料，例如纸质申请表、合同、论文。

计算机文件则是操作系统管理的数据对象。它可以是文本文档，也可以是程序、图片、音频、视频、数据库文件、设备文件，甚至在 Unix/Linux 中，很多设备也可以被抽象为文件。

### 文件的命名
文件名通常由**字母、数字和其他符号组成**，具体格式和长度因操作系统而异。文件命名一般包括**文件名和扩展名**
`文件名.扩展名`
#### 文件名的作用
文件名主要用于**识别文件**。
#### 扩展名的作用
扩展名通常用于表示文件类型或文件特性。

现代系统中，扩展名更多是**约定**而不是绝对依据。

## 引入文件的优点
### 1. 用户使用方便：实现按名存取
用户无需知道数据在辅助存储器中的物理位置，也无需关心数据如何存放到磁盘上，只需要知道文件名并发出操作请求即可。
### 2. 文件安全可靠
用户不能直接随意操作磁盘物理块，而是必须通过文件系统访问文件。文件系统可以提供：
```
权限控制
访问检查
文件保护
数据备份
错误恢复
```
这样可以防止用户有意或无意地破坏文件，也可以防止非法窃取文件内容。
### 文件可备份、可恢复
文件系统可以组织转储或备份，在硬件故障时提高可靠性

### 文件可共享
文件系统可以允许不同用户使用同一个文件，可以是同名共享，也可以是异名共享。

共享可以提高文件和存储空间的利用率。

## 文件系统及其功能
文件系统是操作系统中负责存取和管理信息的模块。它用统一的方式管理用户和系统信息的存储、检索、更新、共享和保护，并为用户提供一整套方便有效的文件使用和操作方法。

### 文件系统中的“文件”有两个视角
同一个文件必须从两个侧面观察：**逻辑文件**和**物理文件**。

**逻辑文件**
逻辑文件是用户看到的文件。

**物理文件**
物理文件是文件在辅助存储器上的实际存放形式。

### 文件系统面向用户的功能

**文件的按名存取**
这是文件系统最基本的功能。
用户通过文件名或路径名访问文件

文件系统需要完成：
```
解析路径 /home/user/a.txt
查找根目录 /
进入 home 目录
进入 user 目录
找到 a.txt 对应的目录项
找到文件控制块或 inode
定位文件数据块
完成读写
```

用户只看到文件名，系统内部则需要进行目录检索、元数据读取、权限检查和物理块定位。
文件名 / 路径名
        ↓
目录项
        ↓
FCB / inode
        ↓
物理块地址
        ↓
磁盘 I/O

### 文件的共享和保护
文件共享是指多个用户或多个进程可以访问同一个文件。
但共享必须伴随保护。否则，多个用户都能随意修改文件，会导致数据破坏。

常见保护机制包括：
```
访问控制矩阵
访问控制表 ACL
用户/组/其他用户权限
读/写/执行权限位
文件属性
加密
备份和恢复
```

### 文件的操作和使用
文件系统要为用户和程序提供一组操作接口。
用户程序看到的是简单接口，文件系统内部却要完成目录查找、权限验证、缓冲区管理、物理块读取等复杂工作。

### 为了实现这些功能，操作系统必须考虑什么？
#### 1. 文件目录的建立和维护
目录的作用是建立：
`文件名 → 文件元数据 / 文件位置`
的映射关系。
#### 2. 存储空间的分配和回收
文件创建或扩展时，文件系统要为它分配磁盘空间；文件删除或截断时，文件系统要回收空间。

#### 3. 数据的保密和保护
保密强调防止未授权读取，保护强调防止破坏、误改、误删。

#### 4. 监督用户存取和修改文件的权限
每次文件访问都需要检查权限。

#### 5. 处理不同存储介质的差异
文件可能存储在磁盘、SSD、光盘、磁带、网络存储等介质上。
文件系统要在不同介质上统一提供类似的文件接口。

# 文件的组织
**文件组织到底在解决什么问题**？
用户看到的是“有名字、有意义的数据对象”，比如 `student.dat`、`main.c`、`report.pdf`；  
磁盘看到的是“一个个固定大小或近似固定大小的物理块”。

```
用户看到的文件内容
    ↓
逻辑结构：字节流 / 记录流
    ↓
成组与分解：逻辑记录如何装入物理块
    ↓
物理结构：文件块在磁盘上如何排列
    ↓
真正的磁盘块 / 扇区 / 簇
```

## 文件的存储
### 文件存储介质：磁带、光盘、磁盘
文件存储介质主要有 `磁带、光盘、磁盘`
其中磁带和光盘更偏向顺序存取，磁盘是典型的直接存取或随机存取设备。

**卷**和**块**
卷是存储介质的物理单位，例如一盘磁带、一张光盘、一个硬盘分区；
块是存储介质上一段连续信息组成的区域，也叫物理记录，并且块是主存和辅存之间交换信息的物理单位

|概念|含义|例子|
|---|---|---|
|存储介质|保存文件的物理设备或材料|磁盘、磁带、光盘、SSD|
|卷|文件系统管理的存储介质单位|一个硬盘分区、一个光盘片|
|块|主存和辅存交换数据的基本物理单位|4KB 块、8KB 块|
|物理记录|存储介质上一块连续信息|通常可以理解为块|

### 块大小为什么重要？
决定块大小时要考虑用户使用方式、数据传输效率、存储设备类型等因素；相邻块之间还可能因为机械动作或块识别需要留有间隙，间隙是不记录用户代码信息的区域。

块大小是文件系统设计中的典型“时间—空间折中”。

**块大一点的好处**
```
一次 I/O 传输的数据更多
顺序读写吞吐量更高
文件块索引数量减少
元数据开销减少
磁盘寻道和旋转延迟被摊薄
```
适合大文件、视频文件、备份文件、日志归档文件。

**块大一点的问题**
```
小文件浪费空间
随机访问时可能读入很多无用数据
缓存污染可能更明显
```

**块小一点的好处**
```
小文件空间浪费较少
随机访问粒度更细
```

**块小一点的问题**
```
大文件需要更多块
索引表更大
I/O 次数可能增多
管理开销更高
```
所以块大小不是越大越好，也不是越小越好，而要根据文件系统目标和设备特性折中。

### 顺序存取存储设备的信息安排
> 顺序存取设备是严格依赖信息的物理位置次序进行定位和读写的存储设备。

磁带机是典型顺序存取设备，优点是容量大、可靠、卷可装卸、便于保存，常用于归档。磁带的块长变化范围较大，块可以很小，也可以很大。
光盘也可作为顺序存取存储设备来理解。

顺序存取设备的特点是：
```
读第 n 个块之前，往往要经过前面的块
适合批量顺序读写
不适合频繁随机访问
```

典型场景
```
备份
归档
批处理输入文件
顺序扫描的大型数据文件
```

从性能角度看，顺序存取设备强调“按物理顺序组织信息”。如果你频繁跳来跳去访问，会非常慢。

### 直接存取存储设备的信息安排
磁盘是一种直接存取存储设备，也叫随机存取存储设备；它通过移臂和旋转这二维组织实现较高存取速度，并且每个物理记录有确定位置和唯一地址，访问任意物理块的时间几乎不依赖于该信息的位置。

这里“几乎不依赖”要理解为操作系统教材中的抽象说法。实际磁盘仍然有：
```
寻道时间
旋转延迟
传输时间
```

所以磁盘随机访问比顺序访问慢很多，但比磁带随机定位强得多。对于 SSD 来说，没有机械寻道和旋转延迟，随机访问性能进一步提升，但仍然存在页、块、擦除、写放大等底层约束。

直接存取设备的意义是：文件系统可以更灵活地选择文件物理结构，比如连续分配、链接分配、索引分配、哈希直接定位等。

## 文件的逻辑结构

**什么是逻辑文件**？
逻辑文件也称为文件的逻辑结构，它是独立于物理环境的、用户概念中的抽象信息组织方式，是用户能观察并处理的数据集合。文件逻辑结构主要分为两种：**流式文件**和**记录式文件**。

注意，逻辑结构不关心文件在磁盘上连续还是分散，也不关心数据块号是多少。那些是物理结构的问题。

### 流式文件
> 流式文件指文件内的数据不再组成记录，只是由一串依次的字节组成的信息流序列；这种文件常常按长度读取所需信息，也可以用插入的特殊字符作为分界。

`文件 = byte[0], byte[1], byte[2], ..., byte[n-1]`
用户程序通过“偏移量 + 长度”访问文件。

**流式文件的特点**

|方面|特点|
|---|---|
|结构|字节流|
|访问方式|按字节偏移和长度访问|
|灵活性|很强|
|操作系统负担|较小，不需要理解记录格式|
|典型系统|Unix/Linux 普通文件|
|典型文件|`.txt`、`.c`、`.jpg`、`.mp4`、`.bin`|
很多现代操作系统倾向于把普通文件看成字节流。至于字节流里面是不是 JSON、CSV、图片、音频、数据库页，那通常由应用程序解释。

### 记录式文件
> 记录式文件是一种有结构的文件，它是若干逻辑记录信息组成的记录流文件；逻辑记录是文件中按信息在逻辑上的独立含义所划分的信息单位。

例如工资文件：
```
记录1：00001 张三 计算机系 76.80 ...
记录2：00002 李四 计算机系 105.00 ...
记录3：00003 王五 物理系 123.60 ...
```
记录式文件可以理解为：
文件 = 记录1 + 记录2 + 记录3 + ... + 记录n
记录 = 数据项1 + 数据项2 + ... + 数据项m

#### 固定长度记录与可变长度记录
记录式文件还可以继续细分。

**固定长度记录**
每条记录长度相同。

优点是定位简单：
```
第 i 条记录的偏移 = 文件起始地址 + i × 记录长度
```
固定长度记录适合直接定位，但可能浪费空间，因为不同记录实际内容长度可能不同。

**可变长度记录**
每条记录长度不一定相同。

优点是节省空间，适合文本、日志、复杂对象；缺点是定位困难，因为要知道前面所有记录长度，或者额外建立索引。

**记录式文件和数据库的区别**
数据库管理系统也支持逻辑记录，但数据库不同于记录式文件。数据库中的记录之间可以通过数据冗余构成某种联系，数据库管理系统支持基于联系的数据查询，而文件系统不支持这种查询。

## 记录的成组与分解
一个物理记录只存放一个逻辑记录可能造成极大浪费；若干个逻辑记录合并成一组，写入一个块，叫记录的成组；每块中的逻辑记录数称为块因子。

逻辑记录：用户意义上的一条记录
物理记录：存储介质上的一个块

如果一个逻辑记录很小，而一个物理块很大，那么“一块只放一条记录”会浪费大量空间。


### 记录成组是什么？
多个逻辑记录 → 合并 → 一个物理块

对于流式文件，指出一个物理记录可以存放很多个连续字节。
### 记录分解是什么？
当存储介质上的一个物理记录读进输入缓冲区后，把逻辑记录从块中分离出来的操作叫记录的分解操作

一个物理块 → 拆出 → 多条逻辑记录

这说明用户请求一条记录时，系统底层可能会把包含它的整个物理块读入内存，然后从中取出目标逻辑记录。

### 成组与分解在哪里完成？
系统设置独立于用户数据区的输入/输出缓冲区；记录成组在输出缓冲区内进行，凑满一块后才写到存储介质；读入时，一个物理记录先进入输入缓冲区，再从块中分离出逻辑记录。

### 非跨块记录与跨块记录
两种可变长度记录成组方式：**可变长度非跨越式组块**和**可变长度跨越式组块**

**可变长度非跨越式组块**
记录长度可变
但一条记录不能跨越两个物理块
如果当前块剩余空间放不下下一条记录，剩余空间就浪费

**优点**
记录读取简单
一条记录一定完整存在于一个块中

**缺点**
块内可能出现未使用空间
空间利用率下降

**可变长度跨越式组块**
记录长度可变
允许一条记录跨越两个或多个物理块
块中尽量不留空闲空间
跨块部分通过指针或描述信息连接

**优点**
空间利用率高
减少块内浪费

**缺点**
记录读取复杂
一条记录可能需要读多个块
需要维护跨块连接信息
如果某个块损坏，恢复更麻烦

### 成组与分解带来的新特征：提前读与推迟写
用户读请求会导致包含目标逻辑记录的整个物理块被读入输入缓冲区，这可能同时读入多个逻辑记录，称为提前读；用户写请求首先写入输出缓冲区，只有缓冲区中的逻辑记录满后才会实际输出，称为推迟写。

#### 提前读 read-ahead
用户只想读 R3，但系统读入整个块：
好处是，如果用户接下来读取 R4，就不用再访问磁盘。

#### 推迟写 delayed write
用户写入 R1、R2、R3 时，系统可能先放在输出缓冲区，不立即写盘：
好处是减少 I/O 次数，提高性能。
风险是：如果系统崩溃，缓冲区中尚未落盘的数据可能丢失。

## 文件的物理结构
文件的物理结构和组织是指文件在物理存储空间中的存放方法和组织关系，又称物理文件。文件的存储结构涉及块的划分、记录的排列、索引的组织、信息的搜索等问题，其优劣直接影响文件系统性能。

物理结构直接影响：
顺序访问速度
随机访问速度
空间利用率
文件扩展是否方便
插入删除是否方便
元数据开销
可靠性

### 顺序文件，也叫连续文件
将一个文件中逻辑上连续的信息存放到存储介质的依次相邻的块中，便形成顺序结构，这类文件叫顺序文件，又称连续文件。磁带文件、光盘文件是典型例子。

**优点**
顺序文件在顺序存取记录时速度较快，批处理文件、系统文件用得最多；如果用磁带存放顺序文件，总能保持快速存取的优点。

**缺点**
建立文件前需要预先确定文件长度，以便分配存储空间；修改、插入、增加文件记录比较困难。

### 连接文件，也叫串联文件
连接结构使用连接字表示文件中各个物理块之间的先后次序。第一块文件信息的物理地址由文件目录给出，每一块的连接字指出文件的下一个物理块位置；连接字为 0 时表示文件到本块结束。

```
目录项：
文件 A 首块 = 9

磁盘块：
9  → 16 → 3 → 25 → 0
A0   A1   A2  A3   结束
```
每个块除了保存数据，还保存一个指针：
```
当前块数据 + 下一块地址
```

**优点**
文件扩展方便
插入删除相对方便
外部碎片问题小
空闲块利用率高
适合动态增长文件

**缺点**
随机访问差
连接字占用空间
指针损坏可能导致后续文件块丢失
不利于磁盘预读和连续传输
磁头可能频繁移动

### 直接文件
直接文件也常被称为散列文件或哈希文件。它的思想是：
根据记录关键字直接计算出记录所在的物理地址

**优点**
按关键字访问速度很快
平均情况下接近 O(1)
适合需要快速查找单条记录的场景

**问题**
冲突
不适合顺序遍历
不适合范围查询
哈希函数设计困难
文件扩展可能导致重散列
空间利用率可能不稳定

### 索引文件
索引文件在文件存储器上分为两个区：
```
索引区
数据区
```
访问索引文件需要两步：第一步查找索引表，第二步获得记录物理地址；如果文件索引已经预先调入主存，可以减少一次内外存信息交换。

```
索引表：
逻辑块0 → 物理块 12
逻辑块1 → 物理块 98
逻辑块2 → 物理块 31
逻辑块3 → 物理块 77

数据区：
12 保存 A0
98 保存 A1
31 保存 A2
77 保存 A3
```

**优点**
索引结构可以看作连接结构的扩展；它具备连接文件的优点，同时克服了连接文件只能顺序存取的缺点，能够直接读写任意记录，便于文件增、删、改。

**缺点**
增加了索引表的空间开销和查找时间。
索引块本身占空间
小文件也可能需要索引结构
访问数据前可能要先访问索引
多级索引会增加寻址次数
索引损坏可能影响整个文件访问

### 一级索引、两级索引、多级索引
```
一级索引块
 ├─ 二级索引块A
 │    ├─ 数据块地址
 │    ├─ 数据块地址
 │    └─ ...
 ├─ 二级索引块B
 │    ├─ 数据块地址
 │    └─ ...
 
 一级索引 → 二级索引 → 三级索引 → 数据块
```
优点是可以支持非常大的文件；缺点是访问深层数据块时需要更多次索引访问。

**假设**
```
块大小 = 1KB
块地址 = 4B
一个索引块可存 256 个地址

一级间接索引可索引 256 个数据块
二级间接索引可索引 256 × 256 个数据块
三级间接索引可索引 256 × 256 × 256 个数据块
```

# 文件目录
**如何根据文件名或路径名，找到文件的真实存储位置和管理信息？**

文件目录是实现文件“按名存取”的关键数据结构；文件系统的基本功能之一就是负责文件目录的建立、维护和检索，并且目录需要永久保存，所以目录本身也被组织成文件存放在磁盘上，称为**目录文件**

## 文件目录、目录项、FCB、inode 的关系
### 文件目录
**文件目录**是由若干个目录项组成的数据结构。每个目录项描述一个文件或一个子目录。

目录的本质不是用户界面上的“文件夹图标”，而是文件系统内部的一种数据结构。它的作用是建立映射：
文件名 → 文件控制信息

### 目录项
**目录项**是目录文件中的一个登记项。它至少要能回答两个问题：
这个名字对应哪个文件？
这个文件的控制信息在哪里？

在简单文件系统中，目录项可能直接保存完整的 FCB：
```
文件名 + 文件类型 + 文件大小 + 文件物理位置 + 权限 + 时间信息 ...
```

在 Unix/Linux 风格中，目录项通常更简洁：
```
文件名 + inode 号
```

这样设计的好处是：目录查找时主要比较文件名，找到后拿到 inode 号，再通过 inode 号去 inode 区读取完整文件信息。目录项变小后，目录检索速度会更快；同一个 inode 也可以被多个目录项引用，从而支持硬链接共享。

### FCB
**FCB** 是 File Control Block，中文通常叫**文件控制块**。它是文件系统用来描述和控制一个文件的数据结构。

FCB 对文件的作用类似 PCB 对进程的作用，它集中保存文件的相关信息；找到 inode，就能获得文件的必要信息。

一个 FCB 通常包含三类信息。

**基本信息**
用于标识文件和描述文件本身

**存取控制信息**
用于控制谁能访问文件、以什么方式访问文件

**使用信息 / 存储信息**
用于文件系统实际定位文件：
```
文件所在设备
文件起始块号
文件数据块索引
直接索引
一级间接索引
二级间接索引
三级间接索引
文件当前打开次数
引用计数
```

### inode 
**inode** 是 Unix/Linux 文件系统中的索引节点，本质上可以看作一种“去掉文件名的 FCB”。
每个文件都有一个磁盘 inode 与之对应，inode 集中存放在磁盘 inode 区；inode 记录文件属性和相关信息，文件访问过程中会频繁用到它，所以系统会在内存中开辟活动 inode 表。磁盘 inode 反映文件静态特性，活动 inode 反映文件动态特性。
```
目录项：文件名 → inode 号
inode：文件属性 + 数据块索引
数据块：真正的文件内容
```

## 文件目录结构
> **系统中有大量文件和用户时，目录应该怎样组织？**

### 一级目录结构
**一级目录结构**是在整个操作系统中构造一张线性表，每个文件占用一个目录项，所有文件都登记在同一张目录表中。
一级目录结构就是将每个文件的相关属性占用一个目录项，构成一张线性目录表；它的问题是用户和文件众多时容易重名，也不利于记忆。
```
系统唯一目录表
┌──────────┬────────┬────────────┐
│ 文件名    │ 属性    │ 物理位置    │
├──────────┼────────┼────────────┤
│ a.txt    │ ...    │ block 100  │
│ b.c      │ ...    │ block 233  │
│ os.pdf   │ ...    │ block 900  │
└──────────┴────────┴────────────┘
```
所有用户的所有文件都放在同一个目录下。

**优点**
实现容易
查找逻辑简单
适合早期小型系统或单用户系统

**缺点**
容易重名
查找效率低
不利于文件分类管理
保护和共享不方便

所以一级目录只适合非常简单的系统。

### 二级目录结构
二级目录结构把目录分成两级：
```
第一级：主文件目录 MFD
第二级：用户文件目录 UFD
```
第一级主文件目录用于管理所有用户文件目录，它的目录项登记系统接受的用户名字以及该用户文件目录的地址；
第二级是用户文件目录，为该用户的每个文件保存一个登记栏；每一用户只允许查看自己的文件目录。

**优点**
**不同用户可以有同名文件**
**用户隔离更好** 采用二级目录时，任何文件存取都通过主文件目录，于是可以检查访问者的权限，避免一个用户未经授权访问另一个用户的文件，从而保证用户文件私有性，实现文件保密和保护
**查找范围缩小** 查找文件时，先根据用户名找到用户目录，再在该用户目录中查文件，不需要扫描全系统所有文件。

**缺点**
对于同一个用户而言，仍然存在文件多、容易重名的问题。
所以现代操作系统普遍使用树形目录结构。

### 树形目录结构
树形目录结构也叫层次目录结构。
每一级目录既可以登记下一级目录，也可以登记文件，于是形成层次文件目录结构；层次目录结构通常采用树形目录结构，它是一棵倒置的有根树，树根是根目录；从根向下，每一个分叉是子目录，树叶通常是文件。

**优点**
它能较好反映现实世界中具有层次关系的数据集合，也能较确切地反映系统内部文件组织结构；
不同文件可以重名，只要它们不在同一个末端子目录中；
还易于规定不同层次或子树中文件的不同访问权限，便于文件保护、保密和共享。

#### 树形目录中的文件定位：路径名
树形目录引入了**路径名**：
在树形目录结构中，一个文件的全名包括从根目录开始到文件为止，通路上遇到的所有子目录路径，也称为路径名；各子目录名之间用 `/` 或 `\` 隔开。

#### 绝对路径名和相对路径名
文件查找可以从根目录查起，即绝对路径名；也可以从当前目录查起，即相对路径名。

**绝对路径名**
从根目录开始写完整路径。
特点是：无论当前目录在哪里，它都唯一确定目标文件。

**相对路径名**
从当前目录开始写路径。
相对路径依赖当前工作目录。

#### 当前目录、`.` 和 `..`
现代操作系统通常为每个进程维护一个**当前工作目录**。用户使用相对路径时，系统就从当前目录开始解析。

```
.   表示当前目录
..  表示父目录
```

#### Windows 与 Unix/Linux 的层次目录差异
一个硬盘分区可以组织成一棵子树；每棵子树可以对应一个逻辑盘符，例如 Windows；也可以把众多子树嫁接成一棵大树，例如 UNIX。

Windows 通常以盘符为入口：
```
C:\
D:\
E:\
```

Unix/Linux 风格
Unix/Linux 只有一个统一根目录：
`/`
## 文件目录管理
### 文件查找
文件查找是文件目录管理的重要工作。“按名存取”文件就是系统根据用户提供的文件路径名搜索各级文件目录，找到该文件。可以从根目录查起，也可以从当前目录查起。

### 目录项查找方法
当系统进入某个目录后，还需要在该目录文件中找目标名字。例如进入 `/home/fei` 后，要找到 `myfile.c` 对应的目录项。

#### 顺序查找法
顺序查找法就是从目录文件的第一个目录项开始，逐项比较文件名。

**优点**
实现简单  
目录项无需排序  
适合小目录

**缺点**
目录很大时效率低
平均要查找 n/2 个目录项
最坏要查找 n 个目录项

#### 二分查找法
如果目录项按照文件名或关键字有序排列，就可以使用二分查找。

**优点**
比顺序查找快
时间复杂度 O(log n)

**缺点**
需要目录项保持有序
插入和删除目录项时可能需要移动数据
目录频繁变化时维护成本高

#### 杂凑法 / 哈希法
哈希法是把文件名通过哈希函数映射到目录表中的某个位置。

**优点**
平均查找速度快
适合大目录
平均接近 O(1)

**缺点**
存在哈希冲突
需要冲突处理机制
目录扩展时可能需要重哈希

### 树形目录查找的性能问题
树形目录虽然表达能力强，但查找路径时可能经过很多目录节点。
当一个文件经过许多目录节点时，使用不方便；系统沿路径查找目录时，往往要多次访问文件存储器，使访问速度大大减慢。
如果把所有文件目录都复制到主存，访问速度虽然加快，但会增加主存开销；
一种有效办法是把常用和正在使用的文件目录复制进主存，既不增加太多主存开销，又能明显减少目录查找时间。

### 目录缓存与 dentry
现代 Linux 中有一个重要结构叫 **dentry**，即目录项对象。
VFS 中的目录项对象代表路径中的一个组成部分，用来存放目录项与对应文件进行链接的信息；VFS 会把最近、最常使用的 dentry 对象放在目录项高速缓存中，加快文件路径名搜索过程，提高系统性能。

### 活动文件表
为什么需要活动文件表？
如果程序每次读写文件都重新走一遍路径解析，会非常慢。

因此文件系统通常采用“打开—读写—关闭”的模式：
```
open：查目录，建立内存数据结构
read/write：直接根据打开文件表操作
close：释放内存结构，必要时回写元数据
```


系统可以为每个用户进程建立一张活动文件表。当用户使用文件之前，先通过“打开”操作，把该文件有关目录信息复制到指定主存区域，有关信息填入活动文件表，以建立用户进程和该文件索引的联系；当不再使用该文件时，使用“关闭”切断用户进程和文件的联系，如果目录被修改过，则更新辅存中对应的文件目录。

**活动文件表中通常保存什么？**
文件描述符 fd
文件打开方式：只读、只写、读写、追加
当前文件偏移量 offset
指向 inode / FCB 的指针
访问权限状态
引用计数
缓冲区状态
文件锁状态

# 文件的共享、保护和保密
文件共享：让多个用户或进程共同使用某些文件
文件保护：防止文件被破坏，保证文件完整性
文件保密：防止文件内容被未经授权的用户窃取

## 文件的安全和保护
文件是长期保存在辅助存储器上的信息资源，比进程中的临时数据更持久，也更容易涉及用户隐私、系统配置、程序代码、业务数据等重要内容。

三个核心层面：
共享控制：允许合理共享
访问控制：防止越权访问
完整性保护：防止文件被破坏或丢失

### 文件共享
文件共享是指不同用户共同使用某些文件。

文件共享不是简单复制。真正的共享通常意味着：
多个名字或多个进程指向同一个文件实体
文件内容只保存一份
不同用户根据权限访问同一文件

### 文件共享的好处
减少用户大量重复性劳动，免除系统复制文件的工作，节省文件占用的存储空间，减少程序设计输入输出文件的次数。

**减少重复劳动**
如果一个班级所有同学都需要一份实验指导书，不必每个人单独维护一份副本，可以共享同一个文件。

**节省存储空间**
如果每个用户都复制一份相同文件，会浪费磁盘空间。共享可以让文件实际只保存一份。

**便于协作**
多个用户共同完成项目时，共享文件可以作为共同工作对象。

**减少 I/O** **操作和管理开销**
系统不需要反复复制同一文件，也不需要维护大量冗余副本，从而降低 I/O 和空间管理成本。

### 文件共享带来的问题：并发控制
允许文件共享的系统中，操作系统应提供手段实现对共享文件的同步控制；多个进程可能同时存取一个文件，如果它们同时读，操作系统可以进行公用控制；但如果有进程写，例如进程 A 修改文件，同时进程 B 读取同一文件，操作系统必须提供同步控制机制，以保证文件数据完整性。

### 读者—写者问题与共享文件
多个读者可以同时读
写者写文件时，其他读者和写者都不能访问
同一时刻通常只允许一个写者写
