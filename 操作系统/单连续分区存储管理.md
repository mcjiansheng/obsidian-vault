[[存储管理]]
单连续分区存储管理的核心思想：
**每个进程在物理内存中占用一块完全连续的存储区域**

包含三种形态：
- 单用户连续分区
- 固定分区
- 可变分区

思想更直观，但灵活性较差。

## 单连续分区存储管理

### 单用户连续分区存储管理

主存只分成两大区域：
```
低地址
┌──────────────┐
│ 操作系统区     │
├──────────────┤
│ 用户区        │
└──────────────┘
高地址
```
单用户连续分区方式把主存划分为**系统区**和**用户区**，用一个**栅栏寄存器**界分两者，硬件在执行时用它进行存储保护；这种方式一般采用静态重定位，硬件实现代价低，典型适用于 DOS 这类单用户单任务系统

这里的“栅栏寄存器”可以理解为一条边界线。用户程序只能访问用户区，不能越过边界访问操作系统区。

如果用户程序访问地址小于栅栏值，说明它试图访问系统区，硬件会阻止。

### 单用户连续分区的特点
非常简单，系统开销地，地址转换也简单
程序装入时可以一次性完成地址转换，也就是 **静态重定位**

**缺点**：**主存中同一时刻只能有一个用户程序运行**，不能支持真正的多道程序设计，CPU 和 I/O 设备利用率较低。

### 固定分区存储管理
为了支持多道程序，操作系统把用户区再划分成多个固定分区。
固定分区方式的基本思想是：支持多个分区，但**分区数量固定、分区大小固定**，可以采用静态重定位，硬件实现代价低，是早期 OS 采用的方式
```
0K     4K     12K     28K     44K     68K     92K
┌──────┬──────┬───────┬───────┬───────┬───────┐
│ OS区  │ 8K  │ 16K    │ 16K  │ 24K    │ 24K  │ ...
└──────┴──────┴───────┴───────┴───────┴───────┘
       分区1  分区2   分区3   分区4   分区5
```
固定分区有两种常见划分方式。

一种是**等长分区**，比如每个分区都是 64KB。这种方式管理简单，但浪费严重。一个只需要 10KB 的程序也要占用 64KB。

另一种是**不等长分区**，比如有 8KB、16KB、32KB、64KB 等不同大小的分区。这样可以让小程序进入小分区，大程序进入大分区，空间利用率比等长分区好。

### 固定分区的主存分配表
固定分区方式需要一张**主存分配表**
表项包括：
分区号、起始地址、长度、占用标志

|分区号|起始地址|长度|占用标志|
|---|--:|--:|---|
|1|8K|8K|空闲|
|2|16K|16K|Job1|
|3|32K|16K|空闲|
|4|48K|16K|空闲|
|5|64K|32K|Job2|

当一个作业申请内存时，系统查表，找一个空闲且容量足够的分区，把作业装入，然后把占用标志改为该作业名或进程号

当作业结束时，系统把对应分区的占用标志改回空闲

### 固定分区的分配方式
固定分区常见有两种作业队列组织方式

第一种是**每个分区一个队列**
例如 16KB 分区有自己的队列，32KB 分区有自己的队列。作业根据大小进入对应队列。
优点是简单；缺点是可能出现某个分区空着，而另一个分区队列排很长，整体利用率不高。

第二种是**所有分区共用一个队列**
系统从队列中选择作业，为它找一个合适的空闲分区
这种方式比分区独立队列灵活，但分配算法稍复杂。

### 固定分区的内零头问题
固定分区最大的问题是**内部碎片**，课件称为“内存内零头”或“内零头”。
`内部碎片 = 已分配分区大小 - 进程实际需要大小`
固定分区的分区越大，可能浪费越多；分区越小，又可能装不下大程序。

## 可变分区存储管理
固定分区的问题是：分区大小事先固定，容易产生内部碎片。于是引入**可变分区存储管理**。
可变分区的思想是：**不预先固定分区大小，而是在进程装入时，根据进程需要动态划分一块连续内存。**

这比固定分区灵活，但会带来新的问题：**外部碎片**。

### 可变分区的分配表
可变分区方式需要维护**已分配区表**和**未分配区表**，并且通常采用链表形式；表项包含起址、长度、标志等信息。
已分配区表：

| 起始地址 | 长度  | 标志  |
| ---- | --- | --- |
| 4K   | 6K  | J1  |
| 46K  | 6K  | J2  |

未分配区表：

| 起始地址 | 长度  | 标志  |
| ---- | --- | --- |
| 10K  | 36K | 未分配 |
| 52K  | 76K | 未分配 |
系统分配内存时查“未分配区表”；进程结束时，从“已分配区表”删除对应项，并把释放出来的空间插入或合并到“未分配区表”。
### 可变分区的分配过程
当进程申请一块大小为 `S` 的内存时，操作系统在空闲分区表中找一个长度大于等于 `S` 的空闲区。

找到后通常有两种情况：
情况一：空闲区大小刚好等于 S
直接把整个空闲区分配给进程。

情况二：空闲区大小大于 S
把前面或后面 S 大小的部分分给进程，剩余部分继续作为空闲区。

如果剩余空间很小，小到没有实际使用价值，系统可能会把整个空闲区都分给该进程，避免产生过小碎片。
## 可变分区的四种分配算法

### 最先适应算法First Fit
**最先适应算法**从空闲分区表的开头开始查找，找到第一个能满足请求的空闲区就分配。
优点是简单、速度较快；缺点是低地址区域容易被反复切割，前部产生很多小碎片。

### 邻近适应算法 Next Fit
**邻近适应算法**也叫循环首次适应。它不是每次都从表头开始找，而是从上一次查找结束的位置继续向后找。
优点是避免每次都从低地址开始，分配更均匀；缺点是可能错过前面较合适的空闲区，导致大空闲区被更快消耗，整体效果不一定优于最先适应。

### 最优适应算法 Best Fit
**最优适应算法**选择能够满足申请要求的最小空闲区。

最优适应看起来最节省空间，但是**最优适配算法最容易产生外零头**。

原因是它总是把最接近需求的空闲区切掉，容易留下大量很小的碎片。例如 300K 分给 212K 后剩下 88K，如果后续作业都需要 100K 以上，这 88K 就很难再用。

### 最坏适应算法 Worst Fit
**最坏适应算法**每次选择最大的空闲区进行分配。
它的想法是：不要把小空闲区切得更碎，而是从最大空闲区中切出一块，剩下的空间仍然比较大，未来还有利用价值。
优点是剩余空闲区不至于太小；缺点是大空闲区会被不断削弱，大作业到来时可能没有足够大的连续区。

### 可变分区的回收
进程结束后，操作系统要回收它占用的连续分区。
进程 X 终止后，可能与前后空闲区合并，也可能形成新的空闲区。
回收时要看被释放分区的相邻区域是不是空闲区。通常有四种情况。

#### 前后都不是空闲区
```
回收前：
[ A进程 ][ X进程 ][ B进程 ]

回收后：
[ A进程 ][ 空闲区 ][ B进程 ]
```
此时新建一个空闲区表项。

#### 前面/后面 是/都是 空闲区
与空闲区合并，形成更大的空闲区

### 可变分区的外零头问题

可变分区虽然减少了固定分区中的内部碎片，但会产生**外部碎片**
可变分区方式会随着进程的内存分配产生一些小的、不可用的内存分区，称为**内存外零头**；任何适配算法都不能避免外零头。

外部碎片的本质：
**总空闲空间足够，但没有足够大的连续空闲空间。**

### 移动技术，也叫紧缩/程序浮动技术
为了解决外部碎片，可以使用**移动技术**，也叫**紧缩**或**程序浮动技术**。
移动技术通过移动分区来解决内存外零头，并且需要动态重定位支撑。

**为什么移动技术需要动态重定位**？
因为进程移动后，它在物理内存中的起始地址变了。如果程序中的地址已经在装入时固定死了，那么移动后就会出错。只有采用动态重定位，CPU 每次访问时根据新的基址寄存器进行转换，进程才能被安全移动。

|对比项|固定分区|可变分区|
|---|---|---|
|分区数量|系统启动前固定|随进程装入和退出动态变化|
|分区大小|固定|按进程大小动态确定|
|每个进程是否连续|是|是|
|主要表结构|固定分区表|已分配区表、空闲区表/链|
|主要碎片|内部碎片|外部碎片|
|分配算法|简单查找可用分区|First Fit、Next Fit、Best Fit、Worst Fit|
|回收处理|标记为空闲|需要和相邻空闲区合并|
|是否需要紧缩|一般不涉及|可能需要|
|地址转换|可静态重定位|常配合动态重定位更灵活|
|优点|简单、硬件开销小|空间利用率比固定分区高|
|缺点|分区固定，浪费明显|外部碎片，管理复杂|
