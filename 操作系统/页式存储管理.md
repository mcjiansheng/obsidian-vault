[[存储管理]]

## 页式存储管理解决什么问题
在前面的连续分区管理中，一个进程必须占用一整块连续主存。这样会产生外部碎片：总空闲空间够，但没有一块足够大的连续空间。
**进程不再要求连续装入主存，而是被切成若干固定大小的页面，分散放入主存的页框中。**

分页存储器把主存划分成多个大小相等的页框；程序逻辑地址也自然分成页；不同页面可以放在不同页框中，不需要连续；页表用于维系进程的主存完整性。
页式管理的核心是：**逻辑上连续，物理上可以离散。**

## 页式存储管理的基本原理
### 页面与页框

|名称|所属空间|含义|
|---|---|---|
|页，Page|进程逻辑地址空间|把进程逻辑地址空间按固定大小切分|
|页框，Page Frame|物理主存空间|把主存按同样大小切分|
|页内位移 / 单元号|页内部|页内具体偏移位置|
页和页框大小必须相同。这样一个逻辑页才能完整放入一个物理页框。

### 页表的作用
因为进程的页面不连续存放，所以系统必须知道：
```
进程的第0页放在哪个页框？
进程的第1页放在哪个页框？
进程的第2页放在哪个页框？
...
```
这张映射表就是**页表**。

所以页式管理中，程序看到的逻辑地址仍然连续，但真正访问主存时必须通过页表转换。

### 页式管理的优缺点
页式管理最大的优点是**消除外部碎片**。因为主存被划成等大的页框，只要有足够数量的空闲页框，就能装入进程，不要求连续。
但是它会产生**内部碎片**。因为进程大小不一定刚好是页大小的整数倍，最后一页可能装不满。

## 页式存储管理的地址转换
页式存储管理的逻辑地址由两部分组成：**页号和单元号**；物理地址也由两部分组成：**页框号和单元号**；地址转换可以通过查页表完成。

### 页式地址结构
逻辑地址形式：`页号 p | 页内位移 d`
物理地址形式：`页框号 f | 页内位移 d`

注意：**页内位移不变**。地址转换只把“页号”换成“页框号”。

### 地址转换工具
```
A = 逻辑地址
L = 页面大小
p = 页号
d = 页内位移
f = 页表[p] 中记录的页框号

p = A / L 取整
d = A % L
物理地址 = f × L + d
```
### 地址转换例题的基本流程
第一步：用逻辑地址除以页大小，得到页号和页内位移
第二步：查页表，得到页框号
第三步：页框号 × 页大小 + 页内位移

### 页表控制寄存器与多道程序环境
在多道程序环境下，每个进程都有自己的页表。
进程表中登记了每个进程的页表；当进程占有处理器运行时，其页表起始地址和页表长度会送入页表控制寄存器。
也就是说，进程切换时，不只是切换寄存器、程序计数器、状态字，还要切换当前进程的页表信息。

地址转换时，硬件会根据页表始址找到当前进程的页表，根据页表长度判断页号是否越界。如果页号超过页表长度，就产生越界中断。课件的多道程序地址转换图中也展示了页表始址、页表长度、页表控制寄存器、快表、越界比较和页框号拼接的过程。

### 不带快表的地址转换代价
如果没有快表，访问一个逻辑地址通常要访问两次主存：

第一次访问主存：查页表，得到页框号
第二次访问主存：根据物理地址访问真正的数据或指令

所以分页虽然灵活，但会增加访存开销。

### 快表TLB
为了解决页表访问慢的问题，引入**快表**，也叫 **TLB，Translation Lookaside Buffer**。
快表本质上是页表项的高速缓存，保存最近使用过的页号到页框号的映射。
```
CPU产生逻辑地址
        ↓
先查快表
        ↓
快表命中：直接得到页框号，形成物理地址
        ↓
快表未命中：查页表，得到页框号，同时把该项写入快表
```
快表存储正在运行进程的若干非连续页表项，其意义是快表访问速度高于内存，可以减少访问内存次数，提高页式寻址效率

计算代价：
主存访问时间 200ns，快表访问时间 40ns，快表命中率 90%，则平均地址转换代价为：
$(200 + 40) × 90\% + (200 + 200 + 40) × 10\% = 260ns$

一般公式为
$EAT = α × (ε + m) + (1 - α) × (ε + 2m) = ε + m + (1- α) × m$
其中
```
EAT：有效访问时间
α：快表命中率
ε：快表访问时间
m：主存访问时间
```

**注意：快表未命中也要计算一次快表命中时间**

### 多级页表
页表本身也要占主存。虚拟地址空间越大，页表可能越大。为了解决页表过大的问题，可以使用多级页表。
多级页表中，系统为每个进程建立一张页目录表，页目录表的每个表项对应一个页表页，而页表页的每个表项给出页面和页框的对应关系；逻辑地址结构由页目录、页表页、位移三部分组成

二级页表地址结构可以理解为：
```
逻辑地址：
页目录号 | 页表页号 | 页内位移
```
转换流程：
```
页目录号 → 查页目录 → 找到某个页表页
页表页号 → 查该页表页 → 找到页框号
页内位移 → 与页框号拼接成物理地址
```

多级页表的本质是：**页表本身也分页**。
二级页表中，页目录项是页表页的索引，页表页项是进程程序页面的索引。

优点是节省页表空间，缺点是增加地址转换层次，所以更依赖 TLB。

## 页式虚拟存储管理
页式虚拟存储管理是分页机制和虚拟存储思想的结合，也是现代操作系统的主流存储管理技术。

把进程全部页面装入虚拟存储器，执行时先把部分页面装入实际内存，然后根据执行行为动态调入不在主存的页，同时进行必要的页面调出；首次只把进程第一页信息装入主存，称为请求页式存储管理。

### 从普通分页到虚拟分页
普通页式存储管理默认：进程的所有页都在主存中。

虚拟页式存储管理允许：
部分页面在主存
部分页面在辅存
程序运行时按需调入

这样，一个程序可以比实际主存更大，因为并不要求全部页面同时在主存中。

### 页式虚拟存储的页表扩展
普通页表只需要记录页号到页框号的映射。虚拟分页中，页可能不在主存，所以页表项必须扩充。
页式虚拟存储管理的页表需要指出每页的虚拟地址、实际地址、主存驻留标志、写回标志、保护标志、引用标志、可移动标志等，并包含主存页框号和辅助存储器地址。

|字段|作用|
|---|---|
|页框号|如果页面在主存，指出所在页框|
|驻留位 / 有效位|表示该页是否在主存|
|辅存地址|如果不在主存，指出它在磁盘的位置|
|修改位 / 脏位|表示该页是否被写过|
|引用位 / 访问位|表示该页最近是否被访问|
|保护位|控制读、写、执行权限|
|可移动位 / 锁定位|标识是否允许被换出或移动|

驻留位 = 1：页面在主存，可以完成地址转换
驻留位 = 0：页面不在主存，产生缺页中断
### 缺页中断
当 CPU 访问某个页面，而该页不在主存时，就会发生**缺页中断**。

CPU 处理地址时，如果页驻留，则获得页框号形成绝对地址；如果页不在内存，则 CPU 发出缺页中断。操作系统处理缺页中断时，如果有空闲页框，就根据辅存地址调入页并更新页表与快表；如果没有空闲页框，就决定淘汰页，调出已修改页，调入新页，并更新页表与快表。

缺页中断属于一种“异常”，但它不一定是错误。对于请求分页系统来说，缺页是正常机制的一部分。

### 页式虚拟存储的优点
第一，扩大程序可用地址空间。程序可以大于物理主存。

第二，提高多道程序度。每个进程只装入部分页面，主存中可以容纳更多进程。

第三，支持进程隔离。每个进程有自己的虚拟地址空间，互不干扰。

第四，支持共享。多个进程的页表可以指向同一个物理页框，例如共享库代码页。

第五，便于按需装入。程序暂时不用的页面不必进入主存，减少主存占用。

### 页式虚拟存储的代价 
页表更复杂
地址转换开销增加
缺页中断代价很高
需要页面置换算法
需要主存与辅存频繁配合
可能出现抖动

## 页面调度
当主存空间已满而又需要装入新页时，页式虚拟存储管理必须按照一定算法把已在主存的一些页调出去；选择淘汰页的工作称为页面调度，选择淘汰页的算法称为页面调度算法。
如果页面调度算法设计不当，会出现刚被淘汰的页面立即又要调入并反复发生，这种现象称为抖动或颠簸

### 页面调度要解决的问题
当发生缺页中断时，系统要把目标页调入主存。如果有空闲页框，很简单：
`直接把页面调入空闲页框`

但如果主存已经满了，就必须选择一个已有页面换出去：
### 影响缺页中断率的因素
影响缺页中断率的因素包括：
分配给进程的页框数，页框数越多缺页率越低；
页面大小，页面尺寸越大缺页率越低；
用户程序编制方法，在大数据量情况下对缺页中断率也有很大影响。

页框数越少 → 能驻留的页面少 → 更容易缺页
页面太小 → 页面数量多 → 页表大，缺页可能增加
页面太大 → 内部碎片和I/O传输浪费可能增加
程序局部性好 → 缺页少
程序局部性差 → 缺页多

### OPT 最佳页面置换算法
当要调入新页面时，首先淘汰以后不再访问的页；
如果都还会访问，就选择距离现在最长时间后才再访问的页

但 OPT 只能模拟，不能真正实现。
原因很简单：OPT 需要知道未来的访问序列，而真实系统无法预知未来。

OPT 的作用主要是作为理论下界：其他算法的缺页次数不可能比 OPT 更少。

### FIFO 先进先出算法
FIFO 的思想是：**最早进入主存的页面最早被淘汰**。

优点是实现简单。缺点是它不考虑页面是否常用，可能把经常访问的页面淘汰掉。

FIFO 还有一个著名问题：**Belady 异常**。通常我们认为页框数越多缺页率越低，但 FIFO 中可能出现页框数增加，缺页次数反而增加的情况。

### LRU 最近最久未使用算法
LRU，Least Recently Used，思想是：**淘汰最近最长时间没有被访问的页面**。

它的依据是局部性原理：过去一段时间没用过的页面，将来短期内也可能不常用。

LRU 比 FIFO 更符合程序局部性，但精确实现成本较高。真实系统中通常通过引用位、时钟算法等方式近似实现。

### CLOCK 时钟算法 / 第二次机会算法
CLOCK 是 LRU 的常用近似算法。

它把所有页框组织成一个环形队列，每个页面有一个引用位 R：
R = 1：最近访问过
R = 0：最近未访问

淘汰时，指针像时钟一样扫描：
遇到 R = 0 的页：淘汰
遇到 R = 1 的页：把 R 清 0，给它第二次机会，指针继续

所以 CLOCK 也叫第二次机会算法。

改进 CLOCK 还会结合修改位 M：
R = 0, M = 0：最好淘汰，没访问过且不用写回
R = 0, M = 1：可淘汰，但要写回
R = 1, M = 0：最近访问过，暂不淘汰
R = 1, M = 1：最近访问过且修改过，最不优先淘汰
## 反置页表
普通页表是“每个进程一张页表”，页表大小与虚拟地址空间大小相关。虚拟地址空间越大，页表越大。
页表设计的一个重要缺陷是页表大小与虚拟地址空间大小成正比。

### 普通页表 vs 反置页表
普通页表是：
```
每个进程的每个虚拟页 → 一个页表项
```
也就是面向虚拟页。

反置页表是：
```
每个物理页框 → 一个表项
```
也就是面向物理内存。

反置页表针对内存中的每个页框建立一个页表，按照页框号排序；表项包含正在访问该页框的进程标识、页号、特征位、哈希链指针等。
因此，反置页表大小主要取决于物理页框数量，而不是虚拟地址空间大小。

### 反置页表项内容
页号：虚拟地址页号
进程标识符：使用该页的进程号
标志位：有效、引用、修改、保护、锁定等
链指针：哈希链

其中，页号和进程标识符要结合起来，才能唯一标识某个进程虚拟地址空间中的一页。
### 反置页表的地址转换过程
基于反置页表的地址转换过程是：
MMU 通过哈希表把进程标识和虚页号转换成哈希值，指向 IPT 的一个表目；
MMU 遍历哈希链找到所需进程的虚页号，该项的索引就是页框号，通过拼接位移生成物理地址；
如果遍历后没有找到匹配页表项，说明该页不在内存，产生缺页中断。

普通页表是：
```
虚页号 → 页框号
```

反置页表是：
```
在所有物理页框中查找：哪个页框装着这个进程的这个虚页？
```

### 反置页表的优缺点
优点：
页表大小与物理内存页框数相关
节省页表空间
适合大虚拟地址空间系统

通过哈希表和反向表结构，只有一项对应于一个实存页，而不是虚拟页，因此不论有多少进程、支持多少虚拟页，页表都只需要实存中的一个固定部分。

缺点：
地址转换更复杂
需要哈希查找和链表遍历
处理共享页面更麻烦
更依赖 TLB

所以反置页表通常和 TLB 配合使用。绝大多数地址转换希望在 TLB 中完成，只有 TLB 未命中时才查反置页表。
